# 真实世界经验引擎核心讨论纪要

> 基于与 AI 的深度技术讨论整理
> 项目：RELL（Real-world Experience Learning Loop）
> 主线：机器人如何通过真实交互经验在物理世界中学习与执行

---

## 一、根本信念：经验主义路线

### 核心分歧

| 主流路线（理性主义/LLM 思路） | 我们的路线（经验主义） |
|----------------------------|---------------------|
| 将物理世界参数化、建模，在模型中做规划再下发执行 | 不建模世界，直接问世界 |
| 相信从过去数据中学到的分布能预测未来（休谟归纳） | 承认归纳问题的逻辑不成立 |
| 试图在神经网络内部压缩出"世界的完整内部表征" | 世界是无法被完全压缩进任何有限模型的 |
| 在线执行只是跑推理 | 核心能力应在在线执行中实时与世界交互获得 |

### 科学定理层面的支撑

1. **休谟归纳问题** — 从过去观察到的有限实例，不能逻辑必然地推导出未来未观察到的实例。LLM 的成功依赖于文本世界变化极慢，物理世界的变化速度使得基于历史数据的归纳式预测不可靠。

2. **量变质变定律（相变不可预测）** — 系统的关键参数越过某个临界点时，行为发生质变。在相变发生之前，基于历史数据的外推无法预测相变的精确位置。"度"就是相变检测器——不在计算中预测，而在实时观测中感知质变是否触发。

3. **哥德尔不完全性定理的工程投影** — 任何基于历史数据的模型（包括 LLM）都存在它无法覆盖和预测的真实物理场景。模型是有限的形式系统，物理世界是无限的非形式系统，前者永远无法完全覆盖后者。

4. **Lorenz 初始敏感性（蝴蝶效应）** — 物理世界是多变量、强耦合、非线性的。初始条件的微小差异在时间演化中被指数级放大。不预测偏差，实时纠正偏差。

### 结论

**用 LLM 的文本世界思路去做物理世界，在已知科学定理层面是不可行的。** 这个框架不试图突破这些定理，而是承认它们，并围绕它们设计务实路线：

- 不预测物理世界，而是实时问物理世界
- 不存储全量状态，而是世界就是状态存储
- 不在离线模型中穷举所有可能，而是在在线执行中持续对齐

## 二、框架全景：六步结构 + LLM 翻译层

```
人类语言指令 → [LLM 翻译层] → 结构化机器指令 → [真实世界经验引擎] → 动作执行
```

LLM 只做语义翻译（将人类语言翻译为结构化机器指令），不参与推理、规划、后果预测。

| 步骤 | 名称 | 产出 | 类比认知阶段 |
|------|------|------|-------------|
| S1 | 空间语义生成与内化 | 空间认知模型（区域、属性、关系） | 感知运动期 |
| S2 | 经验类比检索与后果预测 | 当前情境下的预期后果 | 前运算期 |
| S3 | 概念归纳与跨域迁移 | 概念标签 + 典型经验模式 | 具体运算期 |
| S4 | 任务经验检索与规划 | 类比的当前任务计划 | 具体运算期 |
| S5 | 执行中途恢复 | 偏离情境下的补救动作 | 运动反射 |
| S6 | 人类偏好学习 | 情境-动作-偏好指示 | 社会适应 |

核心特征：**六步彼此独立，不依赖仿真、不依赖 LLM 推理、不依赖神经网络端到端训练、不依赖案例推理（CBR）范式。** 每一步的输出是显式的、可审计的结构化数据。

## 三、核心概念体系

### 3.1 世界就是状态存储

最大的认知转变。这个框架中，所有"需要记录状态"的地方，最终都被物理世界本身替代了：

| 最初以为需要的 | 最终选择的 | 理由 |
|--------------|-----------|------|
| 经验库存全量失败记录 | 只存最优解 | 失败的意义在于收敛到最优解，一旦达到，失败信息在物理世界中（摔碎的杯子） |
| 中断时序列化意图状态机 | 中断回来直接看世界 | 水位不会变、壶不会跑，重新观察即可重建上下文 |
| 过程执行时的内部状态跟踪 | 实时观测当前世界 | 传感器+视觉双通道独立对齐 |
| 离线仿真做大量预演 | 直接在真实世界试 | 结果就是答案 |

物理世界的四个不可替代特性：

1. **零写入成本** — 世界自动维护状态，不需要 INSERT INTO
2. **零同步问题** — 物理世界永远一致，没有缓存过期、主从延迟
3. **零存储上限** — 世界是无限容量的状态存储
4. **自我修复** — 物理世界不会"写错"

**原则：不记录状态，状态在世界中。**

### 3.2 过程（Process）而非原子技能（Atomic Skill）

| 原子技能（不要） | 过程（要） |
|---------------|-----------|
| 离散的、可拼接的、无内部状态的 | 连续的、有内部时序动力学的、不可分割的 |
| "拿"、"放"、"推" | "倒水"、"拿杯子"、"开门" |
| 对应单个动作 | 对应一个完整意图下的事件流 |
| 最优解 = 一个参数（如3N） | 最优解 = 一条参数曲线 + 状态机骨架 |
| 边界模糊，反复争论"拧"和"旋"算不算两个 | 边界自然由意图决定 |

### 3.3 质、量、度 —— 过程的标准描述

| 概念 | 哲学含义 | 工程映射 | 例子（倾斜倒水阶段） |
|------|---------|---------|---------------------|
| **质** | 事物的本质规定性 | 传感器/状态变量的类型 | 壶口倾角、水流速度 |
| **量** | 质的数量表现 | 具体数值/范围 | 倾角 0°→30°，流速 0→15ml/s |
| **度** | 质与量的统一边界，阶段转换的触发器 | 阶段转换条件 | 水面距杯口 < 1cm → 进入回正阶段 |

**度是关键设计** — 它把过程从模糊的"阶段序列"变成可精确检索和执行的带出口条件的有限状态机。

同一过程可适配不同偏好，只需调节"度"的阈值（"倒八分满" vs "倒满"只需改水面距杯口距离参数）。

**失败的自然归因** — 过程在某阶段失败，原因一定是"度的条件一直未被触发"或"触发了错误的度"。

### 3.4 意图状态机与双通道验真

执行过程中，机器人在两件事之间持续对齐：

| 状态机 | 来源 | 做什么 |
|--------|------|--------|
| **意图状态机** | 从经验库检索到的过程最优解骨架 | 期望的状态变化序列 |
| **实际状态机** | 传感器+视觉的实时融合 | 物理世界的当前状态 |

持续对齐方式：

```
数字通道（执行器传感器：力矩、角度、流速）
  + 物理通道（摄像头：水位、位置、接触状态）
  ↓
双通道交叉验证 → 确认实际状态是否在意图状态的容差范围内
```

**两个通道独立使用时都有缺陷，合在一起做交叉验证时，不需要任何一方有多高的精度。** 结论来自双通道独立一致。

### 3.5 感知-动作闭环作为运行时内置机制

这是一个关键抽象剥离：

- **经验库存什么**：过程的阶段序列 + 每个阶段的关键参数（质、量、度）
- **执行引擎自带什么**：感知-动作闭环是固定机制，不存进经验库

执行任何阶段的任何动作时，引擎**自动**把"我该做成什么样"与"实际上做成什么样"实时对齐，有偏差就自动微调。这是全局通用的运行时代理，每个阶段执行时自动启用。

**感知-动作闭环是引擎的能力，不是经验库的内容。**

### 3.6 中断恢复：不需要断点续传

中断后回来，不需要持久化、不需要序列化、不需要存档格式。只需：

1. 重新获取当前实际状态（水位、倾角、手的位置）
2. 按过程阶段的"度"条件匹配当前状态对应哪个人口
3. 从那个阶段开始执行

**世界在你走开的时候不会消失。** 物理世界就是最好的状态存储介质。

## 四、经验库设计原则

### 4.1 只存最优解

失败的意义在于收敛到最优解。一旦达到最优解，失败记录本身没有保留价值。

- 不存失败记录
- 不存冗余的、被淘汰的次优解
- 每个独立事件只保留一条最优解

### 4.2 经验库上限可控

在专用机器人（如家务机器人）场景下，经验库规模是可知的：

| 维度 | 估计上限 |
|------|---------|
| 房间类型 | 5-8 种（厨房、客厅、卧室、浴室、阳台等）|
| 操作物体 | 30-50 种 |
| 任务类型 | 15-20 种 |
| 经验库总量 | ~8000 条（可预填充、可离线归纳）|

### 4.3 经验库不是技能库

- **经验库** — 存储过程（过程的阶段序列 + 质、量、度参数）
- **技能包** — 经验的打包分发单元，像 APP 一样上传、下载、卸载
- **"走到桌子"不是技能**，只是完成"拿杯子"这个过程的必要条件——由端侧编排引擎在执行时自动填充，不进入用户可见的技能列表

### 4.4 不需要跨本体迁移

这个框架限定在人形机器人，不考虑跨机器人共享。每台机器人的经验库是独立的，只从自己的实践中积累。数据不是瓶颈——专用场景的经验库规模是已知且有限的。

## 五、技能包生态愿景

```
机器人 = 硬件平台（人形）
      + 可换装的技能包（经验库）
      + 端侧编排引擎（状态图搜索 + 过程拼接）
```

技能包像手机 APP 一样：
- "厨房包" → 拿、放、倒水、擦桌、开关柜门
- "客厅包" → 拿遥控、端茶、捡东西、开灯
- 用户按需下载，不需要一次装所有技能
- 每个技能包独立迭代，失败只影响那个包

**语言世界可以通才，物理世界必须专才。**

## 六、与当前主流路线的对比

| 维度 | 世界模型 / 物理AI | 因果模型 | 我们的框架 |
|------|-----------------|---------|-----------|
| 核心方法 | 预测下一帧/下一状态 | 建模因果结构（SCM） | 经验检索+实时感知闭环 |
| 对世界的态度 | 压缩进神经网络 | 形式化因果图 | 不建模，直接问世界 |
| OOD 泛化 | 统计外推 | 反事实推理 | 试一次就知道 |
| 面对相变 | 模型需包含临界点参数 | 模型需包含因果阈值 | 用"度"实时感知 |
| 构建成本 | 海量数据+大算力 | 专家定义因果图 | 物理机器人+人工示范 |
| 可审计性 | 黑箱 | 半透明 | 每条决策可追溯至具体经验 |
| 哲学基础 | 理性主义 | 理性主义 | 经验主义 |

## 七、工程自评：最难的是这三个

1. **过程的标准格式定义（质、量、度的Schema）** — 这是整个技能包生态的基础协议。一旦定下来，所有技能包格式依赖它；一旦需要改，所有存量技能包都得迁。

2. **端侧编排引擎的核心逻辑** — 状态驱动的图搜索，输入当前状态+目标状态，图是过程的"度"条件连接的状态变换图，输出一条过程链。没有现成开源实现，需要自研轻量化嵌入式引擎。

3. **第一个种子技能包的数据从哪里来** — 需要开发者预先在标准场景下手动跑出 3-5 个技能的种子经验数据。这没有捷径，是工程启动成本。

**没有需要突破科学定理的地方，每一步在现有技术下都能跑通。**

## 八、几个重要的判断

1. **LLM 只做翻译层** — 不参与推理、规划、后果预测。它是语义翻译器，将人类语言映射为结构化机器指令。

2. **感知-动作闭环是内置引擎机制** — 不是经验库的内容。每个阶段被执行时自动启用，不涉及存储和检索。

3. **冲突不是 bug，是驱动情境描述精度提升的自然信号。** 当两个最优解在相同情境下指向不同动作时，说明情境描述遗漏了关键特征。

4. **主动说"我不知道"** — 因为不是端到端黑箱，系统可以明确知道"当前场景不在已右经验基元的覆盖范围内"。

5. **在限定场景（家务）下，这个框架站得住。** 泛化到任何场景都不行，但不需要——物理世界不需要一个万能机器人，需要的是像 APP 一样可换装的专用技能包。
