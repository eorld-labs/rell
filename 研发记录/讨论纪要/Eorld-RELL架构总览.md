## 二、总体定位：空间语义输入无关 + 经验技能中枢

RELL + P016 的最终目的不是做一个“倒水 Demo”，也不是替代机器人本体、世界模型或 SLAM 系统，而是构建一个能够让自动执行体在细分场景中泛化成功的中心技能中枢。

核心目标：

```
任何空间语义来源
        ↓
统一空间语义中间层
        ↓
任务链 / 过程链 / P016过程Runtime
        ↓
经验库 / 技能库 / 验真与审计
        ↓
数字自动执行体 / 模拟机器人 / 真实机器人
```

其中，机器人在本体系中被视为“真实世界自动执行体”的一种实现形态；数字 Agent、仿真体、真实机器人均可通过 Adapter 接入，只要其能够报告能力、接收过程、回传连续状态变量和目标因果事实观测。

### 2.1 空间语义输入无关性

RELL 不绑定某一种空间数据来源。任何能够提供空间语义的数据源，都可以作为上游输入：

```
机器人深度相机 / 摄像头即时建图
SLAM / VSLAM
世界模型
仿真环境
数字孪生
BIM / CAD
人工标注地图
楼宇平面图
多机器人共享地图
```

这些上游数据进入 RELL 前，需要先归一化为统一的空间语义结构：

```
space
region
object
pose
relation
affordance
constraint
navigation_graph
state_fact
access_rule
```

该层与 P003 的“跨平台三维空间数据互操作方法及系统”直接对应。P003 提供：

- 空间坐标基准归一化；
- 全局参考系与双向转换算子；
- 标准化空间对象模型；
- 位姿描述符、语义描述符、关系约束符；
- 视觉渲染属性与物理交互属性解耦；
- 异构空间计算环境间的协议封装和目标环境映射；
- 面向不同空间计算环境的一致映射基准；具体冲突处理可作为说明书实施例、上层治理能力或后续工程扩展，不作为 P003 当前收缩权项的主战场。

因此，P003 是 RELL 的“空间眼睛”和空间数据互操作底座。P003 不直接承担技能学习、经验迁移或任务规划，而是守住“任意空间语义来源进入 RELL 前的统一坐标、统一语义和目标环境本地化映射”。

P003 与 RELL/P016 的分工如下：

```
机器人建图 / SLAM / 世界模型 / 仿真 / 数字孪生 / BIM / CAD / 人工地图
        ↓
P003：统一空间参考框架 + 语义本体映射 + 空间语义协议包 + 目标环境本地化映射
        ↓
RELL：经验库 / 技能库 / 过程链 / 空间语义沙箱
        ↓
P016：过程模板 + 偶然绑定 + 阶段跃迁 + 因果验真 + 恢复
        ↓
Adapter：数字执行体 / 模拟机器人 / 真实机器人
```

P010 可进一步补充空间语义自动生成；P008 负责主体进入空间和行动边界；P016 则负责在空间语义基础上执行可复用的物理过程经验。

### 2.1.1 P008：主体空间准入与边界层

P008 的核心不是空间语义互操作，也不是机器人技能执行，而是“基于主体物理约束的空间可进入性判断”。其在 RELL 体系中的定位是：

```
P003：空间如何被统一描述
P008：某个主体能否进入该空间、从哪里进入、能在哪里移动、边界如何限制
RELL：主体在该空间中能否加载某个经验或过程链
P016：主体如何按过程模板执行阶段动作并完成验真
Adapter：主体根据自身实现形态把阶段动作落地
```

P008 已经覆盖以下主体约束来源：

- 视点高度；
- 主体包络尺寸；
- 移动半径；
- 转向半径；
- 可跨越高度；
- 可进入最小宽度；
- 可停靠面积；
- 传感器视场；
- 执行器边界。

因此，P008 可作为未来 Executor Profile 的空间准入依据：它回答“这个主体能不能进入和行动”，但不直接回答“这个主体如何完成具体技能动作”。例如，人形机器人高矮、臂长、腿长不同导致的多走两步、少走两步、底盘靠近、手臂伸展等动作适配，不放在 P008 中展开，而交由 Executor Profile、机器人 SDK、运动规划器、ROS/MoveIt、厂商 Adapter 或后续本体适配模块处理。

### 2.1.2 Executor Profile：本体差异的轻量预留

为了避免未来接入不同机器人时重构 Adapter，RELL 样品阶段预留 Executor Profile，但不在第一阶段实现复杂本体泛化。Executor Profile 用于描述自动执行体的最小主体能力画像：

```
executor_id
executor_type: digital_agent / simulated_robot / real_robot
body_profile: fixed_mock / humanoid / wheeled_arm / mobile_base
supported_actions
reachable_workspace
sensor_frames
end_effector_type
payload_limit
precision_level
mobility_constraints
spatial_entry_constraints
```

第一阶段只要求 Adapter 能上报固定 Mock 或模拟执行体画像，并参与审计或能力准入占位；不做复杂 IK、全身规划、不同机器人自动选择步数、跨本体运动轨迹生成。该预留与 P008 的主体物理约束相接，与 P016 的过程模板保持解耦。

### 2.2 中心技能中枢定位

RELL 的商业和工程定位是：

```
让一个新机器人或新自动执行体接入后，
通过载入经验库、过程模板、空间语义和Adapter，
快速知道在当前细分场景中应该如何做事。
```

这意味着经验库不是简单日志，而是中心技能中枢：

- 过程模板保存“事情怎样做”的不变结构；
- 偶然绑定保存“在当前环境中用什么对象、传感器和阈值做”；
- 空间语义提供“在哪里、对象是什么、关系是什么、可执行什么动作”；
- 目标因果事实验真提供“结果是否真的成立”；
- 审计与恢复记录提供“失败后如何处理、下次如何复用”；
- Adapter 负责把同一套经验落到不同执行体。

### 2.2.1 经验不变量契约：经验、概念与技能的边界

经验库不应保存某一次执行的绝对坐标、机器人关节角、固定执行时长或单一本体轨迹。经验库应保存能够跨空间、跨本体迁移的不变量，并通过当前空间语义、执行体画像和 Adapter 在执行时完成具体绑定与运动求解。

RELL 当前将经验沉淀为三类不变量：

1. 拓扑关系不变量：保存对象、执行体、语义区域和交互部位之间的相对关系，例如“执行体到达水源区域”“末端执行器持有杯子”“容器开口与水源资源区对齐”，而不是保存固定坐标。
2. 试探方向与物理约束不变量：保存动作意图方向、允许边界、最大限制和安全约束，例如“向可出水方向倾斜且不超过容器安全角度”，而不是保存某个机器人第六关节旋转多少度。
3. 事实终止条件不变量：保存动作阶段以目标事实或连续状态变量成立作为边界，例如“检测到水流且杯中液位达到目标”或“cup_contains_water 成立”，而不是保存执行两秒后停止。

由此可将三类资产区分如下：

- 概念：由 P012 归纳形成的可迁移抽象，例如“可盛装容器”“水源区域”“空间导航区域”“可抓取对象”。
- 经验：围绕目标因果事实形成的过程链、因果签名和不变量契约，用于说明在某类空间中应满足哪些前提、如何推进以及以什么事实结束。
- 技能：在经验基础上加入过程模板、偶然绑定、验收记录、审计材料、适配说明和版本信息后形成的可交付能力包。

P010 提供空间语义底图和对象关系，P012 提供概念桥接和典型经验模式，P016 负责阶段跃迁、事实验真和最终成立状态回流。Robot Adapter、机器人 SDK、运动学求解器或厂商控制器负责把不变量落到当前机器人本体上。因此，不同高矮、臂长、底盘和执行器结构的机器人可以共享同一经验，但各自通过本体约束和运动规划生成不同的实际动作。

### 2.2.2 计划与执行事实的边界

规划结果、候选路径、动作序列和轨迹均属于待执行假设，不属于已经成立的世界事实，也不因规划阶段通过一次可行性检查而持续取得执行资格。面向真实世界的执行必须遵循以下统一流通原则：

1. 每个动作增量、路径分段或阶段跃迁，在提交运行时状态前，必须结合当前空间语义、最新环境观测和执行主体物理包络重新验真；
2. 运行时世界状态、障碍物、对象位置、本体能力或治理约束发生变化时，依赖旧快照形成的未执行计划立即失效；
3. 重规划必须从最后一个已经验真的主体状态开始，不得从旧计划的预测状态继续；
4. 局部绕行、替代动作或恢复过程必须同时验证完整过程边界和终止状态，不能只验证中间避让点；
5. 无安全可行路径时，执行主体必须停止、说明阻断原因，并按权限边界请求确认、请求移除障碍或进入恢复，而不得以计划已生成作为继续执行的依据。

因此，规划层回答“下一步可能怎样做”，执行闭环回答“当前这一增量是否真实可提交”，事实层只接收经执行与观测确认后的状态。该原则适用于移动、抓取、操作、工具使用及其他连续物理过程。

### 2.2.3 概念层：任务语言与经验调用之间的公共抽象桥

概念层不是动作脚本库，不是直接执行层，也不是当前世界状态的真值表。其作用是在“人类任务语言”与“经验调用/过程链选择”之间提供一个可复用的公共抽象桥，使系统能够先理解“这类事情在说什么”，再决定“当前是否能做、调用哪条经验、采用哪类过程链”。

概念层至少应保存以下几类公共语义单元：

1. `effect_contract`：该概念通常对应的目标结果或结果事实；
2. `applicability_constraints`：该概念在什么对象类型、空间关系、主体能力或任务前提下适用；
3. `runtime_contingency_hints`：运行时常见偶然因素、重绑定位点或再适配提示；
4. `experience_link_policy`：该概念与哪些经验记录、过程模板或任务链存在复用关系；
5. `governance_hints`：该概念是否天然伴随高风险边界、确认要求或权限约束。

因此，概念层的运行定位应为：

```
自然语言任务 / 教学输入
        ↓
统一交互语义层
        ↓
概念候选解析
        ↓
经验候选 / 过程链候选 / 澄清问题
        ↓
RELL 编排判断 + P016 Runtime 执行
```

概念层输出始终是候选，不直接下发动作，也不直接改写任务期运行时世界状态。概念层只回答“这更像哪类事情、可能调哪些经验、还缺什么语义”，不回答“现在就这么动”。

概念层还必须与当前真值来源解耦。诸如“杯子里现在有没有水”“我手里现在有没有拿着杯子”这类问题，不能靠概念层静态推理或历史记忆臆断，而必须优先从当前任务期运行时世界状态快照、执行闭环事实回传和最新观测中读取。概念层只能提供理解框架，不能替代当前世界状态。

以“从 A 走到 B”为例，概念层沉淀的是“主体位置从起点区域迁移到目标区域”的公共抽象，而不是固定路线。其在必然层对应位移目标成立，在因果层对应到达前提与转移事实，在偶然层才根据“过道里多了一张凳子”这样的当前空间语义决定是绕开、移除障碍还是请求确认。由此，概念层负责复用抽象，执行层负责面对真实世界做本地裁决。

概念层也不应被视为一份静态词典。其更稳妥的生长路径应为“真实经验 -> 概念候选 -> 人工确认晋升 -> 后续复用”，从而保证公共概念来自真实任务闭环，而不是凭空枚举。

端侧概念内化遵循“语言触发、因果定义、状态约束、经验支持”的总原则。语言或其他交互信号只负责触发概念候选；概念的稳定内核来自可复用的对象关系、状态跃迁和因果规律；当前任务期状态决定概念能否落地；经验和过程模板负责补齐实现该跃迁所需的过程。概念解析、经验检索和过程执行不得合并为整句输入到固定脚本的直接映射。

概念层、状态层和经验层之间形成闭环数据流：概念层提出目标事实与适用条件，状态层给出当前已成立事实和缺失前提，经验层与编排层据此形成候选过程，P016 Runtime 完成执行与因果验真后再回写已确认事实。任何概念候选均不直接取得执行权，也不得替代运行时世界状态真值。

任务条件化感知遵循以下统一流通原则：

1. 任务输入应激活与目标事实、缺失前提和安全边界相关的概念子图，并据此形成当前感知需求；
2. 概念可以描述“需要观察什么、预期具有什么关系和功能”，但感知输出只能形成带证据和置信边界的对象或关系候选；
3. 感知候选必须结合当前空间语义、对象关系、时间有效性和可用观测证据，才能绑定为任务期运行时对象实例或当前事实；
4. 基于概念、观测和当前状态形成的事件预安排属于候选因果推演，不属于已经成立的未来事实，也不直接取得执行权；
5. 环境变化、对象移动或观测失效后，依赖旧观测形成的实例绑定和未执行推演必须失效，并从最后一个已验真状态重新观察和编排；
6. 任务相关注意力可以减少无关对象的语义分析，但人体、碰撞、动态障碍、保护策略和其他安全通道不得随任务注意力一并裁剪。

因此，端侧主链进一步形成“任务理解 -> 概念激活 -> 主动感知 -> 空间与实例落地 -> 当前状态 -> 经验检索与因果预演 -> 编排 -> 执行验真”的闭环。概念负责生成感知约束，感知负责提供当前证据，空间语义负责关系定位，状态层负责真值管理，任何一层都不得独自把候选升级为可执行事实。

### 2.3 长程任务与滚动执行上下文

倒水只是 P016 单过程 Runtime 的样品。类似“下楼拿快递并放回玄关柜”的长程任务，不应被建模为一个巨大过程模板，而应建模为任务链 / 过程链：

```
leave_home
→ navigate_to_elevator
→ take_elevator_to_lobby
→ navigate_to_package_station
→ identify_package
→ grasp_package
→ return_home
→ place_package_on_cabinet
```

全局层只维护任务目标、过程链顺序、当前节点、已成立关键事实、后续节点和审计索引。当前执行层只保留当前过程模板、偶然绑定、阶段状态、连续状态变量、验真窗口和恢复上下文。

过去阶段完成后，不再常驻当前执行上下文，只沉淀为：

```
established fact
audit record
experience record
optional skill trace
```

这种 Rolling Execution Context 能避免长程任务把全部历史细节常驻在执行内存中，使有限资源集中于当下过程和紧邻未来。

### 2.4 统一交互语义层边界

交互层的角色不是聊天 UI，而是“候选任务输入 -> 本地理解 -> 本地裁决 -> 状态外化”的语用中枢。其任务是将人类语言、教学输入、状态提问或澄清请求收口为结构化交互对象，并把编排层、执行层和治理层中的关键状态回投给人类。

当前建议结构：

```
人类语言 / 工作流文本 / 教学输入 / 状态提问
        ↓
统一交互语义层
        ↓
Structured Interaction Intent + Concept Resolution
        ↓
RELL 准入判断 / 任务链选择 / 教学回路 / P016 Runtime 执行
```

该层至少需要维护以下几类输出：

1. request_type：执行任务、状态提问、教学输入、澄清追问；
2. goal_fact / task_type / candidate_process_chain；
3. object_constraints / spatial_constraints / sequence_constraints；
4. concept_candidates / concept_activation_reason / concept_binding_status；
5. intent_confidence / clarification_needed / clarification_reason；
6. current_status_view：当前动作、下一步、为什么不能做；
7. deviation_view：是否发生步骤偏离、偏离级别、偏离原因；
8. refusal_source：语义失败、能力失败、治理失败中的哪一类。

如果使用 LLM，其角色仍然只是语义翻译器，而不是执行控制器。LLM 输出应限制在候选语义、候选约束、候选澄清点和候选过程链，不直接裁决、不直接控制机器人、不绕过 Runtime，也不替代治理层做高风险动作的授权判断。

统一交互语义层可以调用概念层做桥接解析，但概念层输出必须继续回到 RELL 做 Schema 校验、经验调用、过程链选择和当前可执行性判断。尤其对于当前状态问答，统一交互语义层只能从任务期运行时世界状态快照、执行闭环事实回传或恢复记录中取值，不得让概念层覆盖实时真值。

RELL 负责校验该结构化输出是否符合 Schema、任务模板是否存在、目标事实是否可产出、空间语义和执行体能力是否满足要求，并在必要时要求统一交互语义层回问用户。

### 2.5 教学层：目标约束与过程约束分离

教学输入不应被简单保存为一个自然语言步骤串，而应至少拆成以下四类内部表示：

1. 目标约束：最终要达成的结果事实；
2. 过程约束：推荐或要求遵循的顺序、依赖前提和关键步骤；
3. 可变区间：允许局部重排、路径替代或表达替换的部分；
4. 风格与服从偏好：强调结果导向、过程导向、严格跟随还是允许局部变通。

因此，教学层在架构中的作用不是“录入步骤”，而是：

```
人类教学
   ↓
目标约束 / 过程约束 / 可变区间 / 偏好约束
   ↓
编排引擎与执行主体共同判断
   ↓
执行 / 偏离 / 恢复 / 追认
   ↓
审计 + 端侧偏好记忆体 + 候选经验沉淀
```

当执行主体无法完成任务时，教学还可以通过受控遥操作进入：人类临时取得当前教学会话范围内的动作候选生成权，但执行主体的本体画像、空间碰撞、治理策略、物理执行边界和事实验真链路保持不变。该过程属于控制来源切换，不是绕过执行主体约束的内核替换。

遥操作产生的按键、绝对坐标、固定时长、关节角和单次本体轨迹只作为当前教学会话的临时执行证据。教学结束后，系统只能从已验真的状态跃迁中编译拓扑关系、动作方向与物理约束、因果前提、事实产出与销毁、终止条件和恢复入口。候选经验必须由执行主体在重新绑定当前空间与本体后自主复做，并同时取得物理事实验真和人类任务验收，才能晋升为本地可信经验；单独的人类表扬或单次遥操作完成均不能替代事实成立。

当前样品阶段，自主预算的默认上限由当前人类偏好/教学偏好决定，P015 负责承接这一层运行时偏好约束；在治理层允许范围内，如人类未显式放开局部变通，则默认采用严格跟随。

这里的偏离追认默认不直接进入公共经验库，而优先进入端侧人类偏好记忆体和审计事件流。只有在多次验证且去个体化后，才考虑提升为更高层次的可复用经验。

### 2.6 双向对流：任务下行与解释上行

RELL 在人机协作中不应只有“任务派发 -> 最终结果”的单向链路，而应形成双向对流：

下行链路：

```
候选任务输入
→ 教学输入
→ 补充描述/翻译/澄清
→ 偏好约束
→ 治理约束
→ 执行主体
```

上行链路：

```
当前理解
→ 失败分类
→ 对象定位结果
→ 当前动作与下一步
→ 偏离说明
→ 恢复说明
→ 结果回报
→ 经验演变状态
```

这意味着执行反馈不仅包括“任务完成没完成”，还包括理解状态、偏离状态、恢复状态和经验固化状态。

## 八、样品当前实现口径：因果层优先于自然语言枚举，交互层优先于脚本枚举

RELL 样品不应将每一种用户说法都保存为一条独立经验。自然语言输入首先应被翻译为目标因果事实；编排引擎再基于当前世界事实和过程事实注册表反向补齐前提事实，形成可执行过程链。

当前最小实现：

```
人类输入
  ↓
目标因果事实
  ↓
当前世界事实(P010空间认知模型)
  ↓
过程事实注册表(P016因果层)
  ↓
反向搜索补齐前提事实
  ↓
过程链执行与事实回流
```

例如，用户输入“到水源处接一杯水”时，系统不应要求经验库中存在同名短链经验，而应翻译为目标事实 `cup_contains_water`。由于 `fill_cup_at_water_source` 的前提包括 `cup_in_gripper` 和 `executor_at_water_source`，系统从当前世界事实中发现杯子位于操作台且夹爪为空，因而补齐：

```
move_to_counter
→ pick_up_cup
→ move_to_water_source
→ fill_cup_at_water_source
```

这一区别决定了 RELL 是“经验技能中枢”，而不是“自然语言命令脚本库”。人工教学形成的经验记录仍然有价值，但其主要作用是补齐缺失过程、校验过程链、沉淀执行后果和更新事实注册表，而不是永久绑定某一句话与某一串步骤。

同样，教学输入也不应被保存为刚性脚本。教学首先应形成候选目标、候选过程约束和局部偏好；执行主体在治理层允许的范围内，可基于自主预算判断是严格跟随、局部重排、请求确认还是拒绝执行。这样样品后续才能从“会跑固定例子”走向“能顺畅协作、解释和学习”。

## 九、2026-07-11补充：端侧概念内化层与云脑分工

在当前架构下，统一交互语义层已经开始承担任务理解、状态提问、教学输入和澄清请求的收口职责，但如果长期完全依赖云端模型解释所有自然语言，执行主体会始终缺少稳定的本地工作语义。为此，需要在统一交互语义层与编排引擎之间补上一层端侧概念内化层，使高频任务语言、高频状态语言和高频教学语言能够沉到本体侧，形成端脑自己的概念、状态和经验闭环。

这一层的定位不是聊天层，不是词典层，也不是直接执行层，而是：

```text
自然语言输入
-> 统一交互语义层
-> 端侧概念内化层(Concept Core)
-> 经验候选/目标事实/状态查询槽位
-> 编排引擎与P017迁移适配主链
-> P016 Runtime执行与事实回流
```

### 9.1 端脑与云脑的职责分工

端脑负责当前真实任务的本地裁决，云脑负责陌生语义和外域经验的补给。二者是主次关系，不是替代关系。

端脑至少负责：

1. 当前任务期运行时世界状态快照的读取与解释；
2. 高频对象、动作、状态、时间和指示语义的本地概念化；
3. 高频状态问答，如“杯子里有没有水”“你现在在哪”“你现在在做什么”“下一步做什么”；
4. 高频动作意图识别，如“接一杯水”“拿起杯子”“去水源处”“绕开障碍”；
5. 教学过程中的即时复述、缺口暴露和当前进度说明；
6. 在不依赖云端的情况下维持最小可工作闭环。

云脑至少负责：

1. 陌生任务或长程复合语义理解；
2. 跨设备、跨场景的外域经验检索；
3. 稀有概念、低频概念和候选过程链补给；
4. 经过验证后的公共概念沉淀与共享。

调用原则应固定为：

1. 端脑优先；
2. 当前状态问答优先读取任务期运行时世界状态快照；
3. 本地概念能够解释时，不调用云脑；
4. 只有在陌生、歧义大、端侧概念不足时，才向云脑发起候选补给；
5. 云脑输出始终是候选，不直接越过编排层和执行层。

### 9.2 端侧概念内化层的内部职责

端侧概念内化层建议统一命名为 `Concept Core`，其最小内部职责包括：

1. 概念单元库：沉淀高频对象概念、动作概念、状态概念、主体概念、时间概念和指示概念；
2. 概念解析器：把自然语言压缩为结构化 `intent_frame`、`query_type`、`action_concepts` 和 `object_concepts`；
3. 状态槽位桥接器：把“现在”“下一步”“有没有水”一类问题映射到任务期运行时世界状态快照字段，而不是回退到长期记忆瞎猜；
4. 经验候选桥接器：把动作概念和目标概念组合为候选目标事实、候选过程链或候选经验记录，再交给编排引擎和 P017 主链做可行性裁决；
5. 云脑补给桥：在端侧概念不足时发起外域补给请求，但返回结果仍需回到端侧候选结构中统一裁决。

### 9.3 概念、状态、经验三者的边界

后续工程中需要始终保持以下边界：

1. 概念回答“这是什么类型的对象/动作/状态”；
2. 状态回答“当前世界里的真值是什么”；
3. 经验回答“通常如何从一个事实走到另一个事实”。

因此，“杯子里当前有没有水”必须由任务期运行时世界状态快照回答；“接一杯水通常如何做”可以由经验层和过程链回答；“接水”这一表达属于哪类任务，则由概念层完成理解与桥接。三者相互配合，但不能互相替代。

### 9.4 与现有专利和层级的边界

这一补充层与现有主线的关系应明确固定如下：

1. 与 P013 的关系：P013 负责统一任务语义翻译，端侧概念内化层是 P013 在本体侧的高频语义沉降实现；
2. 与 P015 的关系：P015 负责偏好记录、偏好附着和自主预算来源，概念层不创造偏好，只解释当前偏好约束如何进入任务理解；
3. 与 P017 的关系：P017 负责真实经验的跨空间跨本体迁移适配，概念层只负责把自然语言稳定映射到目标事实、绑定候选、经验候选和澄清缺口；
4. 与治理层的关系：治理层负责法律法规、物理后果边界、代理权预算和保护策略，概念层不自创治理规则，只负责解释治理触发来源。

### 9.5 当前工程落地方向

端侧概念内化层的第一阶段不追求全语义覆盖，只先做两个最小主干：

1. 状态概念主干：稳定回答“有没有水”“现在在哪”“现在在做什么”“下一步做什么”“手里拿着什么”“当前偏好约束是什么”；
2. 高频动作概念主干：稳定理解“接一杯水”“倒水”“拿起杯子”“去水源处”“去操作台”“绕开障碍”。

在这两个主干稳定后，再向指示语义、教学语言拆层和云脑候选补给扩展。只有当概念、状态、经验三者在端侧真正闭成环，执行主体才会从“依赖外部解释才能工作”成长为“能理解自己、能解释自己、能积累自己”的主体。
## 十、P018母案定位：状态优先的交互式任务执行仲裁

在当前架构迭代基础上，P018 不建议被定义为“交互层专利”或“端侧概念内化专利”，而应被定义为：

**以任务期运行时世界状态快照为第一判断依据的交互式任务执行仲裁母案。**

其主问题不是“如何让系统会聊天”，也不是“如何让系统会做概念理解”，而是：

1. 当自动执行体已经处于运行中时，新的自然语言输入、状态提问、教学纠正、人工叫停或环境变化到来后，系统如何先读取当前任务期运行时世界状态快照；
2. 如何基于该快照判断是继续当前任务、暂停当前任务、暂停并切换任务、合并入当前任务、终止当前任务，还是请求人工确认；
3. 如何在允许续行或切换的情况下，基于当前事实裁剪已满足步骤，而不是回到静态初始状态重做；
4. 如何把上述仲裁结果和当前状态解释回投给人类，形成任务下行与解释上行的双向对流。

### 10.1 与交互层的关系

昨天形成的统一交互语义层可以并且应该写入 P018 母案，但它在母案中的定位应是：

- 为运行时仲裁提供结构化输入；
- 为人类提供状态外化和解释输出；
- 统一承接执行任务、状态提问、教学输入、澄清请求、中断请求和任务切换请求。

因此，交互层在 P018 中不是独立主骨，而是运行时仲裁主链的入口层与外化层。

### 10.2 与端侧概念内化的关系

端侧概念内化与交互层强相关，但当前更适合作为 P018 的强关联实施方式，而不是当前母案的主骨。其职责是：

- 为交互层提供本地高频语义支撑；
- 输出候选概念、候选目标事实、候选过程链和候选澄清问题；
- 在本地优先理解失败时再请求云脑补给。

但其输出仍必须回到基于当前任务期运行时世界状态快照的编排与仲裁层做最终验证，不得直接取得执行权。

### 10.3 与 P016、P017 的边界

P016 负责“一个过程如何可靠执行”，重点在过程模板、阶段跃迁、执行闭环、因果事实验真和恢复。

P017 负责“真实经验如何跨空间跨本体迁移适配”，重点在经验不变量契约、本体能力画像、任务期运行时世界状态快照在迁移适配中的事实对齐和任务后释放。

P018 则负责“新输入如何在当前快照约束下触发继续、暂停、切换、确认或终止等仲裁结果”。因此：

- P016 是执行层主链；
- P017 是迁移适配主链；
- P018 是交互驱动的运行时仲裁主链。

### 10.4 当前保护布局建议

当前最稳的保护节奏是：

1. 用 P018 守住状态优先、交互式输入、运行时事件仲裁和基于当前事实的续跑裁剪；
2. 把交互层、状态外化、教学输入写厚，作为母案的重要实施方式；
3. 把端侧概念内化和云脑候选补给写成强关联但不夺主的说明书内容；
4. 待端侧概念内化形成独立闭环后，再单独拉出分案。

### 10.5 一句话收口

P018 在总架构中的位置不是替代 P016 或 P017，而是把“交互输入如何进入真实执行系统”这一主线独立出来，并明确规定：

**任何新的输入事件，必须先经过当前任务期运行时世界状态快照，再进入编排与执行。**
