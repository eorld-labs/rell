# 研发记录制度

## 一、目的

本项目需要形成连续、可复核、可追溯的研发过程证据。研发记录用于证明项目并非仅停留在概念、撰写或事后整理阶段，而是在持续进行工程化设计、实现、验证和迭代。

研发记录应当服务于：

1. 项目管理。
2. 技术路线复盘。
3. 专利审查中的研发实施证明。
4. 后续融资、合作、许可或争议中的证据链。

## 二、基本原则

1. 每天有推进时，应留下研发记录。
2. 研发记录必须对应具体技术链或专利链特征。
3. 研发记录必须写明输入、过程、输出和验证方式。
4. 研发记录应尽量对应到代码、文档、数据、截图、日志或 Git 提交。
5. 研发记录不得只写空泛结论，应写可复核事实。
6. 重要研发记录应与 Git 提交形成对应。

## 三、每日记录最小字段

每日记录至少包括：

1. 日期。
2. 研发主题。
3. 参与人或执行主体。
4. 今日目标。
5. 对应技术层。
6. 对应专利/技术链特征。
7. 输入材料。
8. 研发过程。
9. 产出文件。
10. 验证方式。
11. 验证结果。
12. 问题与风险。
13. 下一步计划。
14. 关联 Git 提交。

## 四、技术层分类

### 4.1 空间治理与对象秩序层

对应：

- P005 空间对象制度流转
- P006 空间对象治理
- P007 空间制度运行

记录重点：

- 空间对象、区域、权限、候选态、确认流程、审计链、多主体冲突。

### 4.2 空间经验世界模型层

对应：

- P010 空间语义自动生成
- P011 物理经验内化机制
- P012 概念归纳跨域迁移
- P013 任务语义翻译
- P014 执行恢复
- P015 人类偏好

记录重点：

- 空间语义、情境-动作-后果、任务经验、概念迁移、补救经验、偏好记录。

### 4.3 运行时执行治理层

对应：

- 代理权边界治理
- 物理执行链路验真
- 代理权预算
- 来源链
- 计划预适配
- 保护策略声明

记录重点：

- 动作声明、计划预验真、来源链、预算裁决、裁决令牌、物理控制验真、保护策略、审计摘要。

## 五、证据类型

可作为研发证据的材料包括：

1. 需求分析文档。
2. 技术方案文档。
3. API 草案。
4. 数据结构定义。
5. 样例输入数据。
6. 源代码。
7. 测试脚本。
8. 运行日志。
9. 输出 JSON。
10. 审计摘要。
11. Git 提交。
12. 截图、录屏或终端输出。
13. 问题修复记录。

## 六、记录与提交关系

建议每日研发记录与 Git 提交形成如下关系：

1. 每次提交摘要使用中文。
2. 提交正文写明技术对应和证据链。
3. 每日研发记录中填写关联提交号。
4. 若当天只有调研或设计，也应记录输入材料和设计产出。

## 七、推荐工作节奏

每日最小推进：

1. 明确一个小目标。
2. 修改一个文档、对象、规则、代码或测试。
3. 跑一次验证或记录未验证原因。
4. 写一条研发记录。
5. 提交一次中文 Git commit。

## 八、注意事项

1. 不把研发记录写成宣传材料。
2. 不把后补记录伪装成当天记录。
3. 不只记录成功，也记录失败、修正和决策原因。
4. 涉及凭据、令牌、隐私、客户信息时，不写入敏感原文。
5. 证据链优先保留可复现文件路径和命令。
