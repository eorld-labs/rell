## 对应专利/技术链特征

1. P016：过程模板定义物理动作的不变过程结构，样品中体现为 `align -> tilting -> maintain_flow -> return` 四阶段倒水过程。
2. P016：偶然绑定数据将当前环境对象、传感器和阈值填充为运行实例，样品中体现为 `pour_water_process_instance.json` 的 `bound_parameters`。
3. P016：Runtime 基于连续状态变量观测值与绑定阈值判断阶段跃迁，Adapter 不直接判断跃迁条件。
4. P016：目标因果事实通过至少两个独立观测通道形成成立判断，样品中体现为 `physical_liquid_level` 与 `digital_flow_integral`。
5. P016：双通道冲突时不做自动融合，降级进入人工确认，防止系统以单一通道继续执行。
6. P016：失败和未知状态受超时、确认窗口、最大恢复次数和人工升级边界约束。
7. RELL：无经验、无绑定、无能力或无观测通道时输出“不会做”，不进入猜测式执行。

## 研发过程

1. 建立阶段零样品目录 `demo_runtime/rell_sample/`，将其定位为 Schema、接口和 Mock 数据冻结层，不实现完整 Runtime。
2. 新增 Runtime 最小数据结构 Schema，包括运行事件、阶段运行状态、过程实例、准入判断、事实观测结果、最终成立状态、执行轨迹和审计摘要。

4. 新增倒水过程实例和三条 Mock 剧本：成功闭环、无水流失败、双通道冲突。
5. 新增 Robot Adapter 合约，明确 Adapter 只负责采样、降噪、限频、事件推送、快照返回和背压接口，不负责判断阶段跃迁。
6. 新增准入规则文档，明确第一阶段只做规则映射、模板存在性、绑定完整性、双通道验真定义和能力词汇表检查。
7. 新增 Schema 受控变更协议，将“冻结”定义为不能隐式变更，而不是绝对不能变更。
8. 扩展阶段零校验脚本，对全部 JSON 解析、倒水阶段顺序、能力词汇表、Mock 时间基准、双通道成立/冲突和失败事件进行门禁检查。
