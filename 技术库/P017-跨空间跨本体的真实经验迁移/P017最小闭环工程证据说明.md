# P017 最小闭环工程证据说明

## 一、中文摘要

本工程证据包用于固化 P017“跨空间跨本体的真实经验迁移执行”的最小特征闭环。证据链通过可重复运行的验证脚本和结构化 JSON 输出，逐段证明：系统能够获取包含目标因果事实、过程链和经验不变量契约的经验记录；获取当前空间语义数据、本体能力画像和任务意图形成迁移上下文；生成任务期运行时世界状态快照；基于经验不变量契约、当前空间语义数据、本体能力画像和任务期快照生成绑定候选以及执行可行性结果；在可执行时通过开放执行闭环接收因果产出事实和因果销毁事实并更新快照；任务结束后释放任务期快照并写入审计记录。

该证据包的重点不是演示界面效果，而是将经验不变量契约、跨空间跨本体适配、任务期状态隔离、执行事实回传、释放令牌和审计记录等技术点落成可运行、可复核、可对照权利要求步骤的工程输出。

## 二、验证命令

单独生成 P017 最小闭环证据：

```powershell
python demo_runtime\rell_sample\validate_p017_minimal_loop.py
```

全量样品检查并同步刷新证据：

```powershell
python demo_runtime\rell_sample\run_all_checks.py
```

## 三、输出目录

```text
demo_runtime/output/rell_sample/p017_minimal_loop/
```

该目录由 `validate_p017_minimal_loop.py` 生成。每次运行脚本会重新生成证据文件，便于审查意见答复前刷新并核对。

## 四、证据文件索引

### 1. evidence_index.json

用途：机器可读的证据索引。

对应内容：列明每个证据文件对应的技术特征、权利要求步骤、关键字段、代码来源和审查答复用途。

主要技术点：将六段最小闭环从“代码存在”提升为“证据文件、代码函数和审查用途的一一对应关系”。

### 2. 01_experience_record.json

用途：证明待迁移经验记录。

对应权利要求步骤：获取待迁移的经验记录。

关键字段：
- `experience_record.target_causal_fact`
- `experience_record.process_chain`
- `experience_record.causal_signature`
- `experience_record.experience_invariant_contract`

代码来源：
- `api_server.py::build_causal_signature`
- `api_server.py::build_invariant_contract`
- `validate_p017_minimal_loop.py::main`

审查答复用途：用于说明本方案复用的是历史真实经验中的目标因果事实、过程链和可迁移不变量契约，而不是绝对坐标、固定轨迹或普通任务模板。

### 3. 02_migration_context.json

用途：证明当前迁移上下文。

对应权利要求步骤：获取当前迁移上下文。

关键字段：
- `intent_translation`
- `current_space_semantic_data`
- `body_capability_profile`

代码来源：
- `api_server.py::migrate_experience`
- `api_server.py::build_space_context`
- `api_server.py::build_default_body_capability_profile`

审查答复用途：用于说明迁移判断同时受当前空间语义、本体能力画像和任务意图约束，不是离线符号规划或静态规则匹配。

### 4. 03_runtime_world_state_snapshot.json

用途：证明任务期运行时世界状态快照。

对应权利要求步骤：基于当前迁移上下文生成任务期运行时世界状态快照。

关键字段：
- `runtime_world_state_snapshot.runtime_world_state_snapshot_id`
- `runtime_world_state_snapshot.lifecycle`
- `runtime_world_state_snapshot.release_status`
- `runtime_world_state_snapshot.established_facts`

代码来源：
- `api_server.py::build_initial_runtime_world_state`
- `api_server.py::get_runtime_world_state`

审查答复用途：用于说明任务期快照是执行安全和状态隔离结构，并非普通长期世界模型或普通内存变量。

### 5. 04_binding_and_feasibility.json

用途：证明跨空间跨本体适配结果。

对应权利要求步骤：对经验步骤进行跨空间跨本体适配。

关键字段：
- `binding_candidate.generation_basis`
- `binding_candidate.step_bindings`
- `execution_feasibility.result`
- `execution_loop_payload.runtime_world_state_snapshot_id`

代码来源：
- `api_server.py::build_binding_candidates`
- `api_server.py::build_execution_feasibility`
- `api_server.py::build_execution_loop_payload`
- `api_server.py::migrate_experience`

审查答复用途：用于说明本方案不是直接对任务目标做规划求解，而是对既有经验步骤进行跨空间、跨本体绑定和执行可行性判断。

### 6. 05_execution_fact_feedback.json

用途：证明开放执行闭环和事实回传。

对应权利要求步骤：在可执行情况下调用执行闭环并依据事实回传更新快照。

关键字段：
- `execution_dispatch.executor_type`
- `execution_dispatch.fact_feedback`
- `execution_dispatch.runtime_world_state_snapshot.established_facts`
- `execution_dispatch.outcome`

代码来源：
- `api_server.py::dispatch_execution_loop_payload`
- `api_server.py::apply_step_to_runtime_world_state`

审查答复用途：用于说明底层执行器可通过开放接口回传结构化事实状态，事实状态可由 SDK、ROS、仿真执行器、数字执行体或动作策略层返回，并被转换为任务期快照更新。

### 7. 06_release_and_audit.json

用途：证明快照释放与审计。

对应权利要求步骤：任务结束后删除或释放任务期运行时世界状态快照并写入审计记录。

关键字段：
- `release.release_status`
- `release.release_token`
- `audit_record.release_status`
- `runtime_world_state_after_release.runtime_world_state_snapshot.snapshot_lifecycle_state`

代码来源：
- `api_server.py::release_runtime_world_state`
- `api_server.py::get_runtime_world_state`

审查答复用途：用于说明快照释放是防状态污染的任务安全控制，并通过释放令牌和审计记录留下可验证证据。

## 五、典型审查质疑对应关系

### 1. 被类比为 PDDL、任务图或普通任务规划

可引用证据：
- `01_experience_record.json`
- `04_binding_and_feasibility.json`

答复重点：本方案不是直接对任务目标进行规划求解，而是读取历史真实经验记录中的经验不变量契约，并结合当前空间语义、本体能力画像和任务期快照对经验步骤进行迁移适配和可行性判断。

### 2. 被质疑只是普通世界模型或状态表

可引用证据：
- `03_runtime_world_state_snapshot.json`
- `06_release_and_audit.json`

答复重点：任务期快照具有任务生命周期、释放状态和释放令牌，用于当前执行期间的事实对齐和状态隔离，任务结束后释放并进入审计，不作为长期世界模型保存。

### 3. 被质疑执行闭环过宽或事实回传不可实现

可引用证据：
- `05_execution_fact_feedback.json`
- `evidence_index.json`

答复重点：开放执行闭环要求执行器返回结构化事实状态，样品中通过 `fact_feedback` 表示因果产出事实和因果销毁事实，并回写任务期快照；执行器类型可以是 SDK、ROS、仿真执行器、数字执行体或策略层。

### 4. 被质疑“不保存绝对坐标”等负面限定缺少技术效果

可引用证据：
- `01_experience_record.json`

答复重点：经验不变量契约中明确将不可迁移参数排除为复用必要内容，保存的是拓扑关系、物理约束和事实终止条件，从而支持跨空间跨本体泛化。

### 5. 被质疑快照释放只是普通内存释放

可引用证据：
- `06_release_and_audit.json`

答复重点：释放动作产生 `release_status` 和 `release_token`，并写入审计记录；其目的在于防止任务期临时事实污染后续任务，而不仅是普通内存管理。
