# 开源边界与商业闭环策略

## 一、定位

本项目不以机器人本体制造、通用世界大模型训练或端到端机器人大脑为第一目标。

本项目的生态位是：

> 面向机器人、Agent 和物理 AI 的经验世界模型、技能经验库与运行时治理底座。

开源的目的不是把全部商业价值交出去，而是通过开放基础框架、样例实现和验证工具，让更多开发者、发烧友、研究者和早期实干者能够跑出结果，帮助项目获得验证、影响力、社区贡献和事实标准地位。

商业化的重点则保留在行业经验库、技能包、治理引擎、企业部署、API/SDK 服务和专利许可中。

## 二、总策略

本项目采用“开源验证，商业收口”的策略。

### 2.1 开源验证

开源层负责回答：

1. 这套方法能不能跑起来。
2. 经验世界模型是否能形成结构化记录。
3. 动作执行前后是否能形成可验证证据链。
4. 社区能否基于这套框架做出自己的场景样例。

### 2.2 商业收口

商业层负责回答：

1. 行业真实经验如何沉淀。
2. 技能包如何验证、认证、分发和收费。
3. 企业和机器人公司如何稳定接入。
4. 高价值治理策略、审计规则和专利组合如何形成收入。

## 三、建议开放内容

以下内容适合开源：

| 类别 | 内容 | 开源目的 |
|---|---|---|
| 核心对象格式 | 空间对象、经验记录、任务记录、裁决令牌、审计摘要的基础 schema | 建立事实标准 |
| 最小 demo | 文件/空间对象整理 demo、最小运行时 pipeline | 让开发者能跑通 |
| 基础 SDK/API | 基础调用接口、样例请求、样例响应 | 降低接入门槛 |
| 示例数据 | 非敏感、低价值、演示级经验数据 | 帮助学习和验证 |
| 评测脚本 | 计划预验真、来源链、预算裁决、经验回写的基础检查脚本 | 形成可复核结果 |
| 插件模板 | 机器人、Agent、仿真环境、IoT 的适配器模板 | 扩大生态 |
| 教程文档 | 入门教程、场景教程、贡献指南 | 吸引社区参与 |
| 通用技能样例 | 不含行业私有数据的基础技能包样例 | 证明技能包模式成立 |

开源层应当足够真实，不能只是宣传壳；但也不应包含项目最有商业价值的行业数据、治理策略和高质量技能库。

## 四、建议保留内容

以下内容应作为商业资产保留：

| 类别 | 内容 | 商业价值 |
|---|---|---|
| 行业经验库 | 工厂、仓储、养老、教育、园区、家庭、医疗等场景经验 | 形成数据壁垒 |
| 高质量技能包 | 已验证可复用的任务技能、补救技能、偏好技能 | 可交易、可授权 |
| 治理策略引擎 | 来源链、代理权预算、计划预适配、保护策略的高级规则 | 企业安全与合规价值 |
| 企业控制台 | 权限、审计、监控、策略配置、组织管理 | 私有化和订阅收入 |
| 数据闭环工具 | 采集、清洗、标注、回写、评估、版本管理 | 帮助客户持续迭代 |
| 认证体系 | 技能包验证、场景认证、执行体适配认证 | 建立平台权威 |
| 私有化部署 | 企业内网、机器人公司、行业客户部署包 | 高客单价交付 |
| 专利许可 | 母案及后续分案覆盖的关键方法、系统和接口 | IP 收入和谈判筹码 |

## 五、分层产品形态

### 5.1 社区开源版

目标用户：

- 发烧友
- 开发者
- 研究者
- 小型机器人团队
- Agent 应用开发者

能力范围：

1. 跑通最小经验世界模型闭环。
2. 生成经验记录和审计摘要。
3. 接入简单 Agent 或模拟机器人。
4. 制作基础技能样例。
5. 提交社区插件和案例。

### 5.2 专业 SDK/API 版

目标用户：

- 机器人创业公司
- Agent 平台公司
- 仿真平台
- IoT 平台
- 场景集成商

能力范围：

1. 稳定 API。
2. 多执行体接入。
3. 技能包加载。
4. 经验库调用。
5. 运行时治理裁决。
6. 审计输出和日志管理。

### 5.3 企业/行业版

目标用户：

- 工厂
- 仓储
- 园区
- 养老
- 教育
- 医疗
- 家庭服务机器人公司

能力范围：

1. 私有经验库。
2. 行业规则包。
3. 保护策略。
4. 多主体权限治理。
5. 任务审计。
6. 技能包认证。
7. 私有化部署和运维支持。

## 六、可能盈利方式

1. SDK/API 订阅。
2. 技能包售卖。
3. 经验库授权。
4. 行业规则包授权。
5. 私有化部署。
6. 企业控制台订阅。
7. 技能包认证服务。
8. 机器人公司联合方案分成。
9. 开放平台交易抽佣。
10. 专利许可授权。

## 七、为什么机器人越强，本项目价值越大

机器人和 Agent 越强，意味着它们能够执行的动作越多、影响的空间越真实、接触的人群越广、造成的后果越不可忽视。

这会带来三个直接变化：

1. 经验价值上升：机器人越强，越需要复用历史经验、任务经验、补救经验和人类偏好。
2. 治理价值上升：机器人越强，越需要动作准入、权限边界、预算控制、来源链、物理控制验真和审计。
3. 平台价值上升：机器人类型越多，越需要跨本体、跨场景、跨任务的经验格式、技能包和认证体系。

因此，本项目不是与机器人能力提升相冲突，而是随机器人能力提升而放大。

可以概括为：

> 机器人越能干，越需要经验复用；机器人越能影响真实世界，越需要运行时治理。

## 八、与上下游关系

本项目与上下游更适合作为合作关系，而不是竞争关系。

### 8.1 与机器人本体公司

本项目不制造本体，而是为其提供：

1. 经验库。
2. 技能包。
3. 运行时治理。
4. 审计闭环。
5. 行业适配层。

### 8.2 与世界大模型/VLA 公司

本项目不训练通用大模型，而是为其输出的候选计划和动作提供：

1. 计划预验真。
2. 来源链校验。
3. 代理权预算。
4. 裁决令牌。
5. 执行后经验回写。

### 8.3 与场景集成商

本项目不抢全部现场交付，而是为其提供：

1. 技能包标准。
2. 行业经验模板。
3. 数据闭环工具。
4. 审计和治理能力。

## 九、竞争边界

本项目的主要竞争者不是单纯机器人本体公司，而是以下类型：

1. 机器人公司自研中间层。
2. 具身智能开发平台。
3. 仿真和训练数据平台。
4. Agent 治理和 AI 安全平台。
5. 机器人云平台和任务调度平台。
6. 掌握行业现场数据的系统集成商。

应对方式：

1. 通过开源建立事实标准。
2. 通过专利组合保护关键路线。
3. 通过经验库和技能包形成数据壁垒。
4. 通过社区案例形成生态影响力。
5. 通过企业版和私有化部署形成商业闭环。

## 十、开源许可建议

开源代码许可需要结合专利策略谨慎确定。

建议原则：

1. 社区最小实现可采用常见开源许可证，降低使用门槛。
2. 商业模块、行业规则、治理策略和高价值技能包不进入开源仓库。
3. 专利许可范围应与开源代码范围、商业授权范围分开说明。
4. 面向企业客户时另行签署商业授权或专利许可协议。

注意：

> 具体许可证选择涉及专利授权范围，应在正式公开前由专业律师复核。

## 十一、近期执行建议

### 11.1 第一阶段

1. 将当前 demo 整理成社区可运行版本。
2. 提供最小对象 schema。
3. 提供一组示例经验记录。
4. 提供基础评测脚本。
5. 写清楚开源贡献边界。

### 11.2 第二阶段

1. 开放插件模板。
2. 收集社区场景案例。
3. 建立技能包格式。
4. 建立经验库样例。
5. 设计技能包认证流程。

### 11.3 第三阶段

1. 建设开放平台雏形。
2. 上线经验库目录。
3. 上线技能包目录。
4. 接入机器人或 Agent 试点伙伴。
5. 形成商业版 API/SDK。

## 十二、结论

本项目应当把“不重要但需验证的能力”开放出去，让更多人帮项目验证路线、扩展案例、形成影响力。

同时，应当把“能够规模化变现的资产”留在自己手里，包括行业经验库、技能包、治理策略、企业控制台、私有化部署和专利许可。

这条路线的核心价值在于：

> 用开源获得验证和生态，用商业闭环沉淀数据、技能、治理和专利价值。
