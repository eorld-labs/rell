# P019 概念包与端侧闭环设计原则 v1

## 一、基本原则

人类语言是触发概念解析的信号，但不是概念的稳定来源。概念的稳定来源是客观世界中可复用的对象关系、状态跃迁和因果规律。

端侧概念包不是自然语言词典，也不是动作脚本。其作用是把语言、界面操作或其他交互信号转化为可由当前状态约束、可由经验支持、可由 P016 验真的候选因果算子。

概念包由五部分构成：

1. `language_adapters`：语言、表达方式或其他交互模态的触发模式；
2. `semantic_roles`：对象、来源、目标、资源和空间角色；
3. `effect_contract`：`requires`、`produces`、`destroys` 和 `verification`；
4. `applicability_constraints`：对象类型、空间关系、执行体能力和治理边界；
5. `experience_link_policy`：按因果算子、目标事实和缺失前提检索经验或过程模板。

## 二、语言提及与现实落地分离

语义角色至少分别维护：

```text
mention_status: explicit / implicit
grounding_status: resolved / candidate / inferred / unresolved
binding_confidence
candidate_policy
fallback
```

两组状态不得合并：

- `explicit + resolved`：语言明确提及，且已绑定当前世界中的唯一实体；
- `explicit + unresolved`：语言明确提及某类对象，但当前空间中无法唯一定位；
- `implicit + inferred`：语言未指定，但可由当前持物状态或唯一可用对象推定；
- `implicit + unresolved`：语言未指定且不能可靠推定，必须澄清或确认。

例如，“接一杯水”中的“杯”通常只表达容器需求，不当然指向某个现实杯子。系统可先检查当前手持容器，再检查附近唯一可用容器；存在多个候选时进入 P018 确认，不得臆造对象绑定。

## 三、事实契约

以 `fill_container` 为例：

```json
{
  "operator": "fill_container",
  "effect_contract": {
    "requires": [
      "container_held",
      "executor_at_liquid_source",
      "liquid_source_available"
    ],
    "produces": ["container_contains_liquid"],
    "destroys": ["container_is_empty"],
    "verification": [
      "liquid_flow_observed",
      "container_liquid_level_reached"
    ]
  }
}
```

`destroys` 表示状态跃迁后不再成立的事实，不表示物理销毁对象。概念解析阶段只形成预测，不得直接提交 `produces` 或 `destroys`。只有 P016 执行反馈和因果验真通过后，Runtime 才能提交新事实并使旧事实失效。

## 四、端侧轻量闭环

```text
交互信号触发概念候选
→ 概念算子给出目标事实和前提事实
→ 当前任务期快照区分已满足与缺失前提
→ 按目标事实和缺失前提检索经验/过程模板
→ 编排层形成候选过程链
→ P016 执行并完成因果验真
→ Runtime 提交已验证事实变化
→ 成功或失败结果回流概念复用记录
```

经验检索发生在概念解析和当前事实对齐之后，不使用“整句话直接对应某条经验”的脚本式匹配。

当前第一阶段工程已经按该原则实现两类最小落地：当任务期快照显示执行体只持有一个兼容对象，或当前空间模型只有一个满足概念类型且状态可用的默认资源时，隐含角色可形成带依据的 `implicit + inferred` 绑定；当概念事实契约存在未满足前提时，系统按“哪些步骤或经验能够生产该缺失事实”生成候选，并按覆盖缺口数量排序。同构经验候选合并展示，避免重复记录掩盖真正的因果生产者。上述结果仍为候选，不因推定或检索命中而取得执行权。

## 五、示例

“现在去走廊尽头的饮水机接一杯水”可拆为：

- “现在”：即时任务激活约束，进入 P018 仲裁，不是物理动作；
- “走廊”：空间区域类型；
- “尽头”：空间拓扑关系；
- “饮水机”：语言明确提及的液体来源类型，仍需落地到当前空间实体；
- “接”：`fill_container` 抽象因果算子；
- “一杯”：目标容器及目标量约束，容器实体可能仍为隐含绑定；
- “水”：资源类型；
- `container_contains_liquid(container, water)`：候选目标事实；
- 水流和液位观测：P016 验真依据。

若当前未持有容器或尚未到达水源，因果层根据当前快照识别缺失前提，再检索获取容器、导航和接水经验。不同空间、饮水机和机器人可以共享同一概念内核，但采用不同对象绑定、过程模板和执行器实现。

## 六、执行权边界

概念候选固定满足：

```text
candidate_only = true
direct_execution_allowed = false
must_reenter_orchestration_layer = true
```

端侧概念层回答“这是什么状态跃迁、涉及哪些角色、还缺哪些前提”；它不回答“当前立即执行哪条底层动作”。

## 七、第一阶段行为回归场景

第一阶段持续以三类行为验证概念、状态和经验是否真正闭环：

1. 对于不会的任务，区分“对象或场景已识别”与“本地动作概念或经验缺失”，说明不会的原因，询问人类并进入对话教学或边教边动；教学完成后形成经验，再次执行应命中该经验。
2. 对于已知过程的任一阶段目标，根据当前世界状态反向补齐缺失前提。“拿起杯子”应在执行体不位于杯子处时先前往操作台；“到水源处接一杯水”应在未持杯时先获取杯子。
3. 对于执行中的新输入与环境扰动，组合中断和替代任务必须分别解析；持物状态下切换陌生任务应保留当前对象并请求人工确认；可绕障碍只调整局部路线，不破坏主过程链和目标事实。

对应自动验证入口为 `validate_p019_behavior_scenarios.py`，结构化行为报告输出到 `demo_runtime/output/rell_sample/p019_behavior_scenarios/behavior_report.json`。

经验层向概念层提供的背书不得直接采用单次执行记录。进入公共经验库并作为概念形成或晋升依据前，经验应先由 P017/P016 相关迁移适配链编译为类型化绑定槽、拓扑关系不变量、试探方向与物理约束不变量以及事实终止条件不变量。来源空间的具体实体绑定仅作为形成证据，不作为规范契约；绝对坐标、单一本体关节角、固定执行时长和轨迹等污染字段应在入库前被拒绝。

## 八、对象概念形成的六种基础范式

六种范式用于解释概念候选如何形成和发展，不表示六套彼此独立的识别模型。

1. **功能—拓扑绑定**：对象身份的稳定部分来自其可承担的功能角色候选及其与支撑面、容纳空间、通道、执行体和其他对象的拓扑关系。外观负责发现候选，功能与拓扑负责约束解释；两者都不能单独证明当前实例事实。
2. **本质约束与可变特征分离**：跨实例共同出现的感知结构只能先形成不变量候选，还需通过反例、反事实、交互或物理验真判断是否真正必要。颜色、尺寸、材质外观、装饰和具体姿态等应进入 `variable_features`，不得因样本不足被误升为概念本质。
3. **来源—结构—功能关联**：概念可保存典型形成来源、内部结构假设和功能候选之间的因果关联，但三者分别维护证据状态。典型来源不是所有实例身份的必要条件，内部结构不能仅凭表面图像确认，功能不能由语言常识直接写成现实事实。
4. **动作—物体协同**：Affordance 表示对象与当前本体、任务和状态之间可否发生某种交互。`affordance_available = object_contract × body_capability_profile × runtime_state × task_context × governance_constraints`；同一对象面对不同本体或治理边界可以得到不同可用动作集合。
5. **类比—迁移**：陌生对象不必从零创建概念，可先检索结构、功能和拓扑最相近的既有概念作为基底，生成“继承项 + 差异项 + 待验证项”。类比只生成候选，不继承基底概念中未经当前实例验证的功能事实、物理阈值或治理许可。
6. **情境—角色再绑定**：同一对象实例可在不同因果链中临时承担目标、资源、工具、支撑、容器、材料或障碍等角色。角色绑定属于任务期状态，必须受当前目标、空间关系、本体能力和治理约束控制；情境变化不直接修改对象身份和基础概念核。

## 九、对象概念包的分层结构

对象概念包至少区分以下内容：

```text
ObjectConcept
├─ perceptual_anchors           用于发现和召回的感知锚点
├─ structural_constraints       结构与拓扑约束候选
├─ variable_features            偶然和可变特征
├─ affordance_contracts         待本体、状态和治理判断的功能契约
├─ formation_models             典型来源、制造、生长或组装模型
├─ context_role_frames          情境角色候选
├─ interaction_verification     可通过何种动作与通道验真
├─ claim_evidence_states        每条声明的独立证据状态
└─ evolution_lineage            类比基底、差异、纠正、版本和来源链
```

感知锚点可以包含检测器特征、局部视觉描述、触觉签名或可替换嵌入，但它们只用于召回候选，不构成概念本体。对象实例保存当前空间绑定、观测时间和状态；对象概念保存跨实例契约。二者不得以分类标签或单次识别结果合并。

## 十、逐条声明证据与确认范围

概念包不得只维护一个总 `confidence_score`。对象身份、视觉结构、功能、来源、物理边界和当前实例状态分别维护声明及证据，例如：

```json
{
  "claim": "soft_or_buffered_surface",
  "claim_type": "structural_or_functional_candidate",
  "status": "uncertain_occluded",
  "evidence_refs": ["observation_x"],
  "verification_required": "touch_or_deformation_observation"
}
```

真实校准至少区分三种确认范围：

1. `object_identity`：人类或多源证据确认当前对象属于某一身份候选；
2. `visual_invariants`：当前观测支持哪些视觉或空间结构，哪些因遮挡仍不确定；
3. `functional_facts`：对象是否可承重、盛液、移动、坐卧或安全交互，必须由独立物理证据、教学验收或 P016 多通道验真支持。

身份确认和视觉确认不得自动扩张为功能确认。纯视觉候选不得生成量化承重、尺寸、力、温度、角度或安全阈值；外部模型即使把数值标为“待验证”，也应由编译门拒绝，待真实测量后再作为物理声明写入。

## 十一、概念形成与发展流程

```text
多模态观测产生对象实例候选
→ 检索已有概念及类比基底
→ 提取共同约束、可变特征和差异假设
→ 编译对象概念核候选
→ 人类审核身份、结构与边界
→ 当前本体和任务情境绑定角色与 Affordance
→ 低风险交互或 P016 执行验真
→ 逐条更新证据状态
→ 形成端侧候选版本、晋级、降级或失效决定
→ 经多条可信经验和跨主体证据积累后，交由 P012 处理公共概念归纳、合并或拆分
```

外部视觉模型、语言模型、生成图片和公共知识库均属于候选生产适配器。它们可以加速概念草案生成和反例构造，但不得自行审核、自行校准、自行晋级或热加载运行时。合成图只作为视觉先验，用户实拍或机器人相机可以提供现实身份与感知结构证据，物理功能仍需独立验真。

主动探索以信息缺口为输入，但不是无约束试错。任何触摸、推动、施力、搬运或其他试探必须先经过本体画像、风险等级、治理许可和保护策略检查；无权限、未知后果或可能伤害人员与对象时，系统应说明缺口并请求教学或确认。

## 十二、与既有层级的职责边界

1. P019 负责端侧对象和动作概念候选的结构、逐条证据、类比差异、角色帧、当前实例落地和本地生命周期，不负责提交当前世界真值，也不替代 P012 的多经验公共概念归纳。
2. P018 依据当前任务期状态决定继续、暂停、切换、澄清或拒绝，不负责定义对象概念本体。
3. P017 负责把已验真经验编译为跨空间、跨本体不变量并完成迁移适配，不负责把单次视觉观察晋升为公共功能概念。
4. P016 负责真实动作、物理反馈、事实终止条件和多通道验真，是功能事实与状态跃迁成立的执行依据。
5. P6 本体画像提供当前主体的能力、方向、传感器、执行边界和适配起点；Affordance 必须与该画像联合求值。
6. 云脑和大模型只提供陌生语义、类比基底、视觉或因果草案候选，不拥有当前事实解释权与直接执行权。
7. P012 接收多条可信经验或跨主体证据，负责公共概念的归纳、合并、拆分和跨域复用；P019 可以向其提交已审计候选，但不能以单次观测或单个端侧实例直接改写公共概念。
