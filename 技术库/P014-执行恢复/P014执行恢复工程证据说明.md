# P014 执行恢复工程证据说明

## 一、中文摘要

本工程证据包用于固定 P014“执行恢复”在当前 `rell_sample` 主链中的最小可运行闭环。证据链通过独立校验脚本和结构化 JSON 输出，逐段证明：系统将恢复事件以独立恢复记录库存储；在执行过程中发生双通道验真冲突时生成结构化恢复记录；恢复记录能够按单条标识和按任务维度查询，并与审计链路关联；运行时事实冲突会触发再适配并生成指向再适配流程的恢复记录；执行闭环中途遇到硬阻断时，系统能够在不中断整体任务语义的前提下触发步骤级恢复和再适配；同时，多类恢复事件会持续沉淀到恢复记录库中，供后续复核和工程举证使用。

该证据包的重点不是展示异常描述，而是把 P014 的核心技术事实收束成可重复运行、可落盘、可复核、可直接对应审查答复口径的工程输出。


## 四、证据文件索引

### 1. `evidence_index.json`

用途：机器可读的证据索引。  
对应内容：列明每个证据文件对应的技术特征、主张步骤、关键字段和代码来源。

### 2. `01_recovery_library_baseline.json`

用途：证明恢复记录以独立恢复记录库存储，而不是仅停留在运行期内存或审计文本中。  
对应技术点：将恢复经验记录为可查询的结构化补救记录。  
关键字段：

- `recovery_library.recovery_records`
- `recovery_library.schema_version`

### 3. `02_conflict_recovery_record.json`

用途：证明执行过程中发生双通道验真冲突时，系统生成结构化恢复记录而不是仅返回失败状态。  
对应技术点：执行目标动作过程中检测偏离并形成候选补救动作。  
关键字段：

- `conflict_run.audit_summary.outcome`
- `conflict_run.recovery_record.deviation_context`
- `conflict_run.recovery_record.recovery_action`
- `conflict_run.recovery_record.recovery_outcome`

### 4. `03_recovery_query_and_audit_linkage.json`

用途：证明恢复记录能够按单条标识和按任务维度查询，并与审计链路关联。  
对应技术点：对执行中途偏离形成的恢复记录进行结构化存储和关联。  
关键字段：

- `recovery_lookup.recovery_id`
- `task_recoveries.recovery_records`
- `audit_lookup.recovery_record_ids`

### 5. `04_runtime_conflict_readaptation.json`

用途：证明运行时事实冲突会触发再适配，并生成指向再适配流程的恢复记录。  
对应技术点：基于偏离和当前执行上下文确定补救动作并切入恢复流程。  
关键字段：

- `readaptation.runtime_conflicts`
- `readaptation.execution_feasibility.result`
- `readaptation.recovery_record.recovery_action.action_type`
- `readaptation.recovery_record.source_refs.runtime_conflicts`

### 6. `05_stepwise_blocked_recovery.json`

用途：证明执行闭环中途遇到硬阻断时，系统触发步骤级恢复和再适配。  
对应技术点：在执行过程中基于当前偏离切换至候选补救动作或恢复流程。  
关键字段：

- `dispatch.outcome`
- `dispatch.fact_feedback`
- `dispatch.stepwise_readaptation.readaptation_id`
- `dispatch.recovery_record.recovery_action.action_type`
- `runtime_state_after_dispatch.runtime_world_state_snapshot.recovery_record_ids`

### 7. `06_recovery_library_snapshot.json`

用途：证明恢复层以独立恢复记录库沉淀多类恢复事件，支持后续复核和工程举证。  
对应技术点：补救记录独立于单次执行返回结果而持续留存在恢复记录库中。  
关键字段：

- `recovery_library.recovery_records`
- `recovery_ids`
- `recovery_action_types`
- `recovery_task_ids`

## 五、典型审查质疑对应

### 1. 被质疑只是“失败日志”或“异常提示”，没有形成恢复技术层

可引用证据：

- `01_recovery_library_baseline.json`
- `02_conflict_recovery_record.json`

答复重点：P014 不是把异常写成自然语言日志，而是把偏离上下文、恢复动作、恢复结果和来源引用沉淀为独立恢复记录实体。

### 2. 被质疑恢复记录不可复核、不能关联到具体任务

可引用证据：

- `03_recovery_query_and_audit_linkage.json`

答复重点：恢复记录既可以按 `recovery_id` 检索，也可以按 `task_id` 检索，并通过 `audit_lookup.recovery_record_ids` 与审计链路绑定。

### 3. 被质疑所谓“恢复”仍然只是人工重跑或重新规划

可引用证据：

- `04_runtime_conflict_readaptation.json`
- `05_stepwise_blocked_recovery.json`

答复重点：系统在运行时事实冲突和执行中途硬阻断两类场景下，都会生成指向再适配流程的结构化恢复记录，并保留触发原因、推荐动作和步骤级阻断证据，而不是仅结束任务后重新来一遍。

### 4. 被质疑恢复结果不会沉淀为后续可举证资产

可引用证据：

- `06_recovery_library_snapshot.json`

答复重点：多类恢复事件被统一持久化到恢复记录库中，支持后续按记录、按任务和按动作类型复核，证明恢复层已经是独立的工程资产层。
