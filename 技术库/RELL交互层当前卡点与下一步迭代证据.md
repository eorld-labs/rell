# RELL交互层当前卡点与下一步迭代证据

## 一、文档目的

本文档用于固定当前 `rell_sample` 在 P011、P013、P014、P015、P017 工程证据链基本补齐之后，继续向“可持续人机协作执行体”推进时所遇到的真实卡点、边界判断和下一步迭代目标。

该文档不是远期愿景描述，而是一次明确的阶段性迭代基线。后续如果交互层真正跑通，可以将本文件作为“跑通前的卡点证据”与后续实现结果进行对照，证明系统并非一次性写死，而是沿着明确问题逐步迭代形成。

## 二、当前判断

当前系统已经具备以下主链能力：

1. 能将部分自然语言任务翻译为结构化语义请求、目标因果事实和过程链。
2. 能在任务期运行时世界状态快照中维护当前任务事实，并支持状态查询。
3. 能在执行过程中形成执行可行性判断、经验缺口记录、恢复记录和偏好约束。
4. 能通过教学、边教边动、经验迁移和执行恢复形成工程证据闭环。

但在继续推进实际人机协作时，当前最明显的瓶颈已经从“能不能执行”转移为“能不能顺畅协作”。

## 三、当前卡点

### 3.1 任务理解仍不够稳定

当前链路已经能处理一部分固定句式，但当用户表达更自然、更省略、更跳跃时，系统未必总能稳定理解“你到底要它做什么”。

这会导致三个具体问题：

1. 用户说的是执行任务、状态提问、教学、澄清还是临时改口，系统未必总能稳定分类。
2. 用户话语中的对象指代、顺序约束、空间约束和结果目标，未必总能稳定抽取。
3. 一旦任务表达稍微复杂，系统容易退化为“无法匹配可执行过程模板”或只能依赖固定句式命中。

### 3.2 系统内部状态尚未充分“说出来”

当前系统内部已经存在大量有价值的结构化状态，例如：

- `semantic_request`
- `intent_translation`
- `execution_feasibility`
- `experience_gap_record`
- `recovery_record`
- `runtime_world_state_snapshot`

但这些状态虽然存在，并不等于已经形成稳定自然的对话输出层。

这会导致两个问题：

1. 用户很难顺畅知道系统当前理解成了什么、接下来准备做什么、为什么没法继续。
2. 教学过程虽然能推进，但缺少稳定的“我理解了什么 / 我缺什么 / 我现在做到哪一步了”的即时回话。

### 3.3 当前系统还缺少持续对话式教学手感

现在的教学链路已经可以形成经验记录，但从人机交互体验看，仍偏向“接口可用”，还不是“交流自然”。

真实顺滑的教学应当至少呈现为：

1. 用户说一句，系统先复述或确认理解。
2. 系统执行或指出缺失前提。
3. 系统说明当前结果和下一步。
4. 用户继续补一句，系统顺势接上。
5. 完成后系统明确说明该经验已经固化。

如果做不到这一点，系统虽然技术上能形成经验，但在实际协作中会始终显得生硬。

## 四、为什么这是当前最优先卡点

当前阶段继续优先解决交互层，而不是先继续增加复杂动作库，原因有三点：

1. 如果任务理解不稳，新增动作只会扩大误解空间，而不会提升可协作性。
2. 如果系统不能把当前理解和当前状态说出来，用户无法高效纠错，也无法高效教学。
3. 我们当前路线不是“大模型直接连续控制”，而是“语义理解进入编排层，编排层和执行层根据真实世界状态自主判断执行”。既然如此，把内部判断外化为稳定可问答的交互层，就是主链的一部分，而不是附属 UI。

进一步说，随着最小状态问答和最小教学闭环开始跑通，当前真正暴露出来的新瓶颈已经不是“交互层有没有入口”，而是“端侧有没有自己的稳定工作语义”。如果每一次状态问答、动作理解和教学复述都仍然依赖云端重解释，那么执行主体就始终无法形成自己的概念坐标，也无法把直接经验真正沉在本体侧。正因如此，当前阶段把端侧概念内化层补进来，不是偏题，而是交互层走向稳定的必要下一步。

## 五、当前迭代目标

下一阶段的目标，不是让系统“更会动”，而是让系统“更会理解当前任务并把自己的内部状态表达出来”。

本轮应优先收敛以下四类能力：

### 5.1 统一任务入口分类

让系统稳定区分以下输入类型：

1. 执行任务
2. 状态提问
3. 教学输入
4. 澄清或追问

### 5.2 当前任务状态可表达

让系统能围绕当前任务期运行时世界状态快照稳定回答以下问题：

1. 我现在在做什么
2. 我下一步做什么
3. 我为什么这么做
4. 我为什么做不了
5. 当前关键事实是什么

### 5.3 教学过程即时反馈

让教学过程具备连续回话能力，包括：

1. 复述当前理解
2. 提示缺失前提
3. 回报步骤成功
4. 说明下一步动作
5. 最终说明经验已固化

### 5.4 大模型边界继续保持克制

下一阶段如果接入更强语义理解，也仍只允许承担：

1. 长程或陌生任务语义理解
2. 候选结构化语义生成
3. 澄清建议生成

不允许承担：

1. 直接动作生成
2. 连续控制
3. 绕过运行时世界状态快照和编排层直接下发执行

## 六、建议的实现顺序

### 第一阶段：统一交互语义层

目标：把“执行 / 询问 / 教学 / 澄清”真正收口到统一交互层，而不是只在接口上并列存在。

### 第二阶段：状态解释器

目标：把当前 `runtime_world_state_snapshot`、`execution_feasibility`、`experience_gap_record`、`recovery_record` 转成稳定可复用的解释输出。

### 第三阶段：教学对话回路

目标：把“你说一句，我回一句，我再动一步”的节奏做出来，而不是只保留最终入库能力。

### 第四阶段：场景内扰动对话化

目标：当中途加门、放凳子、改偏好或出现运行时冲突时，系统不仅能适配，还能明确说出为什么要绕开、为什么要请求确认、为什么要重新教学。

## 七、第三方顾问意见吸收后的补充约束

在上述基线之外，结合外部顾问对人机协作系统从“能动”向“可教、可问、可商量”跃迁的分析，当前文档进一步收口出以下设计约束。

### 7.1 交互层不是 UI 层，而是语用中枢

交互层承担的不是页面展示职责，而是三类翻译职责：

1. 把模糊自然语言翻译成可执行的任务图谱。
2. 把内部运行时状态翻译成可审计、可追问的人类语言。
3. 把教学和纠错过程翻译成可固化的经验回路。

因此，交互层必须同时订阅编排层的 `semantic_request`、`runtime_world_state_snapshot`、`execution_feasibility`、`experience_gap_record` 和 `recovery_record`，并且能够把这些结构化状态稳定外化，而不能退化为“前端说一套、执行端再理解一套”的松耦合壳层。

### 7.2 任务理解必须引入置信度与降级策略

交互层后续不应只输出“解析后的任务意图”，还应输出：

1. 意图置信度；
2. 是否需要人工确认；
3. 触发确认的原因；
4. 候选解释之间是否存在冲突。

当意图置信度不足，或用户中途插入的新约束与当前任务图冲突时，系统应主动降级到“复述理解 + 请求确认”模式，而不是装懂后直接执行。

### 7.3 任务图谱需要支持动态更新，而不是推倒重来

后续用户中途改口、插入偏好、增加空间约束或更换对象时，交互层应优先在当前任务图谱上做局部修正，并把冲突点外化为：

1. 哪个目标变了；
2. 哪个顺序约束冲突了；
3. 哪个偏好与当前执行计划不一致；
4. 现在需要用户确认哪一种解释。

这一步的重点不是自动猜中，而是把“系统正在怎样重新理解任务”说出来。

### 7.4 状态表达需要区分三个时间尺度

后续状态问答层不能把一个快照硬答所有问题，而应区分：

1. 即时状态：当前动作级状态、当前阻断、当前观测；
2. 任务状态：当前子任务、下一步动作、当前可行性结果；
3. 会话状态：当前教学目标、已确认约束、已积累偏好和交互历史。

当用户提问时，交互层应先判断问题针对哪个时间尺度，再从对应状态源取数并组织答案。

### 7.5 状态回答必须可回溯来源

后续状态表达层不应依赖自由润色，而应优先采用“字段驱动 + 模板化解释”的方式，确保每一类回答都能回溯到明确结构化来源。

本阶段建议采用轻量来源约束，而不是直接引入复杂生成系统：

1. 每类回答绑定固定字段来源；
2. 每句关键结论都能映射到快照、可行性结果、缺口记录或恢复记录字段；
3. 映射不上时明确回答“不确定”或转为请求更新当前状态。

### 7.6 教学固化必须分层，而不能把所有反馈都当作永久规则

后续教学与偏好固化时，应至少区分三层：

1. 硬约束：安全、伦理或绝不越过的边界；
2. 情境偏好：仅在特定空间、对象或任务情境下适用的偏好；
3. 风格偏好：表达方式、确认节奏、说明习惯等可随时改变的偏好。

所有固化结果都应带情境标签，避免把一次性的教学要求直接变成永久通用规则。

### 7.7 当前实现节奏继续保持“先窄后宽”

虽然顾问意见提出了较完整的人机协作交互层蓝图，但当前工程上不宜一次铺满。现阶段仍应坚持：

1. 先做“可问”，再做“可教”；
2. 先做单步教学回话，再做多步教学固化；
3. 先做字段驱动的状态解释，再考虑更强语义生成；
4. 先做高频任务和高频问句，再逐步放宽自然语言开放度。

本阶段目标不是构建一个完整对话系统，而是形成一个能稳定支撑执行、问答、教学和澄清的最小交互闭环。

### 7.8 从“小孩拿书”和“翻译拒绝危险动作”抽象出的第一性原理

结合“3 岁孩子不认识字、不会用左右概念，但能用颜色定位书本”和“翻译把英语任务转成日语后，执行主体仍因危险性而拒绝执行”这两个场景，可以进一步抽象出当前架构最应吸收的四条原则：

1. 外部命令不是最终动作，只是候选任务输入；
2. 候选任务必须在接收方自己的概念、感知和能力坐标系里完成理解；
3. 理解任务不等于批准任务，语义转译层和执行裁决层必须分离；
4. 即使理解已经成立，是否执行、能否执行、是否允许执行，仍由执行主体基于自身条件和后果判断独立决定。

这意味着后续交互层不应把自然语言解析直接等同于执行命令，而应把人类输入先降为“候选任务意图”，再经过以下局部闭环：

1. 接收候选任务输入；
2. 在本地概念和感知坐标系里尝试理解；
3. 若失败，则暴露具体失败原因，例如术语不懂、对象无法定位、空间参照不会用、动作概念不懂；
4. 通过补充描述、翻译、示教或澄清建立共享指称；
5. 在共享指称建立后，再进入本地执行裁决；
6. 输出执行、拒绝或继续追问的决定。

这里最值得吸收的不是“让系统更会猜”，而是让系统能明确说出：

1. 我是没听懂这个词；
2. 还是我无法把这个描述映射到当前空间对象；
3. 还是我理解了，但我没有能力做；
4. 还是我理解了，也能做，但这个动作带来不可接受的后果。

### 7.9 与治理层的边界必须明确

上述“执行主体本地裁决”很容易和治理层混淆，因此必须明确边界：

1. 交互层负责的是候选任务输入、本地理解失败分类、共享指称建立、能力可达性判断、当前状态外化和执行前追问/拒绝说明。
2. 编排层负责的是过程链形成、当前任务状态维护、经验调用、再适配和执行衔接。
3. 治理层对应的 P005、P006、P007 及其后续运行时执行治理能力，负责的是制度规则、对象权限、区域边界、保护策略、代理权预算、来源链和高风险动作的制度性约束。

因此，交互层中的“我不能做”至少要区分三类来源：

1. 语义或感知层面不能做：我没听懂、我不能定位、我不会用这个参照方式；
2. 能力或执行层面不能做：我当前没有这个动作能力、没有前提条件、当前世界状态不支持；
3. 治理或制度层面不能做：该动作越过权限边界、保护策略或高风险治理规则。

交互层可以把这些原因外化给用户，但不应自行创造制度规则，也不应替代治理层做授权判断。凡是涉及对象制度、区域规则、主体权限、保护策略和高风险控制边界的“不可执行”，都应明确标注其来源来自治理层或运行时治理校验结果，而不是来自交互层自身的自由判断。

### 7.10 从“煮方便面”故事抽象出的教学层原则

“教新手煮方便面，而执行主体在不破坏结果的前提下局部重排步骤”的场景，进一步说明教学层不能只保存表层步骤顺序。

在这类教学中，输入通常同时混合了以下几类信息：

1. 目标事实：最终要达成什么结果；
2. 概念说明：工具是什么、对象是什么、装置如何工作；
3. 过程链：推荐或要求遵循的步骤顺序；
4. 约束信息：时长、位置、安全条件；
5. 隐含偏好：更重结果是否达成，还是更重是否按原步骤严格照做。

因此，后续教学层不应把整段教学只压成一个“步骤列表”，而应至少拆成四类内部表示：

1. 目标约束：必须成立的结果条件；
2. 过程约束：步骤顺序、先后关系和依赖前提；
3. 可变区间：允许局部重排、替代表达或路径替换的部分；
4. 风格与服从偏好：是否强调严格照做、是否允许自主变通、何种变通必须先确认。

这个故事说明执行主体不是被动背步骤，而是在目标、约束、后果和授权边界下，对执行路径进行本地裁决的主体。因此，教学层后续必须补入以下机制：

1. 目标约束与过程约束分离机制；
2. 步骤重排或路径替代时的偏离识别机制；
3. 偏离可接受性分级机制；
4. 偏离后的解释与追认机制。

### 7.11 双向对流、自主预算与偏离分级

“先放面还是先开火”这个例子进一步说明，真实协作里不只存在“执行 / 不执行”两态，还存在“偏离原教学轨迹但仍可接受”的中间态。因此，后续双向对流不应只表达任务派发和执行反馈，还应表达执行路径在运行中的演变。

这里建议将双向对流固定为两条并行链路：

1. 下行链路：候选任务输入、教学输入、补充描述、翻译、确认、偏好约束和制度约束向执行主体流入；
2. 上行链路：当前理解、失败分类、对象定位结果、能力判断、偏离说明、恢复说明、结果回报和经验固化向外部主体流出。

在这两条链路之间，执行主体本地还应维护一个“自主预算”或“变通权限”概念，用于限定自己能在多大范围内改变原教学步骤。该预算至少可以分为：

1. 严格跟随：必须按教导顺序执行，不得擅自重排；
2. 局部等价重排：在不破坏目标约束和硬约束的前提下，允许调整局部顺序；
3. 重大改动需确认：一旦涉及关键步骤替换、关键对象替换或关键风险变化，必须先回问；
4. 禁止自主变通：涉及高风险动作或治理边界时，不允许自行调整执行路径。

当前样品阶段，自主预算默认不应由执行主体自行放大，而应在治理层允许范围内由当前人类偏好/教学偏好来设定。也就是说，若人类没有显式放开局部变通，则默认采用“严格跟随”；只有在人类明确给出结果导向或允许局部等价重排的前提下，系统才获得有限变通空间。

与之对应，所有偏离都应至少分为三类：

1. 可接受偏离：目标因果事实、硬约束和治理边界均未被破坏；
2. 需确认偏离：目标仍可能成立，但已经超出原教学者明确表达的过程预期或偏好；
3. 不可接受偏离：破坏目标成立条件、关键前提、安全要求或治理边界。

一旦发生偏离，系统后续应能稳定回答：

1. 我偏离了哪一步；
2. 为什么偏离；
3. 我认为哪些目标因果仍然成立；
4. 这次偏离属于可接受、需确认还是不可接受；
5. 这一判断来自目标约束、过程约束、偏好层还是治理层。

一旦人类对某次偏离作出追认，该追认默认也不应直接进入公共经验库，而应优先写入端侧人类偏好记忆体和审计事件流。只有在后续跨任务、跨情境重复验证后，才考虑将其中稳定部分提升为更高层次的公共经验或概念候选。

### 7.12 由此形成的稳定范式

综合前述几个日常协作故事，教学层与双向对流后续可以固定为以下稳定范式：

1. 外部输入首先不是命令，而是候选任务或候选教学输入；
2. 执行主体先在自己的语言、概念、感知和能力坐标系里尝试理解；必要时可经由概念层建立共享抽象，但概念层只负责桥接理解，不直接执行，也不替代当前世界状态真值；
3. 理解失败时，暴露失败原因并请求补充描述、翻译、指认或示教；
4. 理解成功后，不直接等于执行，而要进入本地执行裁决；
5. 教学输入中的目标约束、过程约束、偏好和治理边界必须分层表示；
6. 自主预算的默认上限来自当前人类偏好/教学偏好，并始终受治理层硬约束盖帽；
7. 执行主体可以在自主预算允许的范围内对局部步骤进行保持、重排、替代或拒绝；
8. 所有执行反馈不仅包括结果，还包括理解状态、偏离状态、恢复状态和经验演变状态；
9. 偏离后的追认默认先进入端侧偏好记忆与审计流，不直接进入公共经验库；
10. 治理层对高风险边界、权限边界和保护策略拥有最终约束权，交互层负责解释来源，不负责自创规则。

## 八、建议的阶段性验证问题

当后续实现交互层时，建议优先用以下问题验证：

1. “到水源处接一杯水，然后倒水。”
2. “你现在准备做什么？”
3. “你为什么不继续？”
4. “杯子里现在有没有水？”
5. “先别拿杯子，等我确认。”
6. “教你：先走到操作台，再拿杯子，再去接水。”
7. “过道被凳子挡住了，你怎么办？”
8. “目标杯子现在拿不到，你为什么做不了？”

如果这些问题能够稳定形成“理解 -> 状态读取 -> 回话 -> 执行或解释”的闭环，说明这一轮迭代已经真正越过当前卡点。

## 九、采纳后的最小交互闭环实现要求

综合当前内部判断与外部顾问意见，下一阶段建议将实现范围压缩为以下八项最小要求：

1. 统一输入分类：稳定区分执行、状态提问、教学和澄清。
2. 意图解析带置信度：每次理解输出都附带置信度和是否需要确认。
3. 冲突主动外化：约束冲突、改口冲突、偏好冲突必须说出来。
4. 状态表达分层：至少能稳定回答“当前动作 / 下一步 / 为什么不能做”三类问题。
5. 回答可回溯：所有关键回答都有结构化来源字段可追踪。
6. 单步教学闭环：形成“复述 -> 执行或指出缺前提 -> 回报 -> 确认”的稳定节奏。
7. 不可执行原因分层：明确区分语义/感知失败、能力/执行失败、治理/制度失败三类来源。
8. 偏离分级与自主预算：明确区分可接受偏离、需确认偏离和不可接受偏离，并给出其预算来源。

如果上述八项跑通，再扩展到多步教学、偏好分层面板和更复杂的自然语言开放输入会更稳。

## 十、迭代完成后的判断标准

后续若交互层跑通，至少应满足以下判断标准：

1. 用户不需要频繁改写成系统专用句式。
2. 系统可以把自己的当前理解和当前状态稳定说出来。
3. 教学过程可以连续推进，而不是每一步都像单独调用接口。
4. 中途扰动和不可执行情形可以被解释，而不是只返回结构化错误字段。
5. 大模型即便接入，也仍停留在语义理解边界内，不越过编排层和执行层。

## 十一、结论

当前 `rell_sample` 的主要矛盾，已经不是“有没有能力执行某个任务”，而是“能否以足够自然、足够可解释、足够可持续的方式与人协作完成任务、教学和纠错”。

因此，下一阶段最值得优先投入的，不是继续扩动作集合，而是补齐交互层，把“听懂任务”和“说出状态”做成主链能力。

这既是后续 DEMO 是否真正顺滑的关键，也是未来接入真实机器人之前，最值得先在数字执行体上跑通的一层。
