# 证据链分层披露与日推进计划

## 一、结论

本项目后续应按“每日小阶段、每阶段可验证、每提交有映射”的方式推进。

同时，专利文本逻辑与代码逻辑的最佳对应关系不建议完整公开。完整证据链应作为内部研发证据、审查答复证据、许可谈判底稿和维权底稿保留在私有仓库；公开仓库只展示可运行的工程能力、数据结构、demo 和普通技术说明。

核心原则：

> 公开项目建立影响力，内部证据链建立权利基础。

## 二、为什么完整证据链不宜公开

完整证据链通常包含：

1. 权利要求拆解。
2. 说明书实施逻辑。
3. 技术特征与代码模块的一一映射。
4. 输入样例、输出证据和验证命令。
5. 审查答复、非正常申诉、分案和许可边界判断。

这些内容如果直接公开，会带来以下风险：

1. 竞争者可以快速看懂本项目真正保护点。
2. 潜在竞争者可以按映射表做规避设计。
3. 合作方或客户在谈判前已经掌握本方底牌。
4. 外部人员可能截取内部推理材料，反向质疑创造性、充分公开或实施效果。
5. 后续分案、许可、维权时，本方策略空间会变窄。

因此，完整证据链应只留在内部 Gitee 仓库，不进入 GitHub 开源仓库。

## 三、推荐的四层披露结构

| 层级 | 保存位置 | 可见对象 | 内容边界 | 作用 |
|---|---|---|---|---|
| L0 内部完整证据链 | 私有 Gitee | 内部团队、代理人、必要法律/专利顾问 | 权利要求级映射、代码文件、输出证据、研发记录、提交号、审查策略 | 应对非正常、审查答复、许可谈判、维权 |
| L1 哈希/时间戳证明 | 私有存证或可信时间戳系统 | 内部掌握，必要时出示 | 文件哈希、提交哈希、生成时间、版本清单 | 证明某日已经形成材料，但不暴露内容 |
| L2 合作披露包 | 单独打包，NDA 后提供 | 机器人公司、行业客户、投资/合作方 | 模块级能力、demo 输出、接口说明、部分脱敏研发记录 | 商务验证和技术尽调 |
| L3 公开开源仓库 | GitHub `eorld-labs/rell` | 公众、开发者、爱好者 | schema、demo、API、普通工程说明、示例输入输出 | 建立影响力、吸引开发者、验证范式 |

## 四、公开仓库允许出现的内容

GitHub 开源仓库可以出现：

1. `README.md`。
2. `schemas/`。
3. `demo_runtime/`。
4. 示例输入输出。
5. 普通工程说明。
6. 非权利要求级的模块说明。
7. 贡献指南和使用方式。

GitHub 开源仓库不应出现：

1. `技术库/`。
2. `研发记录/`。
3. 专利说明书、权利要求书、FTO 检索结论。
4. 非正常审查应对材料。
5. “P010-P015 权利要求 -> 代码函数”的细粒度映射表。
6. 内部商业路线、许可策略、分案策略。

## 五、每日小阶段推进方法

每个工作日尽量推进一个小而完整的技术点。

标准节奏：

1. 选定一个 P 编号或技术链特征。
2. 拆出一个可验证的工程对象。
3. 实现 schema、函数、脚本、API 或 eval。
4. 生成样例输入和输出证据。
5. 更新内部证据链文档。
6. 更新当日研发记录。
7. 使用中文提交摘要提交到内部仓库。
8. 推送到内部 Gitee。
9. 如果属于开源白名单，再通过白名单发布流程同步到 GitHub。

每个阶段必须回答四个问题：

1. 这次对应哪个专利文本逻辑？
2. 这次对应哪段代码逻辑？
3. 这次输出了什么可复现实证？
4. 这次证据落在哪个 Git 提交里？

## 七、提交摘要与正文规范

内部提交摘要必须使用中文，并体现“技术特征 -> 代码证据”的关系。

推荐摘要：

```text
实现P011经验内化记录验证链路
补充P014执行恢复输出证据
建立P013任务语义到技能步骤映射
增加P015偏好反馈约束证据
补充P012概念归纳技能包证据
```

内部提交正文建议固定为：

```text
技术对应：
- 对应技术层：
- 对应专利/技术链特征：
- 对应代码模块：

证据链：
- 输入证据：
- 输出证据：
- 验证方式：
- 研发记录：
```

公开 GitHub 提交摘要应避免出现权利要求级信息，可以使用：

```text
Add experience validation demo
Add recovery record output
Add preference feedback examples
Add skill package export
```

## 八、可替代公开的存证办法

为了避免“证据不公开就没有证明力”的担忧，可以采用以下办法：

1. 在内部仓库保留完整证据链。
2. 为关键证据文件生成 SHA256 哈希清单。
3. 将哈希清单、提交号、日期和验证命令写入内部研发记录。
4. 必要时使用可信时间戳、电子数据存证或公证方式固定形成时间。
5. 对外只披露哈希、摘要、demo 结果或脱敏截图，不披露权利要求级映射。
6. 发生审查、申诉、尽调或争议时，再向特定对象提交完整材料。

这样可以做到：

> 平时不暴露核心证据；需要证明时，可以证明材料在对应日期已经存在且未被篡改。

## 十、执行口径

后续所有阶段按以下口径推进：

1. 代码先服务可运行证据。
2. 文档先服务审查和申诉可解释性。
3. 开源先服务影响力和生态参与。
4. 私有证据链服务权利稳定性和商业许可。
5. 每个小阶段都要形成提交和推送。

最终目标：

> 让项目既能被外部开发者用起来，又能在专利审查、非正常申诉、合作尽调和商业授权中拿出清楚、连续、可复现的研发证据。
