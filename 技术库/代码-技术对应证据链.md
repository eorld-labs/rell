## 一、目的

本文用于建立第一阶段 Demo 代码与项目技术体系之间的对应关系，证明工程实现并非孤立脚本，而是对“空间经验世界模型与运行时治理底座”中若干核心技术抓手的初步实施。

当前代码目录：


## 二、总链路对应

Demo 实现的总链路为：

> 执行计划 -> 计划预验真 -> 动作声明 -> 来源链校验 -> 代理权预算 -> 裁决令牌 -> 受控执行 -> 经验记录 -> 审计摘要

该链路对应项目总纲中的“运行时执行治理层”和“空间经验世界模型层”的第一阶段最小实现。

## 七、结论

第一阶段 Demo 已经初步证明：

1. 自动化执行计划可以在执行前进行步骤级边界预验真。
2. 自动化动作可以携带并验证来源链。
3. 自动化执行主体可以受到代理权预算控制。
4. 裁决令牌可以将治理结果转化为执行模式。
5. 高敏对象可以进入候选态而非正式修改。
6. 来源不明动作可以被阻断。
7. 执行后果可以回写为经验记录。
8. 整个过程可以形成审计摘要。

这构成“空间经验世界模型与运行时治理底座”的第一阶段工程化证据链。

### 9.6 证据意义

1. 证明 P016 不是普通状态机：`maintain_flow` 阶段不仅依赖连续液位，还依赖目标因果事实 `cup_has_water` 的双通道最终成立状态。
2. 证明 Adapter 不判断阶段跃迁，阶段跃迁由 Runtime 基于绑定阈值统一判断。
3. 证明双通道冲突不会被单通道强行覆盖，而是阻断后续阶段并进入人工确认。
4. 证明失败场景不会反复猜测执行，而是受恢复次数和人工升级边界控制。
5. 证明样品可以在不购买机器人、不接真实 SDK 的情况下，先形成可演示、可审计、可迭代的工程底座。

## 十、P016 Runtime 最小 API 证据链

### 10.1 目的

`demo_runtime/rell_sample/api_server.py` 用于证明 Runtime Core 可以通过最小 SDK/API 形态对外暴露。第一阶段只提供准入、运行和审计三个外部接口，避免过早扩展平台接口。

### 10.2 接口范围

| API | 技术能力 | 边界 |
|---|---|---|
| `POST /process/admit` | 判断当前任务是否具备模板、绑定、能力和观测通道 | 不做通用相似度检索 |
| `POST /process/run` | 运行单过程倒水实例并返回 `task_id`、运行状态和审计摘要 | 第一阶段同步返回，后续可改异步任务 |
| `GET /audit/{task_id}` | 查询运行审计摘要 | 第一阶段使用进程内存存储 |
| `GET /process/status/{task_id}` | 查询当前状态快照 | 调试接口，不作为对外承诺接口 |

## 十二、SimulatedRobotAdapter 执行体抽象证据链

### 12.1 目的

`runtime_core.py -> SimulatedPouringRobotAdapter` 用于把样品从“预写时间轴回放”推进到“动作命令驱动的低维执行体”。该执行体不模拟完整机械臂动力学，只抽象倒水任务所需的关键物理状态：壶嘴距离、倾角、水流速度、杯中液位、壶内剩余水量和流量积分。

### 12.2 技术对应

| 模块/函数 | 技术能力 | 对应技术链 |
|---|---|---|
| `SimulatedPouringRobotAdapter.execute_stage_action()` | 接收 Runtime 阶段动作并生成状态流 | Robot Adapter 抽象层 |
| `_simulate_align()` | 将壶嘴到杯口距离逐步缩小 | 连续状态变量驱动阶段跃迁 |
| `_simulate_tilting()` | 将倾角逐步增大，并在有水时产生水流 | 物理动作过程阶段 |
| `_simulate_maintain_flow()` | 根据水流累计杯中液位，并生成双通道事实观测 | 目标因果事实验真 |
| `_simulate_return()` | 回正倾角并停止水流 | 阶段完成与过程闭环 |
| `validate_simulated_robot_sample.py` | 固定模拟执行体三路径期望结果 | 样品验收门禁 |

### 12.3 可验证结果

运行：

```powershell
python .\demo_runtime\rell_sample\validate_simulated_robot_sample.py
```

预期结果：

1. `simulated_success` 输出 `completed`，`cup_has_water=established`。
2. `simulated_no_water` 输出 `requires_human_confirmation`。
3. `simulated_channel_conflict` 输出 `requires_human_confirmation`，`cup_has_water=conflicted`。

### 12.4 证据意义

1. 证明 Runtime 与执行体之间已经通过 Adapter 解耦，后续接真机器人时可以替换 Adapter 而不重写 P016 Runtime。
2. 证明连续状态变量不是死数据，而是可以由执行体状态演化生成。
3. 证明双通道验真可在执行体层生成异源观测：物理液位观测与数字流量积分估计。
4. 证明第一阶段不需要完整机器人仿真，也能验证过程模板、偶然绑定、阶段跃迁和目标因果事实闭环。
