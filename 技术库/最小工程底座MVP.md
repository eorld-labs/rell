# 最小工程底座 MVP

## 一、MVP 目标

本 MVP 的目标不是发布通用世界模型，也不是制造机器人，而是在真实数字环境中跑通一条可记录、可裁决、可回写的自动化执行闭环。

MVP 要证明：

1. 自动化执行体的动作可以在执行前被结构化声明。
2. 多步计划可以在执行前进行边界预验真。
3. 动作来源链可以作为执行准入条件。
4. 代理权预算可以被扣减、冻结或降权。
5. 动作执行后可以形成 `{情境, 动作, 后果}` 经验记录。
6. 审计摘要可以串联计划、动作、裁决和后果。

## 二、推荐 MVP 场景

优先选择真实数字环境，避免一开始被机器人硬件复杂度拖住。

推荐场景一：空间内容管理 Agent

任务示例：

> 根据用户意图整理一个三维空间或记忆空间中的对象、内容卡片、媒体资料和权限状态。

可执行动作包括：

1. 移动内容对象。
2. 修改对象标签。
3. 将对象设为私有或共享。
4. 生成候选布置方案。
5. 调用外部 API 获取补充信息。
6. 对敏感对象执行候选态而非正式修改。

推荐场景二：文件系统/知识库 Agent

任务示例：

> 整理某个项目目录，分类文档，生成索引，移动低风险文件，对高风险文件只生成建议。

可执行动作包括：

1. 读取文件元数据。
2. 生成分类计划。
3. 移动或重命名低风险文件。
4. 对敏感文件生成候选变更。
5. 记录执行结果。

推荐场景三：智能家居 API 模拟

任务示例：

> 根据空间情境和用户偏好调整灯光、门锁、空调或插座状态。

可执行动作包括：

1. 查询设备状态。
2. 生成控制计划。
3. 对低风险设备直接控制。
4. 对门锁、厨房设备等高风险动作触发验真。
5. 记录后果和人类反馈。

## 三、MVP 主流程

MVP 主链路如下：

1. 用户输入任务。
2. 系统生成执行计划。
3. 计划转化为步骤摘要。
4. 执行计划边界预验真。
5. 为每个动作生成动作声明。
6. 校验动作来源链。
7. 校验代理权预算。
8. 生成裁决令牌。
9. 执行动作或限制动作。
10. 采集执行后果。
11. 生成经验记录和审计摘要。

## 四、MVP 模块

### 4.1 Planner Adapter

负责接收外部 Agent、大模型、规则引擎或人工输入生成的执行计划。

早期可以不自研复杂规划器，直接允许人工编写计划或调用现有模型生成计划。

输入：

1. 任务目标。
2. 当前空间/文件/设备状态。
3. 可用工具列表。

输出：

1. `ExecutionPlan`
2. `StepSummary`

### 4.2 Plan Precheck Service

负责执行计划边界预验真。

检查维度：

1. 是否访问受限对象。
2. 是否包含不可逆动作。
3. 是否超出工具调用预算。
4. 是否超出时间窗口。
5. 是否对高风险对象执行正式修改。
6. 是否需要人类确认。

输出：

1. `PlanPrecheckResult`
2. 推荐重新规划方向。

### 4.3 Provenance Service

负责来源链生成和验证。

检查维度：

1. 是否存在根授权。
2. 是否存在任务来源。
3. 输入来源是否可信。
4. 工具调用理由是否与任务目标一致。
5. 前序裁决是否有效。

输出：

1. `ProvenanceChain`
2. 来源链状态。

### 4.4 Agency Budget Service

负责代理权预算治理。

预算维度：

1. 工具调用次数。
2. 高风险动作次数。
3. 数据读取范围。
4. 数据外发范围。
5. 自主执行等级。
6. 候选态和正式态变更额度。

输出：

1. 预算适配结果。
2. 预算裁决对象。

### 4.5 Decision Gateway

负责整合计划预验真、来源链、代理权预算和治理规则的结果。

输出：

1. 放行。
2. 限权放行。
3. 参数裁剪。
4. 候选态执行。
5. 人类确认。
6. 阻断。
7. 重新规划。

### 4.6 Execution Adapter

负责真正执行动作。

早期可接入：

1. 文件系统操作。
2. 数据库操作。
3. 空间对象状态修改。
4. Mock 智能家居 API。
5. 真实智能家居 API。

### 4.7 Experience Store

负责保存经验记录。

至少保存：

1. `ExperienceRecord`
2. `TaskExperienceRecord`
3. `RecoveryExperienceRecord`
4. `PreferenceRecord`

### 4.8 Audit Console

早期不需要复杂后台，但需要可查看：

1. 任务计划。
2. 步骤预验真结果。
3. 来源链。
4. 预算扣减轨迹。
5. 裁决令牌。
6. 动作后果。
7. 经验记录。

## 五、MVP 验收标准

MVP 完成不以界面华丽为标准，而以闭环证据为标准。

验收标准：

1. 能导入一组空间对象或文件对象。
2. 能接收一个任务目标并生成多步计划。
3. 能识别至少一个越界步骤。
4. 能为动作生成来源链。
5. 能基于预算限制某个动作。
6. 能将低风险动作执行到真实数字环境。
7. 能将高风险动作转为候选态、确认态或阻断态。
8. 能记录动作后果。
9. 能把后果回写为经验记录。
10. 能生成审计摘要。

## 六、后续扩展

MVP 跑通后，再按以下顺序扩展：

1. 加入人类偏好记录调用。
2. 加入补救经验记录。
3. 接入真实智能家居设备。
4. 接入低成本小车或机械臂。
5. 加入物理控制因果验真。
6. 加入保护策略声明。
7. 加入多主体空间治理。

## 七、避免误区

1. 不要一开始做完整机器人大脑。
2. 不要一开始追求复杂模型训练。
3. 不要只做静态数据集。
4. 不要把治理做成事后日志。
5. 不要把经验记录做成普通操作日志。
6. 不要把空间对象只当展示对象。

MVP 的核心是执行闭环，而不是展示概念。
