# RELL 状态优先与事件仲裁原则

日期：2026-07-11

## 一、原则

物理智能的默认出发点不是“任务”，而是“当前世界状态”。

在 RELL 体系中，这条原则具体表现为：

1. 任意新的输入先被视为事件；
2. 任意新的事件先读取当前任务期运行时世界状态快照；
3. 只有在读取当前世界状态之后，系统才判断下一步动作；
4. 任务、概念、经验、交互都不能绕开当前世界状态直接决定最终执行。

## 二、为什么必须这样

物理世界不是静态脚本环境，而是持续变化的现实环境。

因此：

- 已经拿在手里的杯子，不能被系统当成还在桌上；
- 已经成立的事实，不能被旧任务链条忽略；
- 中途叫停、新命令插入、环境扰动出现时，不能继续沿用旧推导；
- 新事件必须与当前持物、当前位置、当前障碍、当前目标状态一起判断。

如果不这样做，系统就会出现“明明杯子已经在手上，却还要重新去拿一次”的伪智能行为。

## 三、对各层的约束

## 二点五、命令的第一性解释

在 RELL 中，命令首先表达的不是“脚本步骤”，而是“希望当前世界进入什么状态”。

例如：

- “拿起杯子”首先对应目标事实 `cup_in_gripper`
- “到水源处”首先对应目标事实 `executor_at_water_source`
- “接一杯水”首先对应目标事实 `cup_contains_water`

随后系统再根据当前世界状态反推缺失前提，而不是把原句直接当作不可拆解的固定动作脚本。

因此统一主线应为：

1. 解析显式动作或目标表达；
2. 投影为目标事实或目标状态；
3. 结合当前运行世界状态补齐前提链；
4. 仅将补齐后的候选链交给编排层和执行层。

这条规则必须是通用规则，而不是针对某个句式的特判。否则今天修好“拿起杯子”，明天换成“到服务位”“放下水壶”“回到门口”又会重新失效。

### 1. 交互层

交互层不能把自然语言直接翻成执行。

交互层只能输出：

- 当前输入是什么事件；
- 需要补充什么共享参照；
- 当前事件更像继续、暂停、切换还是澄清；
- 当前解释依赖了哪些运行时事实。

### 2. 概念层

概念层不能替代当前世界状态。

概念层负责：

- 把高频语义映射成可复用抽象；
- 生成候选概念、候选链路和候选解释；
- 为编排层提供语义桥接。

概念层不负责：

- 直接声明某个事实当前已经成立；
- 直接下发最终动作；
- 绕过运行时快照决定下一步。

### 3. 经验层

经验层提供的是可迁移结构，不是当前真值。

经验层可以说：

- 完成某类目标通常需要哪些前提；
- 某类过程通常按什么顺序推进；
- 哪些步骤可被当前事实裁剪掉。

经验层不能说：

- 当前对象一定还在原位；
- 当前本体一定还空手；
- 历史任务中的绑定值仍然直接有效。

### 4. 编排层

编排层必须成为“当前世界状态 + 新事件”的统一仲裁者。

编排层至少要能输出以下结果：

- `continue_current_task`
- `pause_current_task`
- `pause_and_switch_task`
- `merge_into_current_task`
- `abort_current_task`
- `request_human_confirmation`

### 5. 云脑 / 大模型层

云脑和大模型只能补语义，不得替代当前世界判断。

它们可以提供：

- 长程任务理解；
- 陌生概念补给；
- 候选经验召回；
- 候选澄清问题。

它们不能提供：

- 绕过本地快照的最终动作决定；
- 对当前物理事实的拍脑袋认定；
- 对底层执行的直接接管。

## 四点五、从输入细节到第一性原理的统一翻译

为避免系统只会“修一个句子、会一个句子”，所有新的自然语言输入都应先被翻译为以下两类结构中的至少一种：

- 目标事实；
- 显式步骤所投影出的结果事实。

例如：

- “拿起杯子”不应被当作孤立技能名，而应投影为 `cup_in_gripper`；
- “到水源处”不应被当作局部导航口令，而应投影为 `executor_at_water_source`。

随后，编排层必须基于当前任务期运行时世界状态快照判断：

- 该目标事实是否已经成立；
- 达成该目标事实还缺哪些前提；
- 历史链路中哪些步骤已可被当前事实裁剪；
- 当前手持状态是否与新候选链兼容。

这样系统处理的就不是“这句话像不像某个预设命令”，而是“当前世界离目标事实还差什么”。

## 五、最小判断顺序

后续主干实现统一遵守以下顺序：

1. 读取当前任务期运行时世界状态快照；
2. 接收新事件；
3. 判断事件对当前活动链、挂起链和候选链的影响；
4. 输出仲裁结果；
5. 基于最新快照重新推导下一步。

## 六、典型例子

### 例 1：拿着杯子后继续接水

若当前快照中 `cup_in_gripper` 已成立，则“去水源处接水”只能从当前快照继续推导，不能再回到“去拿杯子”。

### 例 2：中途叫停

若当前执行到一半收到“别做了”，则系统应先把该输入识别为中断事件，并据当前快照决定：

- 直接暂停；
- 进入待确认；
- 先处理手持对象；
- 或再切换到新候选链。

### 例 3：换任务

若当前手里还拿着杯子，又收到新的任务指令，则不能先假设世界已经清空。系统必须把“手持杯子”这一现实状态纳入新的任务仲裁。

## 七、对当前样品的直接要求

当前样品后续迭代统一遵守：

1. 所有任务入口先查当前快照；
2. 所有新命令先走事件仲裁；
3. 所有候选链允许按当前事实裁剪；
4. 所有状态问答优先来自当前快照；
5. 所有云脑输出都必须回到本地编排层重新裁决。

这条原则不是一个局部补丁，而是 RELL 走向真实物理智能的主干约束。
