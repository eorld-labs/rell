# RELL 样品落地 0 到 1 计划

## 一、阶段目标

当前阶段不再以“讲清楚理论”为目标，而以形成可演示、可联调、可进入基金或产业对接库的样品为目标。

样品应当证明三件事：

1. RELL 能把真实世界任务拆成可复用的过程模板。
2. RELL 能把过程模板绑定到当前环境、对象、传感器和阈值。
3. RELL 能根据因果事实验真结果推进、恢复、拒绝执行或回写经验。

一句话目标：

> 做出一个最小但完整的真实世界经验引擎样品，让外部合作方看到它能跑、能解释、能联调、能扩展。

## 二、样品定义

第一阶段样品不要求制造机器人本体，也不要求训练大模型。样品应当是一个可运行的软件系统，包含：

1. 一个可视化或命令行演示界面。
2. 一组过程模板示例。
3. 一组偶然绑定示例。
4. 一个任务输入和计划生成流程。
5. 一个阶段推进模拟器。
6. 一个因果事实验真模块。
7. 一个“不会做/不可执行”准入判断模块。
8. 一个经验记录和审计输出模块。
9. 一个可供机器人公司或仿真系统调用的 API。

样品不追求覆盖很多任务，只追求一个闭环跑通。

推荐首个样品任务：

> 倒水。

第一阶段只支持单过程实例，不做“倒水并端到指定位置”的多动作过程链。端到指定位置、抓取杯子、移动杯子等动作进入阶段二。

倒水样品只用于展示：

- 过程模板。
- 连续状态变量。
- 阶段跃迁。
- 偶然绑定。
- 因果事实验真。
- 最终成立状态回流。
- 失败恢复和经验回写。

第一阶段倒水不做真实液体仿真，也不做真实视觉识别。水流、液位、倾角等变量由离散事件、模拟曲线或手工数据文件驱动。验收标准是展示阶段跃迁和事实验真闭环，不要求看起来像真水。

## 二点五、架构冻结原则

为避免样品做出后发生大规模重构，本阶段先冻结以下边界。

### 2.5.1 RELL 不做底层机器人执行模型

RELL 不直接实现底层运动规划、轨迹生成、伺服控制、抓取控制、避障导航和电机驱动。

这些能力由机器人厂商 SDK、ROS/MoveIt、仿真平台或后续合作方提供。

RELL 本阶段自己实现的是：

- 过程模板运行。
- 偶然绑定解析。
- 阶段跃迁判断。
- 因果事实验真。
- 能力准入与不会做机制。
- 恢复决策。
- 经验回写和审计。
- Robot Adapter 标准接口。

### 2.5.2 Robot Adapter 是硬边界

RELL Runtime 只通过 Robot Adapter 与机器人或仿真系统交互。

Robot Adapter 至少提供：

- `report_capabilities()`：报告机器人能力和可用传感器。
- `execute_stage_action(stage, context, callback)`：异步执行当前阶段动作或模拟动作，并通过回调或事件返回状态。
- `subscribe_state_variables(process_instance_id, handler)`：订阅连续状态变量事件。
- `subscribe_observation_channel(fact_id, handler)`：订阅事实验真的观测通道事件。
- `pause_streams(process_instance_id)`：可选，向 Adapter 发送背压信号，暂停非必要状态事件推送；MockRobotAdapter 可空实现或暂停事件回放。
- `resume_streams(process_instance_id)`：可选，恢复状态事件推送；MockRobotAdapter 可从当前阶段最新剧本位置继续。
- `get_latest_snapshot(process_instance_id)`：可选，返回最新连续状态变量快照，用于人工等待恢复后重新对齐状态。
- `stop(reason, callback)`：异步停止执行，`reason` 可为 `cancelled`、`failed`、`manual_stop` 等，callback 可选，用于返回最终停止状态。
- `request_human_confirmation(prompt, callback)`：异步请求人工确认，并通过回调或事件返回确认结果。

第一阶段实现 `MockRobotAdapter`，后续再实现 `RosRobotAdapter`、`SimulationAdapter` 或厂商 SDK Adapter。

`read_continuous_state()` 和 `read_observation_channel()` 只能作为调试或兼容接口，不作为 Runtime 主循环依赖。

### 2.5.3 治理层只做挂点，不做满协程

样品阶段只保留治理挂点：

- 计划准入。
- 来源链/授权检查。
- 物理因果验真。
- 恢复或人工介入。
- 后果验真与审计。

不在第一阶段实现完整代理权预算、来源链传播、多主体空间治理和高频治理协程。相关能力通过接口位置预留，避免样品被治理系统拖重。

其中“来源链/授权检查”在第一阶段只保留接口占位，不实现来源链解析、授权令牌校验或代理权预算计算。

### 2.5.4 因果编排先做规则匹配

样品阶段不做复杂规划器。第一阶段只支持单过程实例，不运行多动作过程链。

`produces_fact` 和 `destroys_fact` 先作为审计输出字段，不参与运行时编排。

多动作过程链、并行分组、过程链断裂回溯进入阶段二或后续分案方向，不作为第一阶段必做能力。

### 2.5.5 Runtime 采用事件驱动状态机

P016 Runtime 不允许实现为阻塞式 `while` 死循环。

真实世界中，连续状态变量来自流式传感器数据，而阶段跃迁是离散事件。为避免 CPU 占用过高、突发失败响应迟缓和后续难以接入真实机器人，Runtime 必须采用事件驱动或响应式状态机。

本阶段约定：

- Robot Adapter 负责采样、降噪和限频。
- Robot Adapter 向 Runtime 推送 `state_update`、`observation_update`、`failure_event` 等事件。
- Runtime 接收事件后更新 `stage_runtime_state`，并基于绑定后的连续状态变量观测值与阈值比较结果，判断是否触发阶段跃迁、恢复、验真或停止。
- Runtime 不直接高频读取硬件传感器。
- Adapter 不直接判断阶段跃迁条件是否满足。
- 如需要拉取数据，只能通过 Adapter 提供的非阻塞接口完成。
- 每个事件必须包含 `event_id`、`sequence`、`timestamp`、`process_instance_id` 和 `stage_id`。
- Runtime 按 `sequence` 或 `timestamp` 处理事件，丢弃重复事件和明显乱序事件。
- Runtime 在等待人工确认时进入 `awaiting_human_confirmation` 状态；该状态下不触发新的阶段跃迁，只接收 `human_confirmation` 或 `stop` 事件。第一阶段不单独区分 `cancel`，取消操作统一表达为 `stop(reason="cancelled")`。
- Runtime 进入 `awaiting_human_confirmation` 状态时，应向 Adapter 发送背压信号或挂起非必要状态订阅，避免 `state_update` 事件持续堆积。
- Runtime 从人工等待状态恢复时，应清空或标记丢弃等待期间积压的中间状态事件，并从最新状态快照重新开始消费。

Runtime 内部必须实现串行事件队列或 Actor Model：

- 所有 Adapter 推送事件先进入 Runtime Event Queue。
- Runtime 主状态机单线程消费事件。
- 任何模块不得并发写入 `stage_runtime_state`。
- 高频 Mock 数据必须经过 Adapter 限频后入队。
- `execution_trace` 必须记录事件时间戳、事件序号、消费顺序、跃迁前状态、跃迁后状态和触发原因。

### 2.5.6 核心数据结构先冻结再编码

阶段零必须先输出并锁定核心数据结构定义，再开始业务代码开发。

必须冻结的结构包括：

- `task_intent`
- `process_template`
- `occasional_binding_data`
- `process_instance`
- `stage_runtime_state`
- `fact_observation_result`
- `final_establishment_state`
- `admission_decision`
- `execution_trace`
- `audit_summary`

每个结构必须至少提供：

- JSON Schema 或等价结构定义。
- 倒水任务完整实例。
- 必填字段和可选字段说明。
- 字段版本号。
- 兼容策略。

兼容策略固定为：

- minor 版本号递增：允许新增可选字段，旧数据仍可运行。
- major 版本号递增：字段语义变更或必填字段变化，旧数据必须重新生成。

Schema 冻结不是禁止变更，而是禁止偷偷变更。若开发中发现 Schema 缺失必要字段，允许按受控变更协议修改：

1. 更新对应 JSON Schema。
2. 更新倒水 Mock JSON。
3. 更新 schema version。
4. 评估并记录对 Runtime、验真器、经验回写和审计输出的影响。

原则上，阶段零未通过 Mock JSON 全链路流转验证，不进入 Runtime 业务代码开发。

阶段零硬门槛：

- 倒水任务完整 Mock JSON 必须跑通最小数据流。
- 双通道 Mock 信号定义必须完成，并至少包含一个冲突配置。
- 未达到上述两项，不进入阶段一开发。

### 2.5.7 因果验真必须有超时和确认窗口

因果事实验真不得无限等待 `unknown` 状态。

每个目标因果事实必须配置：

- `verification_timeout_ms`：最长等待时间。
- `confirmation_window`：持续时长、连续确认次数或积分窗口。
- `confidence_threshold`：成立判断的最低置信度。
- `on_timeout`：超时后的动作，例如转为 `not_established`、触发恢复、请求人工确认或中止过程。
- `max_recovery_attempts`：同一阶段最大恢复次数。
- `recovery_escalation`：超过恢复次数后的动作，例如 `request_human_confirmation` 或 `stop`。

如果动作完成后事实成立存在物理延迟，验真器应先进入等待窗口；窗口结束仍未成立时，再输出 `not_established` 或 `unknown_timeout`，避免演示卡死。

恢复流程不得无限循环。超过 `max_recovery_attempts` 后，Runtime 必须进入人工确认、停止或不可继续执行状态。

### 2.5.8 第一阶段不依赖 LLM 自由输出

第一阶段样品的意图解析不依赖 LLM 自由生成结构化 JSON。

推荐做法：

- 用规则映射或硬编码 JSON 模板覆盖倒水样品任务。
- 对 `task_intent` 做 Schema 校验。
- 未通过校验的输入进入不可执行路径。

如后续引入 LLM API，必须增加：

- 固定输出 Schema。
- Output Parser。
- 字段别名归一化。
- 重试机制。
- 无法解析时的拒绝执行策略。

LLM 只作为翻译层，不允许把未经校验的 LLM 输出直接传入绑定解析器或 Runtime。

### 2.5.9 API 必须区分有状态任务和无状态查询

倒水任务是多阶段运行过程，不能把 `POST /process/run` 设计成含义不清的一次性同步接口。

样品阶段 API 约定：

- `POST /process/run` 创建异步执行任务，返回 `task_id`、初始阶段和当前状态。
- `GET /process/status/{task_id}` 查询当前阶段、运行状态、最近事件和等待中的验真结果。
- `POST /process/events/{task_id}` 可选，用于外部 Adapter 推送状态事件。
- `GET /audit/{task_id}` 只用于查询已形成的审计摘要，不承担实时状态查询。

所有长过程必须具有任务会话状态。调用方超时不等于 Runtime 立刻中止，是否中止必须通过明确的 `stop(reason)` 接口表达；取消操作统一使用 `stop(reason="cancelled")`。

### 2.5.10 MockRobotAdapter 必须支持故障注入

MockRobotAdapter 不能只模拟理想成功路径。

第一阶段 Mock 必须通过配置文件支持至少以下故障：

- 倾斜达到上限但无水流。
- 杯子偏移或滑落。
- 传感器短时遮挡。
- 数字通道和物理通道冲突。
- 通道一与通道二判断冲突。
- 阈值过严导致阶段无法跃迁。
- 观测延迟导致事实暂时 unknown。

样品必须同时跑通成功路径和至少两个失败路径，确保恢复、验真、超时和不会做机制在纯软件阶段已经被测试。

MockRobotAdapter 不应在代码中硬编码复杂物理逻辑。第一阶段采用时间轴事件脚本或 JSON 配置驱动 Mock，例如：

```json
[
  { "time_ms": 0, "event": "stage_started", "stage_id": "tilting" },
  { "time_ms": 800, "event": "state_update", "variable": "tilt_angle", "value": 30 },
  { "time_ms": 1200, "event": "state_update", "variable": "water_flow_rate", "value": 3.2 },
  { "time_ms": 2500, "event": "inject_fault", "fault_type": "sensor_occlusion" }
]
```

MockRobotAdapter 只负责按剧本回放事件、注入故障和限频推送，不演变为微型物理仿真器。

Mock 时间轴中的 `time_ms` 以当前阶段启动事件 `stage_started` 为相对零点，不以系统启动时间为绝对零点。Mock Adapter 应在收到 Runtime 阶段跃迁事件后，再启动对应阶段的剧本时钟，避免 Mock 数据与 Runtime 当前阶段脱节。

### 2.5.11 双通道 Mock 必须异源

即使在 Mock 环境中，目标因果事实验真也不能做成“同源假双通道”。

第一阶段至少为“杯中有水”设计两个不同信号来源或不同计算路径：

- 通道一：基于深度相机或液位传感器读取模拟液位高度曲线。
- 通道二：基于水流速度与时间积分，并结合倾角变化估计注入量。

通道一可视为物理观测通道，通道二可视为数字估计通道。

审计输出必须显式展示：

- 每个通道的原始输入。
- 每个通道的成立判断。
- 通道一致或冲突结果。
- 最终成立状态。

第一阶段冲突仲裁策略固定为降级处理：当双通道结果为 `conflicted` 时，不做复杂自动融合，不采信单一通道继续执行，而是阻断后续阶段并触发 `request_human_confirmation`。多传感器融合、置信度加权仲裁和自动选择可信通道进入后续版本。

人工确认返回后，若操作员选择继续，Runtime 清空或丢弃等待期积压事件，并从最新状态快照重新进入验真窗口；若操作员选择停止，则进入终止流程。第一阶段不要求前端提供可信通道选择器。

### 2.5.12 阶段跃迁和事实验真必须打通

倒水样品不能做成“传统状态机 + 后置成功检测”。

至少一个关键阶段跃迁必须依赖事实验真结果。例如：

- `maintain_flow -> return` 不仅检查液位或液面阈值，还必须等待 `cup_has_water = established`。
- 如果 `cup_has_water = not_established`，Runtime 不进入正常回正完成路径，而是回到倾斜阶段、触发恢复或请求人工确认。

最终成立状态必须参与阶段推进、恢复执行或后续动作触发。

### 2.5.13 API 第一阶段只保留最小外部面

为避免 API 过早绑架内部模型，第一阶段外部 API 只保留三个：

- `POST /process/admit`：准入判断。
- `POST /process/run`：运行一次过程实例，返回 `task_id`。
- `GET /audit/{task_id}`：查询审计摘要。

`GET /process/status/{task_id}` 可作为调试接口保留，但不作为对外承诺接口。

其他接口先作为内部函数、CLI 命令或调试入口，待 Runtime Core 稳定后再决定是否外放。

### 2.5.14 技能包草案不做自动泛化

第一阶段的技能包草案等于本次运行的结构化记录，包括：

- 使用的过程模板。
- 使用的偶然绑定。
- 阶段轨迹。
- 因果事实结果。
- 恢复动作。
- 审计摘要。

第一阶段不承诺自动提炼新模板、自动学习阈值或自动生成可销售技能包。技能包草案必须经过人工审核后才能入库。

### 2.5.15 可视化只做轻界面

第一阶段可视化只做：

- 文字状态流。
- 简单阶段进度条。
- 事实验真结果卡片。
- 通道一致/冲突展示。
- 不会做原因展示。

不做 3D 倒水动画，不做复杂实时图表，不做真水视觉效果。炫酷演示进入阶段二或阶段三。

### 2.5.16 阶段三目标平台提前选择

阶段二启动前必须选择阶段三目标机器人或仿真平台，例如 Isaac Sim、Gazebo、ROS/MoveIt 或某一厂商 SDK。

阶段三拆成三个独立里程碑：

1. 能读取一条真实或仿真状态数据。
2. 能驱动一次阶段推进。
3. 能完成一次端到端倒水样品。

每个里程碑单独验收，避免最后阶段才发现联调不可行。

### 2.5.17 第一阶段经验检索只做规则映射

第一阶段不做通用相似度检索、向量检索或复杂经验召回。

规则示例：

- “倒水”
- “倒一杯水”
- “给客人倒水”

均映射到：

- `process_template: pour_water`
- `default_binding: home_a_kitchen_daytime`

无法命中规则时进入不可执行路径，并输出缺少模板或缺少规则映射。

### 2.5.18 最小能力词汇表

阶段零必须定义 Robot Adapter 最小能力词汇表，用于准入判断。

倒水样品最小能力包括：

- `align_container`：能够将容器壶嘴与目标杯口对齐。
- `tilt_container`：能够控制容器倾角。
- `hold_pose`：能够保持当前姿态。
- `return_to_level`：能够回正容器。
- `observe_tilt_angle`：能够观测倾角。
- `observe_flow_rate`：能够观测或模拟水流速度。
- `observe_liquid_level`：能够观测或模拟液位。
- `request_human_confirmation`：能够请求人工确认。

准入判断只检查上述明确能力，不做模糊能力推断。

### 2.5.19 执行体能力画像只做轻量预留

不同自动执行体具有不同的主体物理约束和本体能力边界。人形机器人、轮式机械臂、固定机械臂、数字执行体和仿真体在身高、机身包络、臂展、转向半径、可跨越高度、传感器视场、末端执行器和可达空间上可能存在差异。

该问题与 P008 的“主体物理约束参数”和“空间可进入性判断”直接衔接。P008 负责回答：该主体能否进入当前空间、从哪里进入、能在哪里行动、边界如何限制。RELL/P016 不在第一阶段解决复杂本体运动规划，只保留 Executor Profile 接口占位。

第一阶段 Adapter 应能上报最小执行体能力画像：

- `executor_id`。
- `executor_type`：`digital_agent`、`simulated_robot` 或 `real_robot`。
- `body_profile`：`fixed_mock`、`humanoid`、`wheeled_arm`、`mobile_base` 等。
- `supported_actions`。
- `reachable_workspace`。
- `sensor_frames`。
- `end_effector_type`。
- `payload_limit`。
- `precision_level`。
- `mobility_constraints`。
- `spatial_entry_constraints`。

第一阶段只将该画像用于能力准入占位、审计记录和后续真实 Adapter 对接说明，不做以下能力：

- 复杂 IK 或全身运动规划。
- 自动判断多走几步或少走几步。
- 不同身高、臂长机器人之间的轨迹自动迁移。
- 自动生成真实机器人底层控制指令。

未来阶段中，Executor Profile 可以接入 P008 的主体化空间可进入性结果，并由机器人 SDK、ROS/MoveIt、仿真平台或厂商 Adapter 负责将同一过程模板实例化为不同本体可执行动作。

## 三、整体架构

RELL 样品架构以“任务输入 -> 经验检索与模板选择层 -> P016 Runtime -> Robot Adapter -> 经验回写”为主线。

第一阶段不实现多动作编排引擎。“经验检索与模板选择层”只负责单模板加载、规则映射和准入判断，不进行多动作规划。

其中 P016 Runtime 是样品核心，承担：

- 过程模板实例化。
- 偶然绑定解析。
- 阶段推进。
- 因果事实验真。
- 最终成立状态回流。
- 不会做准入。

准入职责分层：

- 经验检索与模板选择层负责规则映射、模板存在性和绑定完整性的初筛。
- P016 Runtime 负责 Robot Adapter 能力是否满足当前过程实例、观测通道是否可用、恢复边界是否满足的最终准入判断。

### 3.1 任务输入层

输入人类任务意图，例如：

- “给客人倒一杯水。”
- “把杯子端到桌子右侧。”
- “整理桌面，但不要移动票据。”

输出结构化任务意图：

- 任务目标。
- 目标对象。
- 约束条件。
- 人类偏好。
- 风险标记。

第一阶段任务输入层使用规则映射或固定 JSON 模板，不把 LLM 自由输出作为 Runtime 入口。

倒水样品采用关键字或规则映射，不做通用语义相似度检索。

### 3.2 经验检索层

根据任务意图检索已有经验：

- 是否命中规则映射。
- 是否有可复用过程模板。
- 是否有失败恢复记录。
- 是否有人类偏好记录。

如果没有经验，不直接猜测执行，而是进入“不会做/待共创”路径。

本层输出能力准入所需的初始材料：

- 命中的过程模板。
- 命中的历史经验。
- 命中的恢复记录。
- 命中的偏好记录。
- 缺失项列表。

第一阶段规则固定为倒水相关输入映射到 `pour_water`。经验检索不引入向量数据库或通用相似度模型。

### 3.3 过程模板层

存放可复用物理过程结构：

- 阶段序列。
- 连续状态变量。
- 阶段跃迁条件。
- 参数槽。
- 失败条件。
- 恢复入口。

例如倒水过程：

- 对齐阶段。
- 倾斜阶段。
- 保持流动阶段。
- 回正阶段。

本层不直接调用机器人，只向 P016 Runtime 提供可运行的过程定义。

### 3.4 偶然绑定层

把过程模板绑定到当前环境：

- 当前杯子。
- 当前水壶。
- 当前机器人执行器。
- 当前传感器。
- 当前阈值。
- 当前空间坐标。
- 当前用户偏好。

如果绑定不完整，系统不进入真实执行，只输出缺失项。

本层输出过程实例：

- 已绑定对象。
- 已绑定传感器。
- 已绑定阈值。
- 已绑定确认窗口。
- 已绑定偏好参数。

### 3.5 P016 Runtime 层

P016 Runtime 是样品的核心运行层。

它接收：

- 过程模板。
- 偶然绑定数据。
- 因果事实声明。
- Robot Adapter 提供的连续状态变量和观测通道。

它输出：

- 当前阶段。
- 阶段跃迁结果。
- 目标因果事实成立状态。
- 恢复或人工介入请求。
- 经验轨迹。

Runtime 内部是事件驱动状态机，响应 Adapter 推送的状态更新、观测更新和失败事件。禁止用阻塞式同步轮询承担主运行逻辑。

倒水样品的 Runtime 输入来自模拟曲线或手工数据文件，不接入流体物理引擎。

### 3.6 Robot Adapter 与执行模拟层

第一阶段可以先用模拟数据驱动阶段推进。

后续可接入：

- 仿真环境。
- 机器人 SDK。
- 摄像头/深度相机数据。
- 机器人本体状态数据。
- 人工标注的执行日志。

该层负责输出连续状态变量读数，例如：

- 倾角。
- 水流速度。
- 液位高度。
- 杯口距离。
- 抓取力。
- 接触状态。

Robot Adapter 的职责是把不同机器人或仿真系统的数据统一成 RELL 可消费的连续状态变量和观测通道。

第一阶段 `MockRobotAdapter` 必须支持成功路径、失败路径和传感器异常路径，不能只返回理想状态。

MockRobotAdapter 使用与未来真实 Adapter 相同的异步事件机制，避免 Mock 成功而真实 SDK 接不上。

### 3.7 因果事实验真层

不只判断动作是否执行，而是判断目标事实是否成立。

例如：

- 壶口是否对准杯口。
- 是否出现持续水流。
- 杯中是否确实有水。
- 杯子是否稳定放置到目标位置（阶段二）。

事实验真输出最终成立状态：

- established：事实成立。
- not_established：事实不成立。
- conflicted：观测冲突。
- unknown：信息不足。

本层是 P016 Runtime 的验证子模块，不应被拆成与过程运行完全无关的独立系统。否则会造成“动作跑完了，但事实验真和阶段推进脱节”的重构风险。

每个事实验真任务必须配置确认窗口和超时策略。`unknown` 只能是中间状态，不能无限等待。

事实验真结果必须回流到阶段推进。至少一个阶段跃迁必须等待目标因果事实 `established` 后才能触发。

如果事实验真触发恢复，同一阶段恢复次数不得超过配置的 `max_recovery_attempts`。

### 3.8 经验回写与审计层

记录本次任务全过程：

- 使用了哪个过程模板。
- 使用了哪些偶然绑定。
- 阶段如何推进。
- 哪些因果事实成立。
- 失败在哪里发生。
- 如何恢复。
- 哪些经验可复用。
- 哪些缺失需要共创。

输出：

- 经验记录。
- 恢复记录。
- 偏好记录。
- 可人工审核的技能包草案。
- 审计摘要。

### 3.9 最小闭环数据流

样品最小数据流固定如下：

```text
task_intent
  -> experience_search_result
  -> admission_decision
  -> process_template
  -> occasional_binding_data
  -> process_instance
  -> stage_runtime_state
  -> fact_observation_result
  -> final_establishment_state
  -> recovery_or_advance_decision
  -> execution_trace
  -> experience_record / audit_summary
```

后续代码实现应围绕这条数据流拆模块，避免先做零散功能再回头重构。

阶段零必须用倒水任务的完整 Mock JSON 跑通以上数据流，再进入 Runtime 代码实现。

`stage_runtime_state` 与 `execution_trace` 的边界必须在阶段零评审中明确：

- `stage_runtime_state` 是当前状态快照，描述当前阶段、当前变量、等待状态、重试次数和最新事实状态。
- `execution_trace` 是追加写日志，记录事件 ID、事件顺序、阶段跃迁、事实验真结果、恢复动作和人工确认结果。
- 验真器和实时状态查询读取 `stage_runtime_state`。
- 审计摘要、经验回写和技能包草案读取 `execution_trace`。

## 四、核心攻关项目

### 4.-1 Schema 与 Mock 数据冻结

目标：在编码前冻结核心数据结构，降低后期字段变更导致的连锁重构。

最小能力：

- 定义核心 JSON Schema。
- 定义 `process_template.schema_version`。
- 定义 `occasional_binding.schema_version`。
- 输出倒水任务完整 Mock JSON。
- 完成 Schema 校验。
- 完成最小数据流串联。

验收标准：

- `task_intent -> admission_decision -> process_instance -> stage_runtime_state -> fact_observation_result -> final_establishment_state -> execution_trace` 可以用 Mock JSON 串通。
- 所有过程模板和偶然绑定文件必须通过 Schema 校验后才能运行。

### 4.0 P016 Runtime Core

目标：形成样品的核心运行内核。

最小能力：

- 接收过程模板和偶然绑定数据。
- 创建过程实例。
- 管理当前阶段。
- 调用 Robot Adapter 获取连续状态变量。
- 判断阶段跃迁条件。
- 调用因果事实验真器。
- 根据最终成立状态输出推进、恢复、后续动作或拒绝执行。
- 响应 Adapter 推送事件，而不是阻塞式轮询。

验收标准：

- 不依赖真实机器人，仅用 MockRobotAdapter 即可跑通倒水过程闭环。
- 突发失败事件可以中断当前阶段并触发恢复或停止。
- `maintain_flow -> return` 等至少一个阶段跃迁由事实验真结果参与触发。
- 人工确认请求发出后，Runtime 主状态机暂停阶段跃迁判断，直到收到确认、取消或停止事件。

### 4.1 过程模板运行器

目标：让过程模板不只是 JSON，而是可以被运行。

最小能力：

- 读取过程模板。
- 读取当前阶段。
- 读取连续状态变量。
- 判断阶段跃迁条件。
- 推进到下一阶段或终止。

验收标准：

- 倒水过程可以从对齐、倾斜、保持、回正跑完整个阶段链。

### 4.2 偶然绑定解析器

目标：把模板参数槽绑定到当前环境。

最小能力：

- 对象绑定。
- 传感器绑定。
- 阈值绑定。
- 空间绑定。
- 偏好绑定。

验收标准：

- 同一个倒水模板可以加载家庭 A 和家庭 B 两套绑定，并产生不同阈值和传感器配置。

### 4.3 因果事实验真器

目标：让系统判断结果事实，而不是只判断动作完成。

最小能力：

- 读取目标因果事实。
- 读取一个或多个观测通道。
- 形成成立判断。
- 冲突时输出仲裁结果或请求人工确认。
- 支持确认窗口、置信度阈值和超时策略。
- 支持两个独立观测通道来自不同 Mock 信号或不同计算路径。

验收标准：

- 能判断“杯中有水”“杯子在目标位置”等事实是否成立。
- 能把事实验真结果输出为 `final_establishment_state`。
- 传感器遮挡或持续 unknown 时不会卡死，能够按策略转入恢复、人工确认或失败。
- 审计摘要能展示每个通道的判断和最终仲裁结果。
- 双通道冲突时，第一阶段默认阻断后续动作并请求人工确认。

### 4.4 能力准入与不会做机制

目标：没有经验、缺模板、缺绑定、缺验真条件时，系统明确拒绝真实执行。

最小能力：

- 检查是否有过程模板。
- 检查是否有偶然绑定。
- 检查是否有因果事实声明。
- 检查是否有观测通道。
- 检查 Robot Adapter 是否报告对应执行能力。
- 检查是否命中明确规则映射。
- 输出不可执行原因和待补任务。

验收标准：

- 输入未知任务时，系统输出“当前不会执行”，并列出缺失项，而不是生成猜测动作。

### 4.5 经验记录与技能包生成

目标：把一次运行沉淀为后续可复用经验。

最小能力：

- 记录阶段轨迹。
- 记录成功和失败。
- 记录恢复动作。
- 记录人类确认。
- 生成技能包草案。

验收标准：

- 一次倒水 demo 后，能输出经验记录、恢复记录、审计摘要和技能包草案。

### 4.6 Robot Adapter 接口

目标：让 RELL 与底层机器人执行模型解耦。

最小能力：

- 定义统一 Adapter 协议。
- 实现 `MockRobotAdapter`。
- 支持从模拟数据或执行日志读取连续状态变量。
- 支持返回观测通道判断材料。
- 支持停止、恢复和人工确认请求。
- 支持状态事件推送。
- 支持故障注入配置。
- 使用异步事件接口实现 Mock，不依赖同步读取主循环。
- 不直接判断阶段跃迁条件是否满足。

验收标准：

- P016 Runtime 不直接读取具体机器人 SDK，而是只依赖 Adapter 接口。
- MockRobotAdapter 可以分别跑通成功路径、无水流失败路径和传感器冲突路径。
- 人工确认请求不阻塞 Runtime 主状态机。

### 4.7 外部 API 与联调接口

目标：让机器人公司、仿真系统或第三方工具可以接入。

第一阶段外部最小 API：

- `POST /process/admit`
- `POST /process/run`
- `GET /audit/{task_id}`

调试接口：

- `GET /process/status/{task_id}`
- `POST /process/events/{task_id}`

调试接口只用于本地演示、前端调试和联调排障，不写入第一阶段对外 API 承诺文档。

验收标准：

- 外部系统可以通过 API 完成一次任务准入、过程运行和审计查询。
- `POST /process/run` 返回 `task_id`，不阻塞等待整个过程完成。
- 调试时可以通过 `GET /process/status/{task_id}` 查询当前阶段和最终成立状态。

## 五、样品演示脚本

### 演示一：有经验，可以执行

输入：

> 给客人倒一杯水。

前置条件：

- 系统已预置倒水过程模板。
- 系统已预置家庭 A 的偶然绑定数据。
- MockRobotAdapter 已报告倒水所需最小能力。

系统输出：

1. 检索到倒水过程模板。
2. 绑定家庭 A 的杯子、水壶、传感器和阈值。
3. 阶段推进：对齐 -> 倾斜 -> 保持流动 -> 回正。
4. 通过两个独立 Mock 通道验证杯中有水。
5. 生成经验记录和审计摘要。

### 演示二：环境变化，仍可复用

输入：

> 在家庭 B 倒一杯水。

系统输出：

1. 复用同一个倒水过程模板。
2. 使用家庭 B 的不同杯子、水壶、阈值和传感器。
3. 阶段推进逻辑不变，偶然绑定变化。
4. 证明跨场景复用能力。

### 演示三：缺少绑定，不执行

输入：

> 用未知容器倒水。

系统输出：

1. 找到倒水过程模板。
2. 无法完成对象绑定或阈值绑定。
3. 输出不可执行原因。
4. 生成待标定任务。

如演示需要继续，可走“人类确认后可执行”的降级路径，由人工补充缺失绑定后再运行。

### 演示四：结果验真失败，触发恢复

输入：

> 倒水过程中水流未出现。

系统输出：

1. 倾斜阶段达到最大倾角。
2. 水流事实未成立。
3. 审计记录显示预设失败标签：无水流、壶嘴堵塞或倾斜异常。
4. 触发恢复流程或请求人工介入。

说明：上述失败标签仅为审计日志中的预设文本，不强制实现根因诊断算法。

### 演示五：传感器 unknown，不会卡死

输入：

> 倒水过程中深度相机短时遮挡。

系统输出：

1. 物理通道进入 unknown。
2. 验真器进入确认窗口等待。
3. 超时后根据策略触发重新观测、人工确认或恢复。
4. 审计摘要记录 unknown 持续时间和处置结果。

### 演示六：Mock 故障注入

输入：

> 使用故障配置运行倒水样品。

系统输出：

1. MockRobotAdapter 在倾斜阶段注入无水流异常。
2. Runtime 接收失败事件。
3. 因果事实未成立。
4. 系统触发恢复或声明不可继续执行。

### 演示七：双通道冲突与仲裁

输入：

> 使用通道冲突配置运行倒水样品。

系统输出：

1. 通道一基于模拟液位判断杯中有水。
2. 通道二基于流速积分判断注入量不足。
3. 验真器输出 conflicted。
4. Runtime 按策略请求人工确认、重新观测或触发恢复。
5. 审计摘要展示两个通道的输入、判断和仲裁过程。

## 六、落地测试联调路径

### 阶段零：架构与接口冻结

周期：3 至 5 天。

交付：

- P016 Runtime 模块边界。
- Robot Adapter 接口。
- Robot Adapter 最小能力词汇表。
- Runtime 事件结构定义，包含 `event_id`、`sequence`、`timestamp`、`process_instance_id` 和 `stage_id`。
- 核心数据流定义。
- 核心 JSON Schema。
- 倒水任务完整 Mock JSON。
- API 会话机制定义。
- 不会做规则清单。
- Mock 数据曲线文件。
- 双通道 Mock 信号定义。
- 样品不做事项清单。

目标：

- 防止后续开发时把 RELL 写成机器人底层执行系统，或把治理层一次性做过重。
- 防止 Schema 未冻结就开写 Runtime，造成后期字段连锁重构。

进入阶段一的硬门槛：

- 倒水任务完整 Mock JSON 全链路跑通。
- 双通道 Mock 信号定义完成，并至少包含一个冲突配置。

阶段零采用迭代冻结策略：

1. 先冻结 `process_template`、`occasional_binding_data` 和 `stage_runtime_state`。
2. 跑通最小 Mock 数据流。
3. 再冻结 `fact_observation_result`、`final_establishment_state`、`execution_trace` 和 `audit_summary`。

宁可阶段零多花两天，也不带着不稳定 Schema 进入 Runtime 开发。

### 阶段一：纯软件样品

周期：1 至 2 周。

交付：

- 本地可运行 demo。
- JSON Schema。

## 八、样品验收标准

样品达到以下标准，即可作为对接库初版材料：

1. 可以本地启动。
2. 可以输入至少一个真实世界任务。
3. 可以通过规则映射加载倒水过程模板。
4. 可以完成偶然绑定。
5. 可以跑出阶段推进。
6. 可以进行目标因果事实验真。
7. 可以在缺失条件时明确拒绝执行。
8. 可以输出经验记录和审计摘要。
9. 可以通过 API 被外部系统调用。
10. 可以用 3 分钟演示讲清楚商业价值。
11. RELL Runtime 不直接依赖某个机器人 SDK。
12. Robot Adapter 可以替换为模拟、仿真或真实机器人适配器。
13. Runtime 不使用阻塞式死循环作为主运行逻辑。
14. 核心数据结构有 Schema 和倒水 Mock JSON 实例。
15. 因果验真 unknown 状态有超时处置，不会无限等待。
16. 第一阶段意图解析不依赖未经校验的 LLM 自由输出。
17. `/process/run` 是异步任务接口，并支持状态查询。
18. MockRobotAdapter 至少支持两个故障注入场景。
19. 倒水样品不依赖流体物理引擎或真实视觉识别。
20. 双通道验真来自不同 Mock 信号或不同计算路径。
21. 至少一个阶段跃迁由事实验真结果参与触发。
22. 第一阶段只支持单过程实例，多动作过程链不进入 Runtime。
23. 第一阶段外部 API 不超过三个核心接口。
24. 技能包草案只是可人工审核的结构化记录，不做自动泛化承诺。
25. 可视化为轻量状态展示，不做 3D 动画。
26. 第一阶段不实现通用编排引擎或多动作过程链。
27. Adapter 只推送状态与观测事件，阶段跃迁条件由 Runtime 判断。
28. Robot Adapter 最小能力词汇表已定义并参与准入判断。
29. `GET /process/status/{task_id}` 标注为调试接口，不作为第一阶段对外承诺接口。
30. Runtime 支持事件去重和乱序事件丢弃。
31. Runtime 具有 `awaiting_human_confirmation` 等待状态。
32. 恢复流程配置最大重试次数和人工升级策略。
33. Schema 版本兼容规则采用 major/minor 策略。
34. Runtime 内部采用串行事件队列或 Actor Model，禁止并发写状态。
35. MockRobotAdapter 由时间轴事件脚本或 JSON 配置驱动，不硬编码复杂物理逻辑。
36. 双通道冲突时默认阻断后续动作并请求人工确认。
37. Schema 允许受控变更，但必须同步更新 Schema、Mock JSON、版本号和影响记录。

## 九、当前优先级

优先级从高到低：

1. 架构与接口冻结。
2. Schema 与 Mock 数据冻结。
3. P016 Runtime Core。
4. Robot Adapter 接口与 MockRobotAdapter。
5. 过程模板运行器。
6. 偶然绑定解析器。
7. 因果事实验真器。
8. 不会做机制。
9. 双通道 Mock 与审计展示。
10. 倒水任务样品。
11. 最小 API。
12. 轻量可视化界面。
13. 仿真或机器人联调。
14. 技能包和经验库市场化。
15. 多主体空间治理。

## 十、结论

RELL 从 0 到 1 的关键不是一次做大，而是做出一个有说服力的样品：

> 一个机器人任务从意图进入，经过经验检索、过程模板、偶然绑定、阶段推进、因果验真、失败恢复和经验回写，最终形成可复用技能资产。

只要这个闭环跑通，就能支撑基金沟通、机器人公司试点、开源生态传播和后续商业化推进。
