# P011 物理经验内化机制说明书初稿

---

## 一、技术领域

本发明涉及智能体控制、机器人控制、数字执行体控制、经验学习、动作后果预测和交互反馈技术领域，尤其涉及一种基于真实交互经验记录的后果预测与经验自进化方法、经验模型积累方法、动作后果预测方法、系统及计算机可读存储介质。

本发明可以应用于物理执行主体和数字执行主体。所述物理执行主体包括机器人、机械臂、移动机器人、无人机、自动驾驶设备、工业控制设备或其他能够在真实物理环境中执行动作的设备。所述数字执行主体包括软件智能体、GUI 智能体、工具调用智能体、数字员工、虚拟角色、游戏智能体、API 调用智能体或其他能够在真实数字环境中执行动作的执行主体。

在一些实施方式中，物理执行主体可以包括由软件智能体、控制程序或数字执行体控制的机器人、机械臂、无人机或其他物理设备。此时，数字执行体可以用于生成、选择或下发控制指令，物理执行主体通过物理机构在真实物理环境中产生动作后果。与此不同，数字执行体也可以直接在真实数字环境中执行 GUI 操作、API 调用、工具调用、文件操作或业务流程动作，并在真实数字环境中产生可观测数字后果。

本发明将物理执行主体和数字执行体统一称为执行主体，并不要求二者在硬件形态或动作通道上相同，而是基于二者均满足以下共同技术条件：动作执行前存在可描述情境，执行主体能够执行动作，动作后产生可观测后果，并且情境、动作和后果之间的对应关系能够被记录、检索并用于后续候选动作后果预测和经验模型反馈更新。

## 二、背景技术

现有机器人或自动控制系统通常依赖传感器建模、动力学模型、几何约束、控制规则或仿真引擎计算动作后果。该类方式在环境变化、模型不完备、传感误差或交互对象复杂时，容易出现计算成本高、泛化差、对建模精度依赖强的问题。例如，机械臂在面对陌生抽屉、杯子、门把手或柔性物体时，如果依赖完整动力学模型计算动作后果，需要对物体形状、材质、摩擦、约束关系和接触状态进行精确建模，实际部署难度较高。

现有强化学习、情景记忆或经验回放方案常以状态、动作、奖励、下一状态等数据记录训练过程，并通过策略网络、价值网络或其他模型进行动作选择。该类方案中的经验通常服务于模型参数训练或策略优化，后果常表现为奖励值、成功失败标签、下一状态或训练样本，并不一定作为可检索经验字段用于当前候选动作的后果预测。

现有路径复用、工作流复用、工具调用记忆或 GUI 轨迹复用方案，可以记录成功路径、动作序列、工具序列、工作流或操作轨迹，并在相似任务中复用。该类方案主要解决复用哪条路径、调用哪个工具、恢复哪段工作流或推荐下一步操作的问题，并不以动作执行后的可观测后果作为基础经验字段，也不一定在动作确定前针对候选动作的预期后果进行类比预测。

现有 action store、skill registry、case-based planning 或 decision pattern map 方案，可能根据执行结果、性能指标或状态空间变化更新动作库或决策模式。该类方案虽然可能记录动作执行结果或状态空间变化，但其结果通常作为动作库管理信号、性能指标、模式变化或案例评分使用，并非作为经验记录中的后果字段参与下一次候选动作后果预测。

现有机器人可供性学习、对象-动作-效果关系、上下文可供性、对象-动作复合体或动作模型学习方案，可能表示物理对象、动作和动作效果之间的关系，并用于机器人操作、规划、控制或动作效果预测。该类方案通常面向物理机器人与物理对象之间的可供性关系，或用于训练分类器、神经网络、动作模型或规划域模型。

现有 Web 智能体或 LLM 智能体方案，可能在网页任务中记录页面交互轨迹、失败经验、网页状态转移或认知地图，并在真实执行前对候选网页动作进行前瞻模拟。该类方案通常服务于网页导航或网页任务完成，其经验主要表现为网页状态转移图、网页交互摘要、认知地图或页面环境约束。

综上，现有技术仍缺少一种以真实交互形成的情境描述、动作和后果之间的对应关系为基础，直接面向候选动作后果预测，并能在真实执行后基于实际后果持续更新的通用经验机制。尤其是，现有技术通常将物理机器人经验和数字执行体经验分别处理，缺少一种能够同时适用于真实物理环境和真实数字环境的统一经验内化机制。

## 三、发明内容

### 3.1 要解决的技术问题

本发明要解决的技术问题至少包括：

1. 如何使执行主体在不依赖完整物理规则、完整环境模型或高保真仿真引擎的情况下，预测候选动作在当前情境下可能产生的后果。
2. 如何将真实交互产生的后果沉淀为可复用、可迁移、可局部修正的经验模型。
3. 如何使不同物理空间或数字环境中的经验可被相似情境复用，并在真实执行后继续反馈更新。
4. 如何使物理执行主体和数字执行主体能够使用统一的动作后果经验机制。

### 3.2 总体技术方案

为解决上述技术问题，本发明提供一种基于交互经验类比检索的后果预测与经验自进化方法，包括：

S1，获取目标空间或目标数字环境的经验情境描述信息；

S2，基于经验模型，对候选动作在所述经验情境描述信息所表征的当前情境下的预期后果进行预测；其中，所述经验模型为基于真实交互积累的多个情境描述、动作和后果记录的集合；

S3，至少基于所述预期后果，从所述候选动作中确定目标动作；

S4，执行所述目标动作，并将本次经验情境描述信息、目标动作和实际后果作为新记录反馈入所述经验模型。

在一些实施方式中，步骤 S2 中的预测包括：基于所述经验情境描述信息，在所述经验模型中检索情境相似度较高的历史记录；基于检索到的历史记录中的动作与后果之间的对应关系，对所述候选动作的预期后果进行类比推断。

在一些实施方式中，所述真实交互为在真实物理环境或真实数字环境中执行并产生可观测后果的动作。所述真实数字环境可以包括实际运行的软件系统、网页、应用、操作系统、API、数据库、工具链、文件系统、业务系统或虚拟世界运行实例。

### 3.3 经验模型

所述经验模型为基于真实交互积累的多个经验记录的集合。每条经验记录至少包括：

1. 情境描述，用于描述动作发生前的环境、对象、状态、任务约束或上下文。
2. 动作，用于表示执行主体执行或拟执行的物理动作、数字动作或复合动作。
3. 后果，用于表示动作执行后产生的可观测状态变化、执行结果、约束影响、异常、成功或失败原因。

经验记录可以采用表结构、图结构、向量索引、日志索引、事件链、自然语言记录、结构化字段、半结构化字段或其组合实现。经验记录不要求情境描述、动作和后果物理存储于同一数据对象中，只要能够建立可检索的对应关系即可。

在不同实现中，情境描述也可以被称为状态、上下文、观察、任务上下文、环境描述、页面状态、业务状态或场景描述；动作也可以被称为操作、指令、工具调用、API 调用、控制量、执行步骤或子任务；后果也可以被称为结果、反馈、状态变化、返回值、执行影响、状态差异、异常、效果或 outcome。上述名称变化不影响其作为情境-动作-后果对应关系的技术实质。

所述后果不等同于单一奖励值、动作评分、工作流完成标签或动作库性能指标。所述后果是用于预测和比较候选动作影响的经验字段。例如，物理后果可以包括物体被移动、掉落、卡住、倾斜、破损、路径偏移、力反馈异常、任务完成或失败；数字后果可以包括页面跳转、字段写入成功、API 返回错误、权限拒绝、数据状态改变、文件生成、流程进入下一状态或工具调用失败。

### 3.4 情境描述

在真实物理环境中，情境描述可以包括空间语义信息、物体类别、物体相对位置、可操作部件、接触关系、遮挡关系、任务目标、安全边界、执行主体位姿、末端执行器状态、载荷、速度或接触状态。

在真实数字环境中，情境描述可以包括 GUI 页面状态、控件位置、控件语义、焦点状态、输入字段内容、DOM 信息、辅助功能树信息、API 状态、工具可用性、权限状态、任务上下文、数据对象状态、业务流程状态、虚拟世界对象状态或角色状态。

经验情境描述信息可以由传感器、摄像头、深度相机、力传感器、定位模块、空间语义系统、视觉识别系统、环境感知系统、GUI 解析系统、日志解析系统、API 元数据读取模块、用户指令或任务上下文提供。本发明不以某一种情境描述生成方式为必要限定。

### 3.5 动作

所述动作可以包括物理动作、数字动作或复合动作。

物理动作可以包括抓取、推动、拉动、旋转、移动、避障、点击实体按钮、开关设备、改变姿态或调节力控参数。

数字动作可以包括点击、输入、拖拽、选择、调用工具、调用 API、修改文件、提交表单、执行脚本、发送消息、更新数据库记录或触发业务流程。

复合动作可以包括由多个子动作构成的动作序列。一个复合动作可以整体作为候选动作，也可以拆分为多个子动作分别进行后果预测。

### 3.6 后果

所述后果为动作执行后在真实物理环境或真实数字环境中产生的可观测影响。

物理后果可以包括目标物体位置变化、姿态变化、打开、关闭、倾倒、滑落、破损、卡滞、碰撞、力反馈异常、路径偏移、到达目标或未到达目标。

数字后果可以包括页面元素变化、页面跳转、字段值变化、接口返回、错误信息、权限拒绝、数据库变化、文件变化、工具执行结果、业务流程状态变化、虚拟世界对象状态变化、资源消耗或安全告警。

在一些实施方式中，后果可以包括结构化描述、自然语言描述、状态差异、事件集合、日志片段、异常码、返回值、状态标签、置信度或其组合。

## 五、具体实施方式

以下结合实施例对本发明进行说明。应理解，以下实施例仅用于说明本发明，并不用于限定本发明的保护范围。在不冲突的情况下，各实施例中的技术特征可以相互组合。

### 5.1 方法总体流程

如图1所示，在一个实施例中，基于交互经验类比检索的后果预测与经验自进化方法包括以下步骤。

步骤 S1，获取目标空间或目标数字环境的经验情境描述信息。

所述经验情境描述信息用于描述执行动作前的当前情境。对于物理执行主体，经验情境描述信息可以来自视觉感知、空间语义识别、力觉感知、定位信息或任务信息。对于数字执行主体，经验情境描述信息可以来自页面截图、DOM、辅助功能树、应用状态、API 元数据、日志、数据库状态、文件状态或业务上下文。

步骤 S2，获取候选动作。

候选动作可以来自执行主体的动作库、任务规划模块、人工输入、示教数据、工具注册表、API 列表、历史经验中出现过的动作或由 LLM、VLM、RAG、规则引擎、规划器生成的动作。候选动作可以是单步动作，也可以是动作序列。

步骤 S3，基于经验模型，对候选动作在当前情境下的预期后果进行预测。

在一个实施例中，系统基于当前情境描述，在经验模型中检索与当前情境相似的历史经验记录。相似度可以基于向量相似度、结构化字段匹配、对象关系图相似度、GUI 状态相似度、任务目标相似度、约束相似度或其组合计算。

检索到的历史经验记录可以包括与候选动作相同的动作，也可以包括与候选动作相似的动作。系统基于历史记录中的动作与后果之间的对应关系，对候选动作的预期后果进行类比推断。在一些实施方式中，系统可以根据当前情境与历史情境之间的差异，对历史后果进行映射、修正或置信度调整。

步骤 S4，至少基于所述预期后果，从候选动作中确定目标动作。

在一个实施例中，系统根据预期后果是否满足任务目标、是否满足安全约束或权限约束、后果置信度、执行成本、时间成本、资源消耗、可逆性或失败代价，从候选动作中选择目标动作。动作确定可以包括选择一个目标动作、对多个候选动作排序、剔除风险动作、调整动作参数或输出风险提示。

步骤 S5，执行目标动作，并采集实际后果。

目标动作可以在真实物理环境或真实数字环境中执行。真实物理环境是指真实设备、真实物体或真实空间构成的运行环境。真实数字环境是指实际运行的软件系统、网页、应用、操作系统、API、数据库、工具链、文件系统、业务系统或虚拟世界运行实例。

步骤 S6，将本次经验情境描述信息、目标动作和实际后果作为新记录反馈入经验模型。

如果实际后果与预期后果一致，系统可以新增经验记录或提高相关记录的可信度。如果实际后果与预期后果不一致，系统可以新增一条经验记录、修正相似历史记录的权重、降低错误预测来源记录的可信度，或者仅更新与本次情境相关的记录，而不修改经验模型中与本次情境无关的其他记录。

### 5.2 经验模型的组织方式

如图2所示，在一个实施例中，经验记录包括情境描述字段、动作字段、后果字段、时间戳、来源主体、环境类型、可信度、权重、任务标签和约束标签。

经验模型可以按照以下一种或多种方式组织：

1. 表结构数据库，用于存储结构化情境字段、动作字段和后果字段。
2. 向量数据库，用于基于情境嵌入、动作嵌入或后果嵌入进行相似检索。
3. 图数据库，用于表示对象、状态、动作和后果之间的关系。
4. 日志索引，用于从交互日志、运行日志或审计日志中检索经验记录。
5. 事件链，用于表示动作前情境、动作和动作后状态变化之间的时间顺序。
6. 分层记忆结构，用于分别存储短期工作记忆、长期经验记录、语义摘要或高置信度经验。
7. 分布式经验库，用于多个执行主体之间共享经验增量。

上述组织方式不改变经验记录的基础实质。只要系统能够根据当前情境检索历史经验，并基于历史动作与后果之间的对应关系预测候选动作后果，即可作为本发明的经验模型实现方式。

### 5.3 后果预测

如图3所示，在一个实施例中，系统基于当前情境检索多个相似历史记录，并对候选动作生成预期后果。

例如，对于候选动作 A，系统在历史记录中检索到多个与当前情境相似且动作为 A 或动作与 A 相似的记录。系统提取这些记录中的后果字段，并根据相似度、记录时间、来源主体可信度、后果稳定性或任务约束，对不同后果赋予不同权重。系统可以输出候选动作 A 的后果分布、最可能后果、风险后果、置信度和建议执行参数。

在一些实施方式中，后果预测可以通过最近邻映射、加权投票、统计聚合、规则过滤、图路径推断、向量检索增强生成、语言模型推理或其组合实现。本发明的创新不在于某一具体相似度公式，而在于基于真实交互形成的情境-动作-后果对应关系，对候选动作的预期后果进行预测并基于实际后果反馈更新。

### 5.4 影子执行

在一个实施例中，在真实执行目标动作之前，系统可以基于预期后果进行影子执行。影子执行可以不依赖物理仿真引擎，而是基于经验模型中的历史后果链推演动作可能造成的后续情境。

例如，若候选动作为“点击提交”，其预期后果为“页面进入确认页”，系统可以继续基于确认页情境检索后续经验记录，判断是否会出现权限拒绝、不可逆提交或需要人工审批。如果影子执行的后果链触发安全约束、权限约束或任务失败条件，系统可以剔除该候选动作、调整动作参数或要求人工确认。

影子执行不同于仅由预设仿真规则计算后果的仿真。影子执行可以使用真实交互经验中的后果链进行低成本推演，也可以与规则引擎、仿真引擎、LLM 或其他预测模块结合使用。

### 5.5 真实执行与反馈更新

如图4所示，执行主体在真实环境中执行目标动作后，系统采集实际后果并写回经验模型。

在真实物理环境中，实际后果可以通过传感器、摄像头、力传感器、位置传感器、状态检测器或任务完成检测器采集。在真实数字环境中，实际后果可以通过页面状态读取、DOM 变化检测、API 返回、日志解析、数据库状态读取、文件系统状态读取、工具返回值或业务流程状态读取采集。

反馈更新可以包括新增记录、更新权重、更新可信度、标记异常、合并相似记录、建立反例记录或局部修正记录。若实际后果与预期后果不一致，系统可以仅修正与本次情境相关的历史记录或其权重，而不修改经验模型中与本次情境无关的其他记录，从而降低错误更新对无关经验的影响。

### 5.6 经验模型积累方法

在一个实施例中，经验模型的积累方法包括：获取多个空间中执行主体执行真实交互所产生的经验情境描述信息、执行的动作和实际后果记录；将所述记录按照经验情境描述信息的相似度组织为经验模型；输出所述经验模型，所述经验模型被配置为被执行主体在预测候选动作的后果时检索使用。

所述多个空间可以包括多个真实物理空间、多个真实数字环境或二者的组合。不同执行主体可以将各自积累的经验记录上传至共享经验库，也可以仅上传经验模型的增量部分、摘要部分、高置信度部分或脱敏部分。

### 5.7 动作后果预测方法

在一个实施例中，动作后果预测方法包括：获取经验模型；获取目标空间或目标数字环境的经验情境描述信息以及待评估的候选动作；基于经验情境描述信息在经验模型中检索情境相似度较高的历史记录；基于检索到的历史记录中的动作与后果之间的对应关系，对候选动作的预期后果进行类比推断。

输出结果可以包括预期后果描述、后果置信度、风险等级、推荐标记、不推荐标记、可执行参数建议或需要人工确认的提示。

### 5.8 系统结构

在一个实施例中，基于交互经验类比检索的后果预测与经验自进化系统包括：

1. 情境获取模块，用于获取目标空间或目标数字环境的经验情境描述信息。
2. 候选动作获取模块，用于获取一个或多个候选动作。
3. 经验模型存储模块，用于存储基于真实交互积累的多个情境描述、动作和后果记录。
4. 相似检索模块，用于基于当前情境描述从经验模型中检索相似历史记录。
5. 后果预测模块，用于基于历史动作与后果之间的对应关系预测候选动作的预期后果。
6. 动作确定模块，用于基于预期后果确定目标动作。
7. 影子执行模块，用于在真实执行前基于预期后果链进行低成本验证。
8. 执行控制模块，用于控制执行主体执行目标动作。
9. 后果采集模块，用于采集目标动作执行后的实际后果。
10. 经验反馈更新模块，用于将当前情境、目标动作和实际后果反馈入经验模型。
11. 经验共享模块，用于向其他执行主体共享经验模型或其增量部分。

上述模块可以部署于机器人本体、边缘计算设备、云端服务器、智能体平台、浏览器扩展、桌面自动化系统或多主体协作网络中。

## 六、实施例

### 6.1 实施例一：机械臂抽屉操作

如图5所示，机械臂面对一个抽屉，当前任务为打开抽屉。系统获取的情境描述包括：目标对象为抽屉，抽屉具有可抓取把手，把手位于抽屉正面，抽屉可能具有滑轨结构。候选动作包括向外拉、向内推、向上抬和旋转把手。

经验模型中包括如下历史经验记录：

| 情境描述 | 动作 | 后果 |
|----------|------|------|
| 抽屉正面、把手可抓取、滑轨结构 | 向外拉 | 抽屉打开 |
| 抽屉正面、把手可抓取、滑轨结构 | 向内推 | 抽屉未打开，产生阻力 |
| 抽屉正面、把手可抓取、滑轨结构 | 向上抬 | 抽屉卡滞 |

系统确定当前情境与历史抽屉情境相似，预测“向外拉”的预期后果为抽屉打开，预测“向内推”的预期后果为未打开并产生阻力，预测“向上抬”的预期后果为卡滞。因此，系统选择向外拉作为目标动作。

若实际执行后抽屉未打开且检测到锁止状态，系统将“带锁抽屉、把手可抓取、向外拉、未打开且受锁止约束”作为新经验记录写入经验模型。后续遇到类似带锁抽屉情境时，系统可以预测直接向外拉难以打开，并优先选择解锁、寻找开关或请求人工协助。

### 6.2 实施例二：机械臂杯子抓取

机械臂面对位于桌面边缘的杯子，杯口向上。候选动作包括垂直上提、水平推和侧向夹取。

历史经验记录显示，在类似杯子靠近桌边的情境下，水平推可能导致杯子倾倒或掉落，侧向夹取在杯壁湿滑时可能导致滑落，垂直上提在夹持稳定时通常能够移取杯子。因此，系统预测垂直上提的预期后果为杯子被成功拿起，预测水平推的预期后果为杯子位置偏移或掉落风险，预测侧向夹取的预期后果为滑落风险。系统选择垂直上提作为目标动作。

该判断不要求系统根据杯子受力、摩擦系数、质量分布和接触点建立完整动力学模型，而是基于相似历史经验中的动作与后果对应关系进行类比预测。

### 6.3 实施例三：移动机器人避障

移动机器人在通道中检测到临时障碍物。情境描述包括通道宽度、障碍物位置、左右可通行空间、任务目标和安全约束。候选动作包括左绕、右绕、停止等待和直接通过。

经验模型中记录了类似通道下左绕成功通过、右侧窄缝导致碰撞风险、直接通过导致接触障碍物的历史经验。系统预测左绕的预期后果为通过障碍区域，右绕的预期后果为碰撞风险较高，直接通过的预期后果为接触障碍物，停止等待的预期后果为任务延迟。系统基于任务目标和安全约束选择左绕。

执行后，系统将实际路径偏移、是否碰撞、是否到达目标、耗时等实际后果写回经验模型。

### 6.4 实施例四：GUI 智能体填写表单

如图6所示，GUI 智能体在真实运行的订单退货页面中执行任务。当前页面包含订单号输入框和提交按钮，当前订单号为空。候选动作包括输入订单号、点击提交、点击取消和切换页面。

经验模型中包括如下历史经验记录：

| 情境描述 | 动作 | 后果 |
|----------|------|------|
| 订单退货表单、订单号为空 | 点击提交 | 页面提示订单号必填 |
| 订单退货表单、订单号为空 | 输入有效订单号 | 输入框状态变为已填写 |
| 订单退货表单、订单号已填写 | 点击提交 | 进入退货确认页面 |

系统预测在订单号为空的当前情境下直接点击提交会触发错误提示，因此先执行输入订单号。输入完成后，当前情境更新为订单号已填写，系统再预测点击提交会进入退货确认页面，进而执行点击提交。

若提交后页面返回权限错误，则系统将“订单号已填写、点击提交、权限错误”作为新经验记录写入经验模型。后续遇到相似权限状态时，系统可以预测提交失败，并先执行权限检查或请求授权。

### 6.5 实施例五：工具调用智能体生成配送标签

工具调用智能体的任务为生成配送标签。可用工具包括订单查询 API、地址校验 API 和物流标签 API。当前任务上下文中存在订单号，但缺少收件地址。

经验模型中记录，在缺少收件地址的情境下直接调用物流标签 API 会返回参数缺失错误；在具有订单号但缺少地址的情境下调用订单查询 API 可以返回地址信息；在获得地址信息后调用地址校验 API 可返回地址有效或地址无效；在地址有效后调用物流标签 API 可生成标签。

系统预测直接调用物流标签 API 的后果为参数缺失错误，预测调用订单查询 API 的后果为返回地址信息。因此，系统选择调用订单查询 API。若订单查询 API 返回地址字段为空，则系统将“有订单号、缺少地址、调用订单查询 API、地址为空”作为实际后果写入经验模型。

### 6.6 实施例六：业务系统数字执行体

数字执行体在客服工单系统中处理退款请求。情境描述包括工单类型、客户状态、订单状态、退款金额、当前权限、审批状态和历史沟通记录。候选动作包括查询订单、提交退款、升级人工、请求客户补充信息和等待客户确认。

历史经验显示，在订单状态为已发货且退款金额超过阈值的情境下，直接提交退款会返回审批拒绝；先升级人工审批可使流程进入人工审核状态；若客户地址信息缺失，请求客户补充信息会使工单进入等待客户状态。

系统基于上述历史动作与后果对应关系，预测候选动作的预期后果，并选择满足任务目标且不违反权限约束的目标动作。执行后，系统将业务流程状态变化、审批结果、权限反馈或客户响应状态作为实际后果写回经验模型。

### 6.7 实施例七：真实数字环境与仿真环境区分

在一个实施例中，真实数字环境包括实际网页应用、实际 SaaS 系统、实际 API 服务、实际数据库、实际操作系统、实际文件系统、实际工具链或实际运行中的虚拟世界实例。执行主体在真实数字环境中执行动作后，会使真实运行系统的状态产生可观测变化，例如页面跳转、接口返回、数据库记录变化、文件生成、业务流程状态改变或虚拟世界对象状态改变。

仿真环境则可以包括离线测试沙盒、游戏引擎基于预设规则计算的模拟场景、数字孪生仿真或仅根据预设参数计算动作结果的环境。本发明所述真实交互区别于仅在仿真规则中计算结果的动作。

### 6.8 实施例八：经验共享

如图7所示，执行主体 A 在仓库 A 中积累“纸箱湿滑、侧向夹取、纸箱滑落”的经验记录，并将该经验记录或其增量部分共享至执行主体 B。执行主体 B 在仓库 B 中遇到相似纸箱情境时，检索到该经验记录，预测侧向夹取存在滑落风险，改用底部托举动作。

经验共享可以基于通信网络、云端经验库、边缘节点或主体间同步机制实现。共享时可以保留来源主体、时间戳、环境类型、可信度和适用条件，以便接收主体在融合经验时进行加权。

### 6.9 实施例九：局部修正

在一个实施例中，系统预测“向外拉抽屉”的预期后果为抽屉打开，但实际后果为未打开并检测到锁止。系统不修改所有抽屉相关记录，而是仅新增或修正与带锁抽屉、阻力异常或锁止状态相关的记录或权重。

通过局部修正，系统可以保留普通抽屉向外拉能够打开的经验，同时增加带锁抽屉直接向外拉无法打开的经验，从而避免一个异常后果污染无关经验。

## 七、与现有方案的区别说明

### 7.1 与强化学习或情景记忆方案的区别

现有强化学习或情景记忆方案通常将经验作为训练样本，用于更新策略网络、价值网络或其他模型参数。其记录中的后果可能表现为奖励值、下一状态或成功失败标签。本发明中的经验模型以真实交互形成的情境描述、动作和后果之间的对应关系作为可检索经验基础，后果为可用于预测候选动作影响的经验字段。本发明不要求通过策略网络直接输出动作，而是可以基于相似经验记录对候选动作的预期后果进行类比预测，并在实际执行后按记录粒度反馈更新。

### 7.2 与路径复用或工作流复用方案的区别

现有路径复用、工具调用路径复用或 GUI 工作流复用方案通常存储成功路径、动作序列、工具序列、工作流或操作轨迹，用于在相似任务中复用路径。本发明并非复用既有路径本身，而是基于相似情境下历史动作与后果之间的对应关系，预测候选动作在当前情境下的预期后果。即使最终动作序列与某条历史路径相同，本发明的动作确定依据仍可以是候选动作预期后果，而非简单重放历史路径。

### 7.3 与 action store 或 skill registry 的区别

现有 action store 或 skill registry 通常用于存储可执行动作、动作版本、动作层级、性能指标或访问策略，并根据执行反馈调整动作库。本发明中的后果不是动作库性能指标或动作版本管理信号，而是经验记录的核心字段和预测对象。候选动作可以来自 action store 或 skill registry，但本发明关注的是在当前情境下基于历史动作后果经验预测候选动作会产生什么后果。

### 7.4 与 case-based planning 的区别

现有 case-based planning 或 decision pattern map 通常基于案例、状态空间模式或历史方案进行动作建议。本发明的基础单元不是一般案例或状态空间模式，而是由真实交互形成的情境描述、动作和后果记录；本发明的目标不是直接给出历史方案，而是预测候选动作的预期后果，并根据预期后果选择、排序、剔除或调整动作。

### 7.5 与机器人可供性学习和动作效果预测的区别

现有机器人可供性学习、对象-动作-效果关系、上下文可供性或对象-动作复合体等方法，通常用于表示物理对象在特定动作下可能产生的效果，或用于训练分类器、神经网络、规划模型或动作模型。该类方法通常面向机器人与物理对象之间的可供性关系。

本发明与其不同之处在于：本发明以真实交互形成的情境-动作-后果对应关系作为可检索经验基础；情境不限于物理对象属性，还包括真实数字环境中的页面状态、接口状态、工具状态、数据状态、权限状态和业务状态；动作不限于机器人动作，还包括 GUI 操作、API 调用、工具调用、文件操作和业务级数字动作；后果不限于物理对象状态变化，还包括页面变化、接口返回、错误信息、数据库变化和流程状态变化。本发明在动作确定前基于相似历史经验记录预测候选动作的预期后果，并在真实执行后将实际后果反馈入经验模型。

### 7.6 与 Web 智能体经验记忆和动作模拟的区别

现有 Web 智能体或 LLM 智能体可以在网页任务中记录历史失败、任务轨迹、网页状态转移或页面交互经验，并在后续任务中进行动作模拟、计划生成或提示词改进。该类方案通常服务于网页导航或网页任务完成，其经验主要表现为网页状态转移图、网页交互摘要、认知地图或网页环境约束。

本发明并不限定于网页任务，也不是单纯记忆失败经验或模拟未来操作序列。本发明以真实交互形成的情境-动作-后果对应关系作为经验模型基础，在动作确定前基于相似历史经验预测候选动作的预期后果，并在真实物理环境或真实数字环境执行后将实际后果反馈入经验模型。即使候选动作由 LLM、Web Agent 或工具调用框架生成，只要其动作选择、排序、过滤或参数调整基于上述预期后果预测完成，仍属于本发明的实施方式。

## 八、有益效果

与现有技术相比，本发明至少具有以下有益效果：

1. 降低对精确物理建模、完整因果规则和高保真仿真的依赖。
2. 使执行主体能够基于真实交互经验预测候选动作后果。
3. 使经验可跨空间、跨主体、跨物理场景和数字场景复用。
4. 通过实际后果反馈实现经验模型自进化。
5. 通过局部修正降低错误更新对无关经验的影响。
6. 通过影子执行降低真实执行中的失败风险和安全风险。
7. 支持物理执行主体与数字执行主体使用统一的动作后果经验机制。

关于相似检索和类比推断，由 5.1、5.3、6.1 至 6.6 支持。关于跨场景迁移，由 5.6、6.8 支持。关于局部修正，由 5.5、6.9 支持。关于影子执行，由 5.4 支持。关于经验共享，由 5.6、6.8 支持。关于真实交互与仿真交互的区别，由 3.2、5.5、6.7 支持。

经验模型积累方法由 5.2、5.6 和 6.8 支持。动作后果预测方法由 5.3 和 5.7 支持。系统权利要求由 5.8 支持。计算机可读存储介质由第九章支持。
