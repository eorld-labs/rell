# P011 物理经验内化机制——说明书提纲

---

## 一、说明书定位

本案定位为母案，保护对象不是单一机器人、单一机械臂、单一 GUI agent 或单一工具调用系统，而是：

> 执行主体在真实物理环境或真实数字环境中，通过真实交互积累 `{情境描述，动作，后果}` 经验记录，并基于相似经验对候选动作后果进行类比预测、动作选择和反馈更新的通用机制。

说明书撰写时应避免把本案写窄为“机器人抓取”或“物理操作”。机器人、机械臂、无人车、无人机属于物理执行主体实施例；GUI 智能体、工具调用智能体、数字员工、游戏/虚拟世界角色、API 执行体属于数字执行主体实施例。

---

## 二、技术领域

建议写法：

本发明涉及智能体控制、机器人控制、数字执行体控制、经验学习、动作后果预测和交互反馈技术领域，尤其涉及一种基于真实交互经验记录的后果预测与经验自进化方法、经验模型积累方法、动作后果预测方法、系统及计算机可读存储介质。

可展开覆盖：

1. 物理执行主体：机器人、机械臂、移动机器人、无人机、自动驾驶设备、工业控制设备。
2. 数字执行主体：软件智能体、GUI 智能体、工具调用智能体、数字员工、虚拟角色、游戏智能体、API 调用智能体。
3. 环境类型：真实物理环境、真实数字环境。
4. 技术路线：经验记录、相似情境检索、类比后果预测、影子执行、反馈更新、经验共享。

### 2.1 物理执行主体与数字执行体的关系

说明书中建议主动定义：

> 在一些实施方式中，物理执行主体可以包括由软件智能体、控制程序或数字执行体控制的机器人、机械臂、无人机或其他物理设备。此时，数字执行体可以用于生成、选择或下发控制指令，物理执行主体通过物理机构在真实物理环境中产生动作后果。与此不同，数字执行体也可以直接在真实数字环境中执行 GUI 操作、API 调用、工具调用、文件操作或业务流程动作，并在真实数字环境中产生可观测数字后果。本发明将物理执行主体和数字执行体统一称为执行主体，并不要求二者在硬件形态或动作通道上相同，而是基于二者均满足以下共同技术条件：动作执行前存在可描述情境，执行主体能够执行动作，动作后产生可观测后果，并且情境、动作和后果之间的对应关系能够被记录、检索并用于后续候选动作后果预测和经验模型反馈更新。

防御意义：

1. 防止请求人主张“物理执行体和数字执行体不是同一类技术对象，母案概括过宽”。
2. 防止竞争方主张“机器人本来就是数字控制，所以数字执行体没有独立意义”。
3. 强化本案的统一技术抽象：不是统一硬件形态，而是统一动作前情境、动作、动作后果和经验反馈机制。

## 三、背景技术

### 3.1 现有基于规则/模型计算的动作决策方式

现有机器人或自动控制系统通常依赖传感器建模、动力学模型、几何约束、控制规则或仿真引擎计算动作后果。该类方式在环境变化、模型不完备、传感误差或交互对象复杂时，容易出现计算成本高、泛化差、对建模精度依赖强的问题。

防御点：

> 本案不是利用第一性原理物理公式或预设仿真规则计算后果，而是利用真实交互沉淀的经验记录进行类比预测。

### 3.2 现有强化学习/情景记忆方式

现有强化学习、情景记忆或经验回放方案常以 `{状态，动作，奖励，下一状态}` 或类似四元组记录训练数据，并通过策略网络、价值网络、LSTM、DQN、GNN 等模型进行动作选择。其经验通常服务于模型参数训练或策略优化。

应写入区别：

> 该类方法中的“后果”通常表现为奖励值、成功/失败标签、下一状态或训练样本，其最终动作选择依赖神经网络前向推理或策略优化。本发明中的经验模型以 `{情境描述，动作，后果}` 为基础经验记录，其中后果为结构化的可观测后果描述；本发明通过相似经验类比预测候选动作后果，而非通过策略网络输出动作。

### 3.3 现有路径复用/工作流复用/工具调用记忆方式

现有系统还可能记录路径、工作流、工具调用序列、GUI 轨迹、历史案例、成功规划路径或动作库，并在相似任务中复用。

应写入区别：

> 该类方案主要解决“复用哪条路径”“调用哪个工具”“选择哪个动作版本”或“恢复哪段工作流”的问题，并不以动作执行后的可观测后果作为基础经验字段，也不针对候选动作的预期后果进行类比预测。

### 3.4 现有 action store / case-based planning 方式

现有 action store、skill registry、case-based planning 或 decision pattern map 可能根据执行结果、性能指标或状态空间变化更新动作库或决策模式。

应写入区别：

> 该类方案虽然可能记录动作执行结果或状态空间变化，但其结果通常作为动作库管理信号、性能指标、模式变化或案例评分使用，并非作为 `{情境描述，动作，后果}` 经验记录中的“后果”字段参与下一次候选动作后果预测。

### 3.4A 现有案例推理 CBR 方式

现有案例推理（Case-Based Reasoning, CBR）通常是一种通用问题求解方法，其基本流程可以包括检索相似历史案例、复用或适配历史案例中的解决方案、测试或修正方案以及保留新案例。该类方法与本案在“检索历史经验并保留新经验”这一抽象层面存在表面相似性，因此是本案需要主动区分的底层近似方向。

应写入区别：

> 现有案例推理方法通常以问题和解决方案为主要案例结构，通过检索相似历史案例并复用或适配其中的解决方案来求解当前问题。本发明与其不同之处在于，本发明中的经验记录以真实交互形成的 `{情境描述，动作，后果}` 对应关系为基础；系统检索历史经验记录的目的不是直接取得待复用或待适配的解决方案，而是基于历史动作与后果之间的对应关系预测当前候选动作的预期后果，并至少基于该预期后果确定目标动作。进一步地，本发明在真实执行后将实际后果反馈入经验模型，并可仅修正与本次情境相关的记录或权重，从而使经验模型在后续相似情境中改变后果预测和动作确定。

防御口径：

1. 不应主张 CBR 绝对不能存储动作或结果，而应主张 CBR 的通用框架并不必然公开本案的真实交互后果预测闭环。
2. 区别重点放在检索目的、后果地位、动作确定依据和反馈更新方式。
3. P010 或空间语义先验可作为情境描述来源，但不写成本案必要条件。

### 3.4B 现有相似度检索/k-NN/向量检索方式

现有 k-NN、向量数据库、相似度检索或检索增强生成方法，通常根据查询样本与历史样本之间的特征距离、向量距离或语义相似度，返回相似样本、分类标签、回归值、文本片段或参考资料。该类技术可作为本发明中检索相似历史记录的一种实现手段，但其本身并不等同于本发明的类比推断机制。

应写入区别：

> 现有相似度检索通常以返回相似样本、相似文本、分类标签、回归值或参考内容为目标。本发明中的相似检索并非目的本身，而是候选动作后果预测的前置步骤；系统检索的是与当前情境相似的历史交互记录，并基于所述历史交互记录中的动作与后果之间的对应关系，对当前候选动作的预期后果进行类比推断。所述预期后果可以包括后果类型、后果分布、置信度、风险等级、约束影响或后续情境变化，并可进一步用于动作选择、动作排序、动作剔除、参数调整或影子执行。本发明的技术实质不在于采用某一种相似度计算公式，而在于将真实交互形成的情境-动作-后果对应关系用于候选动作后果预测和真实执行后的反馈更新。

防御口径：

1. 不应把“语义匹配”写成唯一差异，因为向量检索本身也可以做语义匹配。
2. P010 或空间语义先验可作为情境描述来源或相似度计算特征，但不作为本案必要条件。
3. 区别重点放在检索对象是历史动作-后果对应关系，输出是候选动作预期后果，后续还用于动作确定、影子执行和反馈更新。

### 3.5 现有机器人可供性学习和动作效果预测方式

现有机器人可供性学习、对象-动作-效果关系、上下文可供性、对象-动作复合体或动作模型学习方案，可能表示物理对象、动作和动作效果之间的关系，并用于机器人操作、规划、控制或动作效果预测。

应写入区别：

> 该类方案通常面向物理机器人与物理对象之间的可供性关系，或用于训练分类器、神经网络、动作模型或规划域模型。本发明以真实交互形成的 `{情境描述，动作，后果}` 记录作为可检索经验基础，所述情境、动作和后果不限于物理对象可供性，还覆盖真实数字环境中的页面状态、接口状态、工具状态、权限状态、数据状态和业务状态。本发明在动作确定前基于相似历史经验记录预测候选动作的预期后果，并在真实执行后将实际后果反馈入经验模型。

### 3.6 现有 Web 智能体经验记忆和动作模拟方式

现有 Web 智能体或 LLM 智能体方案可能在网页任务中记录页面交互轨迹、失败经验、网页状态转移或认知地图，并在真实执行前对候选网页动作进行前瞻模拟。例如，某些方案将网页交互结果作为 experience-driven memory，并结合 look-ahead action simulation 选择网页动作。

应写入区别：

> 该类方案通常服务于网页导航或网页任务完成，其经验主要表现为网页状态转移图、网页交互摘要、认知地图或页面环境约束。本发明并不限定于网页导航任务，也不限定于浏览器页面状态转移；本发明以真实交互形成的情境描述、动作和后果之间的对应关系作为经验模型基础，所述情境、动作和后果既可以来自真实数字环境，也可以来自真实物理环境；所述后果不仅包括页面跳转或页面元素变化，还包括接口返回、工具调用结果、文件变化、数据库状态、业务流程状态、权限反馈、物理对象状态变化等。本发明在动作确定前基于相似历史经验记录预测候选动作的预期后果，并在真实执行后基于实际后果更新经验模型。

## 四、发明要解决的技术问题

建议归纳为三层：

1. 如何使执行主体在不依赖完整物理规则、完整环境模型或仿真引擎的情况下，预测候选动作在当前情境下可能产生的后果。
2. 如何将真实交互产生的后果沉淀为可复用、可迁移、可局部修正的经验模型。
3. 如何使不同物理空间或数字环境中的经验可被相似情境复用，并在执行后继续反馈更新。

可写成：

现有技术缺少一种以真实交互形成的 `{情境描述，动作，后果}` 记录为基础，直接面向动作后果预测的经验自进化机制。现有方案或依赖规则/仿真计算，或依赖神经网络策略训练，或仅复用路径、动作库、工具序列或案例评分，不能在当前情境下对候选动作的预期后果进行类比预测并形成持续反馈闭环。

## 五、发明内容

### 5.1 总体技术方案

本发明提供一种基于交互经验类比检索的后果预测与经验自进化方法，包括：

1. 获取目标空间或目标数字环境的经验情境描述信息。
2. 基于经验模型，对候选动作在当前情境下的预期后果进行预测。
3. 至少基于预期后果，从候选动作中确定目标动作。
4. 执行目标动作，并将 `{经验情境描述信息，目标动作，实际后果}` 作为新记录反馈入经验模型。

### 5.2 经验模型定义

经验模型为基于真实交互积累的多个经验记录的集合，每条经验记录至少包括：

1. 情境描述：描述动作发生前的环境、对象、状态、任务约束或上下文。
2. 动作：执行主体可执行的物理动作、数字动作或复合动作。
3. 后果：动作执行后产生的可观测状态变化、执行结果、约束影响、异常、成功/失败原因或其他后果描述。

说明书中要强调：

> 后果不等同于单一奖励值、动作评分、工作流完成标签或动作库性能指标，而是用于预测和比较候选动作影响的经验字段。

### 5.3 情境描述的范围

物理环境中的情境描述可包括：

- 空间语义信息、物体类别、物体相对位置、可操作部件、接触关系、遮挡关系。
- 任务目标、约束条件、安全边界。
- 执行主体位姿、末端执行器状态、载荷、速度、接触状态。

数字环境中的情境描述可包括：

- GUI 页面状态、控件位置、控件语义、焦点状态、输入字段内容。
- API 状态、工具可用性、权限状态、任务上下文、数据对象状态。
- 虚拟世界/游戏/仿真平台中的对象状态、场景规则、角色状态。

### 5.4 动作的范围

动作可包括：

- 物理动作：抓取、推动、拉动、旋转、移动、避障、点击实体按钮、开关设备。
- 数字动作：点击、输入、拖拽、选择、调用工具、调用 API、修改文件、提交表单、执行脚本。
- 复合动作：由多个子动作构成的动作序列。

### 5.5 后果的范围

后果可包括：

- 物理后果：物体被移动、掉落、卡住、倾斜、破损、路径偏移、力反馈异常、任务完成或失败。
- 数字后果：页面跳转、字段写入成功、API 返回错误、权限拒绝、数据状态改变、文件生成、流程进入下一状态。
- 约束后果：触发安全约束、违反任务约束、导致不可逆改变、产生需要回滚的状态。

## 六、核心流程实施方式

### 6.1 经验情境描述信息获取

可由以下任一方式获得：

1. 传感器、摄像头、深度相机、力传感器或定位模块。
2. 空间语义系统输出，例如目标空间中的对象、结构、可操作部件和相互关系。
3. GUI 截图、DOM、辅助功能树、应用状态、日志、API 元数据。
4. 用户指令、任务描述、系统状态或历史上下文。

注意：

> 本案不主张空间语义先验的生成方法本身，说明书可写为“可由外部空间语义系统、视觉识别系统、环境感知系统或数字环境解析系统提供”。

### 6.2 候选动作生成

候选动作可来自：

1. 执行主体的动作库。
2. 任务规划模块。
3. 人工输入或示教。
4. 工具注册表或 API 列表。
5. 历史经验中出现过的可迁移动作。

### 6.3 相似历史记录检索

可基于以下相似度：

1. 情境向量相似度。
2. 结构化字段匹配。
3. 对象关系图相似度。
4. GUI 状态/控件语义相似度。
5. 任务目标和约束相似度。

检索结果可以是一条或多条历史经验记录；可设置相似度阈值、时间衰减、可信度权重或主体来源权重。

### 6.4 候选动作后果类比预测

对候选动作的预测可包括：

1. 查找历史记录中与候选动作相同或相似的动作。
2. 提取历史记录中的后果字段。
3. 根据当前情境与历史情境的差异，对历史后果进行映射、修正或置信度调整。
4. 输出候选动作对应的预期后果及置信度。

关键防御句：

> 类比预测不要求构建完整因果规则模型，也不要求根据第一性原理计算动作后果。

### 6.5 动作确定

目标动作可根据以下因素确定：

1. 预期后果是否满足任务目标。
2. 预期后果是否满足安全约束或权限约束。
3. 后果置信度。
4. 执行成本、时间成本、资源消耗。
5. 可逆性或失败代价。

### 6.6 影子执行

影子执行用于在真实执行前对候选动作进行低成本验证。影子执行可不依赖物理仿真引擎，而是基于经验模型中的历史后果链进行。

可写成：

基于候选动作对应的预期后果，构造一个或多个后续情境，并继续检索经验模型以推演后续动作或后续风险。如果预期后果链触发安全约束、权限约束或任务失败条件，则剔除该候选动作或调整动作参数。

### 6.7 真实执行与后果采集

真实执行包括：

1. 在真实物理环境中执行动作并采集可观测物理后果。
2. 在真实数字环境中执行动作并采集可观测数字后果。

应明确：

> 真实数字环境是指实际软件系统、网页、应用、操作系统、API、数据库、工具链或虚拟世界运行实例，而不是仅由预设仿真规则计算后果的离线仿真环境。

### 6.8 反馈更新

执行后将 `{当前情境描述，目标动作，实际后果}` 作为新记录写入经验模型。

若实际后果与预期后果不一致，可：

1. 新增一条经验记录。
2. 修正相似历史记录的权重。
3. 降低错误预测来源记录的可信度。
4. 仅更新与本次情境相关的记录，不修改无关记录。

## 七、经验模型积累方法

对应独权8。

### 7.1 数据采集

采集多个执行主体、多个空间、多个数字环境中的真实交互记录。每条记录至少包括：

- 交互前情境描述。
- 执行动作。
- 交互后实际后果。

### 7.2 组织方式

经验模型可按以下方式组织：

1. 向量索引。
2. 图结构索引。
3. 表结构数据库。
4. 事件日志。
5. 分层记忆结构。
6. 分布式经验库。

注意写法：

> 组织方式不改变经验记录的基础结构，均以 `{情境描述，动作，后果}` 为可检索单元。

### 7.3 输出与复用

经验模型可输出给同一执行主体或其他执行主体，用于后续候选动作后果预测。

## 八、动作后果预测方法

对应独权9。

### 8.1 输入

1. 经验模型。
2. 当前经验情境描述。
3. 待评估候选动作。

### 8.2 处理

1. 检索情境相似度最高的历史记录。
2. 匹配候选动作与历史动作。
3. 基于历史动作-后果对应关系，类比推断候选动作预期后果。

### 8.3 输出

输出可包括：

- 预期后果描述。
- 后果置信度。
- 风险等级。
- 推荐/不推荐标记。
- 可执行参数建议。

## 十一、具体实施例

### 实施例一：机械臂抽屉操作

情境：

- 目标对象为抽屉。
- 抽屉具有把手。
- 当前任务为打开抽屉。
- 候选动作包括向外拉、向内推、向上抬、旋转把手。

经验记录示例：

| 情境描述 | 动作 | 后果 |
|----------|------|------|
| 抽屉正面、把手可抓取、滑轨结构 | 向外拉 | 抽屉打开 |
| 抽屉正面、把手可抓取、滑轨结构 | 向内推 | 抽屉未打开，产生阻力 |
| 抽屉正面、把手可抓取、滑轨结构 | 向上抬 | 抽屉卡滞 |

预测：

当前情境与历史抽屉情境相似，因此预测“向外拉”的后果为抽屉打开，选择向外拉作为目标动作。

反馈：

若实际抽屉未打开且检测到锁止，则新增“带锁抽屉 + 向外拉 + 未打开/受锁止约束”的经验记录。

### 实施例二：机械臂杯子抓取

情境：

- 杯子位于桌面边缘。
- 杯口向上。
- 候选动作包括垂直上提、水平推、侧向夹取。

后果预测：

历史经验表明，在类似情境下水平推动可能导致杯子倾倒或掉落，垂直上提可成功移取。因此选择垂直上提。

防御点：

该判断不依赖杯子受力的完整动力学计算，而来自相似经验中的动作-后果对应关系。

### 实施例三：移动机器人避障

情境：

- 通道中存在临时障碍物。
- 候选动作包括左绕、右绕、停止等待、直接通过。

经验模型预测各动作后果：

- 左绕：历史中类似宽度通道左侧可通过。
- 右绕：历史中右侧存在窄缝导致碰撞风险。
- 直接通过：历史中导致障碍物接触。

选择左绕，并在执行后将路径偏移、是否碰撞、是否到达目标作为实际后果写回。

### 实施例四：GUI 智能体填写表单

情境：

- 当前页面为订单退货表单。
- 页面包含订单号输入框、提交按钮。
- 候选动作包括输入订单号、点击提交、点击取消、切换页面。

经验记录示例：

| 情境描述 | 动作 | 后果 |
|----------|------|------|
| 订单退货表单、订单号为空 | 点击提交 | 页面提示订单号必填 |
| 订单退货表单、订单号为空 | 输入有效订单号 | 输入框状态变为已填写 |
| 订单退货表单、订单号已填写 | 点击提交 | 进入退货确认页面 |

预测：

当前情境为订单号为空，预测点击提交会触发错误提示，因此先执行输入订单号。

反馈：

若提交后返回权限错误，则将“订单号已填写 + 点击提交 + 权限错误”作为后果写回。

### 实施例五：工具调用智能体

情境：

- 任务为生成配送标签。
- 可用工具包括订单查询 API、地址校验 API、物流标签 API。
- 当前上下文缺少收件地址。

经验预测：

历史经验显示，在缺少收件地址时直接调用物流标签 API 会返回参数缺失错误；先调用订单查询 API 可获得地址信息。

目标动作：

先调用订单查询 API。

实际后果：

若订单查询 API 返回地址字段为空，则写回“缺少地址 + 查询订单 + 地址为空”的后果记录。

### 实施例六：真实数字环境与仿真环境区分

真实数字环境包括：

- 实际网页应用。
- 实际 SaaS 系统。
- 实际 API 服务。
- 实际数据库。
- 实际操作系统或文件系统。
- 实际运行中的虚拟世界或游戏服务器。

仿真环境包括：

- 离线测试沙盒。
- 游戏引擎基于预设规则计算的模拟场景。
- 仅根据预设参数计算动作结果的数字孪生仿真。

说明：

本发明的真实数字环境交互是指动作执行后会使真实运行系统的状态产生可观测变化；该真实交互区别于仅在仿真规则中计算结果的动作。

### 实施例七：经验共享

执行主体 A 在仓库 A 中积累“纸箱湿滑 + 侧向夹取 + 滑落”的经验，发送给执行主体 B。执行主体 B 在仓库 B 中遇到相似纸箱情境时，检索到该经验，预测侧向夹取存在滑落风险，改用底部托举动作。

### 实施例八：局部修正

若当前情境下预测“向外拉抽屉”会打开，但实际后果为“未打开且检测到锁止”，系统不修改所有抽屉相关记录，而是仅新增或修正与“带锁抽屉/阻力异常/锁止状态”相关的记录或权重。

## 十五、撰写时必须保留的防御句

### 15.1 与强化学习/情景记忆区分

现有情景记忆引导的强化学习方法中，经验通常以 `{状态，动作，奖励值，下一状态}` 的形式存储，其后果表征为奖励值或下一状态，最终动作选择依赖策略网络、价值网络或其他神经网络模型。本发明中的经验模型以 `{情境描述，动作，后果}` 为基础经验记录，其中后果为结构化的可观测后果描述；本发明通过相似经验类比检索进行后果预测，并可在记录粒度上进行局部修正。

### 15.2 与路径复用/工作流复用区分

现有路径复用、工具调用路径复用或 GUI 工作流复用方案通常存储成功路径、动作序列、工具序列、工作流或操作轨迹，用于在相似任务中复用路径。本发明并非复用既有路径本身，而是基于相似情境下历史动作与后果之间的对应关系，预测候选动作在当前情境下的预期后果。

### 15.3 与 action store 区分

现有 action store 或 skill registry 通常用于存储可执行动作、动作版本、动作层级、性能指标或访问策略，并根据执行反馈调整动作库。本发明中的“后果”不是动作库性能指标或动作版本管理信号，而是经验记录的核心字段和预测对象。

### 15.4 与 case-based planning 区分

现有 case-based planning 或 decision pattern map 通常基于案例、状态空间模式或历史方案进行动作建议。本发明的基础单元不是一般案例或状态空间模式，而是由真实交互形成的 `{情境描述，动作，后果}` 记录；本发明的目标不是直接给出历史方案，而是预测候选动作的预期后果。

### 15.4A 与案例推理 CBR 区分

现有案例推理方法通常以问题和解决方案为主要案例结构，通过检索相似历史案例并复用或适配其中的解决方案来求解当前问题。本发明不以直接检索解决方案为目的，而是以真实交互形成的情境-动作-后果对应关系作为经验模型基础，基于历史动作与后果之间的对应关系预测候选动作的预期后果，并至少基于该预期后果确定目标动作。进一步地，本发明在真实执行后将实际后果反馈入经验模型，并可仅修正与本次情境相关的记录或权重，区别于一般案例保留或案例库维护。

### 15.4B 与常规相似度检索/k-NN/向量检索区分

现有 k-NN、向量数据库或相似度检索方法通常根据查询样本与历史样本之间的距离或相似度，返回相似样本、分类标签、回归值、文本片段或参考内容。本发明中的相似检索不是最终目的，而是后果类比推断的前置步骤；本发明检索与当前情境相似的历史交互记录，并基于历史交互记录中的动作与后果之间的对应关系，预测当前候选动作的预期后果。所述预期后果可进一步用于目标动作确定、候选动作排序、候选动作剔除、动作参数调整、风险提示或影子执行。因此，即使本发明采用 k-NN、向量数据库或相似度检索作为底层检索工具，其技术实质仍在于基于真实交互形成的情境-动作-后果对应关系进行候选动作后果预测和反馈更新。

### 15.5 真实交互定义

真实交互是指执行主体在真实物理环境或真实数字环境中执行动作并产生可观测后果的交互。真实数字环境包括实际运行的软件系统、GUI、API、数据库、工具链、操作系统或虚拟世界运行实例；其区别于仅根据预设规则、游戏引擎规则、物理动力学规则或仿真参数计算后果的仿真动作。

### 15.6 与机器人可供性/OAE 区分

现有机器人可供性学习、对象-动作-效果关系、上下文可供性或对象-动作复合体等方法，通常用于表示物理对象在特定动作下可能产生的效果，或用于训练分类器、神经网络、规划模型或动作模型。本发明与其不同之处在于：本发明以真实交互形成的情境-动作-后果对应关系作为可检索经验基础；情境不限于物理对象属性，还包括真实数字环境中的页面状态、接口状态、数据状态、权限状态和业务状态；动作不限于机器人动作，还包括 GUI 操作、API 调用、工具调用、文件操作和业务级数字动作；后果不限于物理对象状态变化，还包括页面变化、接口返回、错误信息、数据库变化和流程状态变化。

### 15.7 与 Web Agent 经验记忆/动作模拟区分

现有 Web 智能体或 LLM 智能体可以在网页任务中记录历史失败、任务轨迹、网页状态转移或页面交互经验，并在后续任务中进行动作模拟、计划生成或提示词改进。本发明并不限定于网页任务，也不是单纯记忆失败经验或模拟未来操作序列；本发明以真实交互形成的情境-动作-后果对应关系作为经验模型基础，在动作确定前基于相似历史经验预测候选动作的预期后果，并在真实物理环境或真实数字环境执行后将实际后果反馈入经验模型。即使候选动作由 LLM、Web Agent 或工具调用框架生成，只要其动作选择、排序、过滤或参数调整基于上述预期后果预测完成，仍属于本发明的实施方式。

## 十七、业务规避攻防演练

本节用于在说明书中预埋“换名不换实质”的覆盖逻辑。竞争方在业务实现中未必使用“经验三元组”“后果预测”等术语，可能将其包装为 action memory、workflow trace、tool feedback、state delta、case pattern、execution log、skill memory 等结构。因此，说明书应从功能实质上定义本发明的保护边界。

### 17.1 规避方式一：只做数字执行体，声称不是物理经验

潜在规避：

竞争方将系统限定为 GUI agent、浏览器自动化、API agent、软件机器人、数字员工或虚拟角色，声称其不涉及物理世界，因此不落入“物理经验内化”。

防守写法：

> 本发明中的“经验”不限于物理对象之间的力学交互经验，也包括执行主体在真实数字环境中执行数字动作所产生的可观测后果。真实数字环境包括实际运行的软件系统、网页、应用、操作系统、API、数据库、工具链、虚拟世界运行实例或其他可由数字执行体交互并改变状态的环境。

落点：

只要其在真实数字环境中记录“当前状态/上下文 + 数字动作 + 执行后状态变化或返回结果”，并用于下一次相似情境下动作后果预测，就不因“数字化”而脱离本案。

### 17.2 规避方式二：不叫“情境描述”，改叫上下文、状态、观察或任务上下文

潜在规避：

竞争方将情境描述称为 context、state、observation、screen state、environment snapshot、task context、world model fragment、DOM state、API state 等。

防守写法：

> 本发明中的“情境描述”是指动作执行前能够表征执行环境、对象状态、任务约束、执行主体状态或上下文条件的信息，其名称、编码方式和数据结构不受限制。情境描述可以表现为结构化字段、文本、向量、图结构、图像特征、页面状态、接口状态、日志片段或其组合。

落点：

不让对方通过改名为 state/context/observation 绕开。

### 17.3 规避方式三：不叫“后果”，改叫反馈、结果、状态变化、delta 或返回值

潜在规避：

竞争方将动作后果称为 feedback、result、outcome、state delta、execution result、return value、error code、effect、performance signal、success/failure、tool response 等。

防守写法：

> 本发明中的“后果”是指动作执行后由真实物理环境或真实数字环境产生的可观测变化、返回结果、状态差异、约束影响、异常信息、错误信息、成功或失败信息、对象状态变化、页面状态变化、接口返回、文件或数据变化等。后果可以包含成功/失败标记，但不限于成功/失败标记；优选地，后果包括能够区分不同动作影响的结构化或半结构化描述。

注意：

这里既要覆盖 success/failure，又要保留与 D2 二值奖励的区分。因此应强调“可以包含但不限于”，并优选结构化后果描述。

### 17.4 规避方式四：用 workflow/path/trajectory 代替经验记录

潜在规避：

竞争方声称其只是记录 GUI trajectory、workflow、tool path、operation sequence、relay、automation、skill path，并不是 `{情境，动作，后果}` 经验模型。

防守写法：

> 若路径、轨迹、工作流或操作序列中的至少部分数据能够被解析或组织为动作执行前情境、所执行动作以及动作执行后后果之间的对应关系，并被用于后续候选动作的后果预测，则该路径、轨迹、工作流或操作序列可构成本发明所述经验模型的一种实现形式。

落点：

单纯“复用整条路径”不一定打；但如果其把路径拆成步骤，并根据每一步的历史结果预测当前动作结果，应纳入。

### 17.5 规避方式五：用 action store/skill registry 管理动作库

潜在规避：

竞争方使用 action store、skill registry、tool registry、capability library、action graph 等结构，声称其只是动作库管理，不是经验模型。

防守写法：

> 动作库、技能库、工具库或能力库若仅存储可执行动作本身，不构成本发明的经验模型；但当该动作库、技能库、工具库或能力库进一步存储或关联动作在特定情境下执行后的后果，并基于该后果预测候选动作在当前情境下的预期后果时，其可作为本发明经验模型的具体实现。

落点：

保持对 Microsoft action store 的区分：仅动作库管理不覆盖；一旦引入“情境下动作后果预测”，就进入本案核心。

### 17.6 规避方式六：用 LLM 或神经网络包装，不显式说类比检索

潜在规避：

竞争方使用 LLM、VLM、RAG、embedding retrieval、memory retrieval、agent memory、vector DB 等结构，声称动作选择由模型推理完成，不是类比检索。

防守写法：

> 本发明中的经验模型可以通过数据库、向量库、图数据库、日志索引、记忆库、外部存储、缓存或其他可检索结构实现。候选动作后果预测可以由检索模块、语言模型、视觉语言模型或其他推理模块辅助完成；只要其预测依据包括相似情境下历史记录中的动作与后果对应关系，即属于基于经验记录的类比后果预测。

边界：

如果对方完全没有可检索经验记录，只是神经网络参数直接输出动作，则更像现有 RL/策略网络路线，不是本案最强覆盖对象。说明书应把重点放在“可检索经验记录参与预测”。

### 17.7 规避方式七：只保存日志，预测时临时分析日志

潜在规避：

竞争方不建立名为“经验模型”的数据库，只保存 execution log、event log、audit log、trace log，预测时临时从日志中检索相似记录。

防守写法：

> 经验模型不限于预先构建的专用数据库，也可以由执行日志、事件日志、轨迹日志、审计日志或其索引构成；只要该日志或索引能够提供情境、动作与后果之间的对应关系，并被用于后续动作后果预测，即可视为经验模型。

落点：

防止对方用“我只是日志分析，不是经验库”规避。

### 17.8 规避方式八：只做事前影子执行，不做真实反馈

潜在规避：

竞争方强调其通过 shadow execution、dry run、sandbox check、policy simulation 预测动作风险，但不承认经验反馈更新。

防守写法：

> 影子执行可以作为动作确定前的可选步骤。本发明的全链路实施方式包括真实执行后的实际后果反馈；在一些实施方式中，系统也可以单独利用经验模型进行动作后果预测或影子执行验证。

落点：

独权1打完整闭环；独权9打“只使用经验模型做后果预测”的使用侧。这样即使对方只做预测/影子执行，也可能落入独权9或相关从属实施例。

### 17.9 规避方式九：把真实数字环境说成仿真或沙盒

潜在规避：

竞争方在浏览器、SaaS、API、虚拟世界或游戏服务器中执行真实动作，却称其为 digital simulation 或 virtual environment，以规避“真实交互”。

防守写法：

> 是否属于真实数字环境，不取决于该环境是否以数字形式呈现，而取决于动作是否作用于实际运行的系统实例并产生可观测状态变化。若动作会改变实际运行的软件状态、页面状态、数据状态、接口状态、文件状态、账户状态或虚拟世界实例状态，则属于真实数字环境中的真实交互。

落点：

数字空间常被称为 virtual，这里要把“虚拟呈现”与“仿真计算”分开。

### 17.10 规避方式十：只做跨主体经验共享，不做单体预测

潜在规避：

竞争方声称其核心是多机器人、多 agent 或组织级知识共享，只同步经验、案例、工具路径，不做单主体经验内化。

防守写法：

> 经验模型可以由单一执行主体积累，也可以由多个执行主体共同积累；经验模型的增量部分可以在多个执行主体之间共享、合并、去重、加权或冲突消解。共享后的经验记录可被任一执行主体用于相似情境下候选动作的后果预测。

落点：

经验共享不是另一个发明，而是 P011 的扩展使用方式。

### 17.11 规避方式十一：只预测风险/错误，不预测完整后果

潜在规避：

竞争方不输出完整后果描述，只输出风险等级、错误概率、安全评分、失败概率或 action score。

防守写法：

> 预期后果可以表现为自然语言描述、结构化状态变化、风险类别、错误类型、失败原因、成功概率、安全等级、约束违反标记或其组合。上述形式只要用于表征候选动作执行后可能产生的影响，均可作为预期后果的表达形式。

注意：

这里要避免把 P011 降格成普通 action scoring。因此正式说明书中应优选写“风险/评分是后果表达的一种附加形式”，而不是唯一形式。

### 17.12 规避方式十二：业务层只说“推荐下一步动作”

潜在规避：

产品文案不说后果预测，只说 next best action、recommended action、copilot suggestion、automation suggestion。

防守写法：

> 动作确定可以直接输出目标动作，也可以输出推荐动作、候选动作排序、动作参数建议或禁止动作列表。若上述输出至少部分基于候选动作的预期后果，则属于本发明中“至少基于预期后果确定目标动作”的实现。

落点：

从产品形态上覆盖 copilot/agent assistant/automation suggestion。

### 17.13 规避方式十三：不把三元组存成同一条记录

潜在规避：

竞争方将情境、动作、后果分别存储在不同表、不同日志、不同向量索引或不同服务中，声称不存在 `{情境，动作，后果}` 记录。

防守写法：

> 本发明中的 `{情境描述，动作，后果}` 记录不要求三个字段物理存储于同一数据库行、同一文档或同一数据对象中。只要系统能够通过索引、标识符、时间戳、事件链、会话标识、执行编号或其他关联关系，将动作执行前情境、所执行动作以及动作执行后后果建立可检索对应关系，即可构成本发明所述经验记录。

落点：

防止对方用“分表存储、事件溯源、微服务拆分、日志链路”规避三元组结构。

### 17.14 规避方式十四：只保留统计聚合，不保存原始单条经验

潜在规避：

竞争方不保存单条历史经验，而是保存某类情境下某动作的成功率、失败率、风险分布、后果类别分布或置信度模型。

防守写法：

> 经验模型可以包括原始经验记录，也可以包括由多个原始经验记录聚合得到的统计表示、概率表示、后果分布、风险分布、置信度权重或经验摘要。若该聚合表示仍表征特定情境下动作与后果之间的对应关系，并被用于候选动作后果预测，则属于本发明经验模型的等同实现。

边界：

这里要避免覆盖纯黑箱模型参数训练。因此正式说明书中应强调：聚合结果仍应可追溯或可解释为“情境-动作-后果对应关系”，而非仅为不可解释的神经网络参数。

### 17.15 规避方式十五：只调整动作参数，不改变动作类型

潜在规避：

竞争方声称系统并不选择动作，只是为既定动作推荐力度、速度、方向、点击位置、输入顺序、API 参数或重试次数。

防守写法：

> 本发明中的“候选动作”包括动作类型以及动作参数化实例。确定目标动作可以包括从多个动作类型中选择，也可以包括为同一动作类型确定执行参数、执行顺序、执行位置、执行时机、重试策略或调用参数。

落点：

物理侧覆盖“用多大力、朝哪个方向、走多快”；数字侧覆盖“点哪个控件、传什么参数、先后顺序、是否重试”。

### 17.16 规避方式十六：只预测多步计划后果，不预测单步动作后果

潜在规避：

竞争方把动作包装成 plan、task、workflow、macro action、skill chain，只预测整个计划的结果，不声明预测单个动作的后果。

防守写法：

> 本发明中的“动作”可以为原子动作，也可以为由多个子动作构成的复合动作、宏动作、技能、工具链、工作流或计划片段。动作后果可以为单步后果，也可以为复合动作执行后的阶段性后果、最终后果或后果链。

落点：

把“动作”扩展到复合动作，避免对方用 workflow/plan 粒度绕开。

### 17.17 规避方式十七：后果延迟出现，声称不是动作直接后果

潜在规避：

竞争方记录的是动作后一段时间才出现的结果，例如订单状态稍后更新、机器人抓取后运输途中滑落、API 异步任务数分钟后失败，声称这不是动作执行后的即时后果。

防守写法：

> 本发明中的“后果”不限于动作执行完成瞬间出现的即时结果，也可以包括在预设时间窗口、任务阶段、异步回调、后续状态轮询或后续观测中获得的延迟后果。延迟后果可通过时间戳、任务标识、回调标识或事件链与对应动作关联。

落点：

数字业务里大量是异步任务、队列、回调、状态轮询；物理业务里也有延迟失败，这一条很必要。

### 17.18 规避方式十八：有人类确认，声称不是自动确定目标动作

潜在规避：

竞争方在系统推荐动作后要求人类确认、审批或点击执行，声称最终动作由人决定，不是系统确定。

防守写法：

> 确定目标动作可以包括自动确定并执行，也可以包括输出目标动作、推荐动作、禁止动作、动作排序、动作参数建议或风险提示供人类或其他系统确认。人类确认不改变系统基于预期后果形成动作建议或动作排序的技术过程。

落点：

覆盖半自动 agent、copilot、RPA assistant、机器人遥操作辅助等更常见的商业形态。

### 17.19 规避方式十九：经验模型为预置或第三方导入，不是本机积累

潜在规避：

竞争方使用出厂预置经验库、云端共享经验库、第三方经验包、公开数据集或其他主体上传经验，声称当前执行主体没有“积累”经验。

防守写法：

> 经验模型可以由当前执行主体自身积累，也可以由其他执行主体、云端系统、生产者、服务端、训练系统或第三方系统基于真实交互记录预先积累，并在当前执行主体使用前被下载、导入、同步或初始化。当前执行主体使用该经验模型进行候选动作后果预测，不要求其亲自生成经验模型中的每一条记录。

落点：

这对商业落地很关键：很多产品会卖“预训练/预置经验包”，这一条能把“出厂即会”的形态纳入。

### 17.20 规避方式二十：只记录失败禁忌或安全规则，不记录成功经验

潜在规避：

竞争方只记录“不要做什么”“哪些动作会失败”“哪些组合会违规”，并以黑名单、安全规则、guardrail、policy memory 形式使用。

防守写法：

> 所述后果可以为正向后果，也可以为负向后果；经验模型可以包括成功经验、失败经验、异常经验、安全约束经验或禁止动作经验。基于负向后果剔除候选动作、降低候选动作排序或调整动作参数，属于基于预期后果确定目标动作的一种方式。

落点：

安全类 agent 很可能以 guardrail/policy memory 出现；如果其规则来自历史真实后果并用于动作筛选，应纳入。

## 十八、业务规避攻防总结

正式说明书中建议形成三句总括：

1. 本发明不受“物理/数字”环境形态限制，凡执行主体可感知、可动作、可观察后果的真实运行环境，均可构成本发明的交互环境。
2. 本发明不受“情境、动作、后果”字段名称限制，凡能够表征动作执行前条件、所执行动作以及动作执行后影响的可检索对应关系，均可作为经验记录。
3. 本发明不排除使用 LLM、VLM、RAG、向量数据库、图数据库或其他模型作为辅助模块，但其区别于纯策略网络或纯工作流复用之处在于：候选动作选择前，系统基于相似历史经验记录对候选动作的预期后果进行类比预测。
4. 本发明不要求经验记录以单条显式三字段形式保存，也不要求后果为即时完整描述；分布式关联、统计聚合、复合动作、延迟后果、人类确认或预置经验库，均不改变其基于真实交互经验预测候选动作后果的技术实质。

## 十九、业务逻辑剥离攻防演练

本节用于把 P011 的技术层从具体业务流程中剥离出来。许多商业产品不会以“空间经验内化”或“动作后果预测”命名，而会表现为客服自动化、订单处理、投放优化、审批流、CRM 推荐、RPA、业务 Copilot、数字员工等形态。说明书应明确：业务目标、业务规则、业务字段和业务 KPI 不是本发明的限定核心；本发明的核心在于执行主体对真实物理环境或真实数字环境中的候选动作后果进行经验预测，并基于该预测确定动作。

### 19.1 剥离总原则

可在说明书中设置如下总括：

> 本发明不以特定业务领域、业务流程、业务规则或业务目标为限定。业务系统中的订单、工单、客户、合同、病例、审批、营销活动、库存、物流、账户、权限、预算等信息，可以作为经验情境描述、任务约束或动作后果的一部分，但本发明的技术实质在于：执行主体基于真实交互形成的经验记录，对候选动作在当前情境下的预期后果进行类比预测，并据此确定动作或动作参数。

### 19.2 规避方式一：声称只是客服/订单/CRM 工作流

潜在规避：

竞争方把系统包装成客服工单处理、订单退货、CRM 跟进、销售线索处理或企业流程自动化，声称其解决的是业务流程问题，不是动作后果预测。

防守写法：

> 当业务工作流中的步骤被执行主体转化为真实数字环境中的可执行动作，例如点击、输入、提交、调用接口、读取数据、写入字段、生成文件或发送请求，并且系统基于历史执行后果预测候选动作在当前业务情境下的执行结果时，该业务工作流中的动作后果预测层属于本发明的技术实现。

剥离判断：

业务名词可以不同，但只要存在“当前业务状态/系统状态 + 可执行数字动作 + 执行后系统状态变化/返回结果”，并用于下一步动作选择，就进入 P011 的技术层。

### 19.3 规避方式二：把后果写成业务结果或 KPI

潜在规避：

竞争方把后果表述为转化率、成交率、满意度、处理时长、投诉率、收入、成本、SLA、留存率等业务指标，声称不是动作执行后果。

防守写法：

> 后果可以包括由动作执行直接产生的环境状态变化，也可以包括基于该状态变化计算得到的任务结果、业务状态、错误类型、处理阶段、风险等级或评价指标。业务 KPI 本身不是本发明的必要限定，但当业务 KPI 或业务状态用于表征动作在当前情境下的可观测影响，并参与候选动作后果预测时，其可作为后果字段或后果评价字段的一部分。

边界：

若系统仅做离线业务报表统计，未用于执行主体的候选动作预测，则不作为本案核心覆盖对象。若业务指标反过来用于下一次动作选择，则可纳入。

### 19.4 规避方式三：用业务规则引擎掩盖经验预测

潜在规避：

竞争方声称动作由规则引擎、审批规则、路由规则、SOP、policy engine 或流程编排器决定，不是经验模型决定。

防守写法：

> 本发明可以与业务规则、权限规则、安全策略、审批规则或流程编排规则共同使用。业务规则可用于限定候选动作范围或约束动作执行；经验模型用于在候选动作范围内预测动作后果。若系统在规则筛选之外，进一步基于历史真实交互记录预测候选动作后果并据此排序、选择或调整动作，则不因同时存在业务规则而脱离本发明。

剥离判断：

纯 if-then 规则不是本案核心；“规则筛选 + 经验后果预测”是本案可打的商业实现。

### 19.5 规避方式四：把动作称为业务操作而不是技术动作

潜在规避：

竞争方称其动作是“退款”“审批”“派单”“创建标签”“关闭工单”“发送报价”“提交线索”等业务操作，不是物理动作或数字动作。

防守写法：

> 业务操作可以由一个或多个底层可执行动作实现。所述动作可以表现为业务操作、系统操作、界面操作、接口调用、数据写入、文件生成、消息发送或其组合。本发明中的动作既包括底层原子动作，也包括能够在真实数字环境中产生可观测后果的业务级复合动作。

剥离判断：

“退款”这个词是业务层；其背后调用支付 API、写入退款状态、生成审计记录，是 P011 可识别的数字动作和后果。

### 19.6 规避方式五：把情境限定为业务字段

潜在规避：

竞争方称其只使用客户等级、订单金额、工单类型、合同状态、库存数量、审批阶段等业务字段，不使用空间或环境状态。

防守写法：

> 在数字环境中，业务字段可以构成环境状态或任务上下文的一部分。订单状态、工单类型、账户权限、库存数量、合同阶段、页面字段值、接口参数、数据库记录状态等，均可以作为真实数字环境中的情境描述信息。

剥离判断：

数字空间里“业务字段”就是环境状态的一种表达。不要让对方用“这是业务数据，不是环境数据”绕开。

### 19.7 规避方式六：把系统做成垂直行业专用

潜在规避：

竞争方把系统限定在医疗、法律、金融、物流、制造、客服、教育、游戏运营等垂直行业，声称其发明点是行业模型或行业流程。

防守写法：

> 本发明可应用于不同业务领域。行业数据、行业规则或行业术语仅用于构成具体情境描述、候选动作集合或后果表达，不改变其基于真实交互经验预测候选动作后果的技术实质。

剥离判断：

行业外壳不重要；看其是否有经验记录、后果预测和动作确定。

### 19.8 规避方式七：把业务目标作为动作选择核心

潜在规避：

竞争方声称动作选择是为了最大化成交率、降低成本、提升满意度、缩短处理时间或满足 SLA，因此是业务优化，不是动作后果预测。

防守写法：

> 任务目标或业务目标可以作为动作选择的评价条件。本发明并不限定目标函数的业务含义；只要系统在选择动作前预测候选动作的预期后果，并将该预期后果与任务目标或业务目标进行比较，即属于至少基于预期后果确定目标动作。

剥离判断：

目标可以是业务的，但预测层是技术的。P011 保护的是预测动作后果并据此选动作，不是某个业务目标本身。

### 19.9 规避方式八：只提供推荐，不直接执行业务动作

潜在规避：

竞争方称其产品只是向坐席、运营、律师、医生、销售、审批人推荐下一步动作，没有自动执行真实操作。

防守写法：

> 执行主体可以为自动执行主体，也可以为辅助人类决策的半自动执行主体。系统基于经验模型输出推荐动作、动作排序、风险提示或参数建议，并由人类确认后执行的，仍然可以构成本发明中目标动作确定过程的实施方式。

剥离判断：

业务上“人点最后一下”不应切断技术过程。尤其 Copilot 类产品常见。

### 19.10 规避方式九：把动作后果外包给第三方系统

潜在规避：

竞争方动作执行在第三方 SaaS、支付平台、物流平台、邮件服务、数据库、代码仓库或云服务中完成，声称后果不是其系统产生。

防守写法：

> 动作后果可以由执行主体所在系统产生，也可以由被调用的外部系统、第三方服务、远程 API、数据库、操作系统或网络资源产生。只要执行主体能够接收、观察、读取或记录该后果，并将其用于经验模型更新或后续动作后果预测，即可构成本发明中的实际后果。

剥离判断：

P011 不要求后果由本机产生；只要执行体观察并使用后果即可。

### 19.11 规避方式十：将业务流程拆成多个微服务

潜在规避：

竞争方将情境识别、动作推荐、执行、日志记录、后果分析分别放在不同微服务、插件、浏览器扩展、云端模块或第三方系统中，声称没有单一系统实施完整方法。

防守写法：

> 本发明的方法步骤可以由同一设备执行，也可以由多个设备、多个服务、多个软件模块、浏览器扩展、云端服务器、边缘设备或第三方系统协同执行。只要上述组件协同完成经验情境获取、后果预测、动作确定和后果反馈中的相应步骤，即属于本发明的分布式实施方式。

剥离判断：

把业务系统拆成微服务不影响方法链条成立。

### 19.12 规避方式十一：声称核心是业务数据清洗或字段映射

潜在规避：

竞争方称其只是做字段抽取、字段映射、数据清洗、实体对齐、表单填充或跨系统数据同步。

防守写法：

> 数据抽取、字段映射或数据同步可以作为情境描述生成、动作参数确定或后果采集的辅助步骤。若系统进一步基于历史经验预测某一字段写入、接口调用、提交动作或同步动作在当前情境下的后果，并据此确定是否执行或如何执行该动作，则属于本发明的实施方式。

剥离判断：

单纯 ETL/同步不是本案；“基于历史后果判断这次怎么填、是否提交、提交会怎样”是本案。

### 19.13 规避方式十二：把失败归因写成业务异常分类

潜在规避：

竞争方不保存“后果”，而是保存业务异常类型，例如地址无效、余额不足、权限不足、客户未响应、合同状态不匹配、库存不足、审批驳回。

防守写法：

> 后果可以表现为业务异常类型、系统错误类型、状态拒绝原因、约束不满足原因或任务失败原因。上述异常分类若来源于动作执行后的真实反馈，并被用于后续相似情境下候选动作的后果预测或动作筛选，则属于本发明中的后果表达。

剥离判断：

业务异常就是数字环境后果的一种结构化表达。

### 19.14 规避方式十三：只在业务沙盒中验证

潜在规避：

竞争方把动作先在业务沙盒、测试账号、预生产环境、灰度环境中运行，声称不是“真实数字环境”。

防守写法：

> 真实数字环境可以包括生产环境，也可以包括预生产环境、灰度环境、测试账号环境或受控运行实例，只要该环境为实际运行的系统实例，并且动作执行会产生可观测状态变化。仅根据预设规则离线计算结果而不改变任何运行实例状态的仿真，区别于本发明中的真实数字环境。

边界：

这里要小心：纯 sandbox simulation 不要过度覆盖；但真实测试账号/灰度环境应纳入。

### 19.15 规避方式十四：把业务经验说成人类 SOP

潜在规避：

竞争方声称系统只是把人类 SOP、专家经验、业务手册、操作指南转成自动化，不是机器真实交互积累。

防守写法：

> 经验模型可以包括人工输入或专家提供的初始经验，但本发明的优选实施方式包括基于真实交互产生的实际后果对经验模型进行更新。若系统在使用 SOP 或专家经验后，继续记录动作执行后的实际后果并用于修正后续动作后果预测，则属于本发明的经验自进化机制。

剥离判断：

纯 SOP 自动化不是核心；SOP + 执行后果反馈 + 经验更新是核心。

### 19.16 规避方式十五：业务层只暴露“置信度”

潜在规避：

竞争方不输出预期后果文本，只输出“建议置信度”“成功概率”“推荐分”“风险分”。

防守写法：

> 预期后果可以直接输出为后果描述，也可以间接表达为后果类别、成功概率、失败概率、风险分、推荐分、置信度或约束满足概率。若该评分或概率来源于相似情境下历史动作后果的对应关系，则其属于预期后果的表达或评价形式。

边界：

说明书中应优先写“后果类别/后果描述 + 置信度”，避免被解释成普通推荐系统评分。

### 19.17 规避方式十六：把业务流程作为黑盒工具

潜在规避：

竞争方不展开底层业务系统，只把“退款工具”“派单工具”“审批工具”“发票工具”作为黑盒工具，记录调用成功/失败并推荐工具。

防守写法：

> 工具调用可以作为动作的一种形式。即使被调用工具内部逻辑不可见，只要系统记录调用前情境、调用动作以及调用后的返回结果、状态变化或错误信息，并基于该记录预测后续工具调用的预期后果，即属于本发明的数字执行体实施方式。

剥离判断：

这条对工具调用 agent 非常重要：不需要看到工具内部，只要记录调用前后即可。

### 19.18 规避方式十七：把业务动作写成自然语言任务

潜在规避：

竞争方让 LLM 输出“请联系客户确认地址”“建议创建退货单”“下一步检查权限”等自然语言建议，声称没有机器动作。

防守写法：

> 动作可以由自然语言、结构化指令、函数调用、脚本、API 请求、GUI 操作或其他机器可解析形式表示。当自然语言建议被系统用于驱动后续执行、人工确认执行或转换为可执行动作时，该自然语言建议可构成候选动作或目标动作的表达形式。

边界：

如果只是纯人工咨询建议，没有系统记录真实执行后果，覆盖弱；若建议进入执行闭环，覆盖强。

### 19.19 规避方式十八：把业务后果人工录入

潜在规避：

竞争方不自动采集后果，而由坐席、审核员、运营人员手工标注“成功”“失败”“客户拒绝”“已解决”等。

防守写法：

> 实际后果可以由传感器、系统日志、接口返回、页面状态、数据库状态自动获得，也可以由用户、操作员、审核员或其他系统输入确认。只要该后果与已执行动作建立对应关系并反馈入经验模型，即可构成本发明的实际后果采集方式。

剥离判断：

人工标注不应绕开经验模型，只影响后果采集来源。

### 19.20 规避方式十九：只做业务合规审查

潜在规避：

竞争方声称系统只检查动作是否合规、安全、可审计，不预测业务或技术后果。

防守写法：

> 合规审查、安全审查或审计审查可以作为动作确定前的约束条件。若系统基于历史真实执行后果预测候选动作可能导致的权限拒绝、合规失败、审计异常、数据泄露风险或流程违规，并据此剔除、排序或调整候选动作，则属于基于预期后果确定动作的一种实施方式。

剥离判断：

纯规则合规不是核心；经验化合规后果预测是核心。

### 19.21 规避方式二十：把业务流程封装为“代理人服务”

潜在规避：

竞争方以 Agent-as-a-Service、RPA-as-a-Service、business copilot、automation service 形式提供能力，用户只看到业务结果，不知道内部是否有经验模型。

防守写法：

> 本发明的系统可以部署为本地软件、云服务、插件、浏览器扩展、机器人控制服务、智能体平台服务或自动化服务。服务形态不影响其内部是否执行经验情境获取、候选动作后果预测、动作确定和后果反馈更新等步骤。

剥离判断：

产品交付形态不重要；看内部技术链条。

### 19.22 规避方式二十一：把“不动作、等待、升级人工”排除为动作

潜在规避：

竞争方声称系统并未执行业务动作，只是选择等待、暂缓、升级人工、请求补充信息、终止流程、回滚流程或保持当前状态。

防守写法：

> 本发明中的动作包括主动改变环境状态的动作，也包括等待、延迟执行、请求补充信息、升级人工、终止流程、回滚流程、维持当前状态或不执行某候选动作等控制动作。若系统基于历史经验预测执行或不执行某动作后的后果，并据此选择等待、升级、终止或回滚，则属于基于预期后果确定目标动作的一种形式。

剥离判断：

业务系统里“先不要点提交”“先升级人工”“先等客户确认”本质上也是动作选择。

### 19.23 规避方式二十二：声称业务后果由多步共同造成，不能归因于单个动作

潜在规避：

竞争方称业务结果通常由多个步骤、多个系统、多人协作或长链路共同导致，无法形成单一动作与后果的对应关系。

防守写法：

> 本发明中的动作与后果对应关系可以是一对一关系，也可以是一对多、多对一或多对多关系。对于由多个动作共同导致的后果，经验模型可以记录动作序列、复合动作、阶段性动作组或关键动作集合与后果之间的对应关系，并用于后续相似情境下的后果预测。

剥离判断：

不要求每个业务结果都精确归因到单步点击；能记录“动作组/流程片段→后果”即可。

### 19.24 规避方式二十三：用 A/B 测试、多臂老虎机或实验平台包装经验预测

潜在规避：

竞争方声称动作推荐来自 A/B test、multi-armed bandit、实验平台、灰度策略或在线优化，不是经验模型。

防守写法：

> 在线实验、A/B 测试、多臂老虎机、灰度策略或在线优化可以作为经验模型的更新或权重调整方式。若系统根据历史真实执行中不同动作产生的后果，更新特定情境下候选动作的后果分布、风险分布、成功概率或推荐权重，并据此选择后续动作，则属于本发明经验模型自进化的一种实现。

边界：

单纯统计页面转化而不形成可用于候选动作后果预测的情境-动作-后果对应关系，不作为核心覆盖；但 bandit 若以情境为条件选择动作并根据真实结果更新，则应纳入。

### 19.25 规避方式二十四：使用多租户、群体经验或匿名经验，声称不是自身业务经验

潜在规避：

竞争方将多个客户、多个企业、多个执行主体的历史结果匿名化、聚合化或社区化，形成 community model、tenant model、best practice model，声称不是当前业务系统自己的经验。

防守写法：

> 经验模型中的记录或聚合表示可以来源于当前执行主体，也可以来源于多个租户、多个用户、多个组织、多个执行主体或匿名化群体经验。来源主体、租户边界或匿名化处理不影响其作为经验模型的性质；只要该经验被用于当前情境下候选动作后果预测，即属于本发明的实施方式。

剥离判断：

SaaS 产品很常见“全网最佳实践/社区自动化推荐”，这条要压住。

### 19.26 规避方式二十五：将用户画像、客户分群或角色权限作为业务推荐逻辑

潜在规避：

竞争方将动作推荐解释为基于用户画像、客户分群、角色权限、账户等级、组织策略或个性化推荐，而不是环境情境。

防守写法：

> 用户画像、客户分群、角色权限、账户等级、组织策略、历史偏好或个性化标签可以作为情境描述信息或任务约束的一部分。若系统在相同或相似画像/权限/分群条件下，基于历史动作后果预测当前候选动作后果，则不因其使用个性化或权限信息而脱离本发明。

剥离判断：

业务画像不是另一个世界，它是数字环境上下文的一部分。

### 19.27 规避方式二十六：将预测结果提前批量生成，执行时只读取缓存

潜在规避：

竞争方离线或定时批处理生成推荐动作、风险分、下一步动作清单，执行时只是读取缓存结果，声称没有在当前动作选择时进行后果预测。

防守写法：

> 候选动作后果预测可以在动作确定时实时执行，也可以在动作确定前通过离线计算、定时任务、预计算、缓存或批处理方式生成。只要所述预计算结果在当前情境下被用于确定目标动作、推荐动作、动作排序、动作参数或禁止动作列表，即属于本发明中的基于预期后果确定动作。

剥离判断：

不要让对方用“推荐是昨晚算好的”绕开。

### 19.28 规避方式二十七：以可解释性、审计或复盘报告形式呈现经验

潜在规避：

竞争方不在主流程中显式展示经验预测，只在解释面板、审计报告、复盘报告、运营洞察或决策理由中展示“类似案例中这样做会导致某结果”。

边界：

单纯事后报表不作为核心；但解释层参与下一步动作决定或人类确认，即可纳入。

### 21.1 总体防御战略

P011 的无效防御不能只依赖“现有技术没有三元组”这一句话。应构建四层防线：

1. **结构防线**：经验模型以 `{情境描述，动作，后果}` 或可关联等同结构作为基础经验单元。
2. **功能防线**：经验记录用于预测候选动作在当前情境下的预期后果，而不是仅用于训练策略、复用路径、管理动作库或生成工作流。
3. **来源防线**：经验来自真实物理环境或真实数字环境中的真实交互后果，而不是仅由预设仿真规则计算得到。
4. **闭环防线**：实际后果反馈进入经验模型，形成可局部修正、可迁移、可共享的经验自进化机制。

### 21.25 非专利文献/开源 Agent 攻击

攻击主张：

3. 其后续使用通常是生成计划、改写提示词、提升策略或复用技能，而不是候选动作后果预测。
4. 很多开源/产品材料需核实公开日、可获得性、版本内容和证据链完整性。

进一步应把非专利攻击拆成两条主线，而不是笼统归为“智能体记忆”：

| 非专利攻击线 | 典型材料 | 近似点 | P011 防御重点 |

| 数字侧 Web Agent 经验记忆 | WebATLAS、experience-driven memory、look-ahead action simulation、web agent memory | 网页交互经验、候选动作模拟、预测未来后果 | 其多面向网页导航/任务完成；P011 是跨物理/数字空间的统一情境-动作-后果经验模型，且真实后果反馈改变后续预测和动作确定 |

说明书预埋：

> 本发明中的经验记录不是一般自然语言反思、任务摘要、提示词记忆或技能描述，而是可用于预测候选动作后果的情境-动作-后果对应关系。

补充预埋一：

> 现有机器人可供性学习、对象-动作-效果关系、上下文可供性或对象-动作复合体等方法，通常用于表示物理对象在特定动作下可能产生的效果，或用于训练分类器、神经网络、规划模型或动作模型。本发明与其不同之处在于：本发明以真实交互形成的情境-动作-后果对应关系作为可检索经验基础；情境不限于物理对象属性，还包括真实数字环境中的页面状态、接口状态、数据状态、权限状态和业务状态；动作不限于机器人动作，还包括 GUI 操作、API 调用、工具调用、文件操作和业务级数字动作；后果不限于物理对象状态变化，还包括页面变化、接口返回、错误信息、数据库变化和流程状态变化。本发明在动作确定前基于相似历史经验记录预测候选动作的预期后果，并在真实执行后将实际后果反馈入经验模型。

补充预埋二：

> 现有 Web 智能体或 LLM 智能体可以在网页任务中记录历史失败、任务轨迹或页面交互经验，并在后续任务中进行动作模拟、计划生成或提示词改进。本发明并不限定于网页任务，也不是单纯记忆失败经验或模拟未来操作序列；本发明以真实交互形成的情境-动作-后果对应关系作为经验模型基础，在动作确定前基于相似历史经验预测候选动作的预期后果，并在真实物理环境或真实数字环境执行后将实际后果反馈入经验模型。即使候选动作由 LLM、Web Agent 或工具调用框架生成，只要其动作选择、排序、过滤或参数调整基于上述预期后果预测完成，仍属于本发明的实施方式。

### 21.32 诚实信用/非真实发明创造攻击

攻击主张：

在新实施细则语境下，请求人可能尝试以诚实信用或非真实发明创造活动为由攻击，尤其当说明书过度抽象、没有具体实施例时。

防御策略：

1. 保留真实研发脉络：P010 与 P011 的关系、检索记录、技术问题演进。
2. 说明书提供足够具体的工程实施例。
3. 避免只用宏大概念，不给可执行步骤。
4. 内部存档保留检索词、近似分析、立项书、权利要求迭代记录。

说明：

这不是主要无效战场，但应避免给对方可乘之机。

### 21.33 数据合规/公共利益攻击

攻击主张：

业务系统实施例涉及用户数据、日志、医疗、金融、账号权限等，请求人可能旁敲侧击主张违法获取数据、违反公共利益或无法合法实施。

防御写法：

> 本发明中的经验记录可在授权、脱敏、匿名化、权限控制或合规审计条件下采集和使用。本发明不要求获取未经授权的数据，也不限定使用任何违法数据来源。

作用：

该内容主要用于降低专利法第五条、数据合规或公共利益方向的外围攻击风险。

### 21.34 取证困难反向攻击

攻击主张：

对方可能在无效中主张本案的“经验模型”“后果预测”是内部算法不可见，权利要求实质上无法验证，属于不清楚或保护范围不确定。

防御思路：

1. 专利权利要求不以侵权取证便利性为授权条件。
2. 说明书应定义外部可观察迹象：推荐动作随历史执行结果改变、相似情境下引用历史结果、失败后调整推荐、日志中存在前后状态关联等。
3. 业务攻防章节中已覆盖日志、解释面板、审计输出、推荐理由。

预埋句：

> 经验模型是否参与后果预测，可以通过系统输出的推荐理由、历史相似记录引用、执行后推荐变化、动作风险排序变化或审计日志中的情境-动作-后果关联关系体现。

### 21.35 分案/后续申请反噬攻击

攻击主张：

后续若基于 P011 分出工具调用经验、业务自动化经验、经验共享等分案，请求人可能用后续文件反过来解释母案范围，称母案未支持这些方向。

防御策略：

1. 母案说明书中预埋所有分案方向的原始支持。
2. 后续分案不要引入母案没有的“新技术实质”。
3. 母案中明确数字执行体、工具调用、业务系统只是同一经验后果预测机制的应用场景。

预埋句：

> 不同实施例之间的差异在于情境描述、动作形式和后果表达不同，其共同技术机制均为基于真实交互经验记录的候选动作后果预测。

### 21.36 无效二次战场：权利要求解释

攻击主张：

请求人可能在无效中主张对权利要求作窄解释或不利解释，例如：

- “后果”只应理解为物理后果。
- “真实数字环境”不包括测试账号或虚拟世界。
- “执行主体”不包括人机协同系统。
- “经验模型”必须是显式数据库。

防御策略：

说明书定义要足够明确，避免把解释权交给请求人。正式撰写中应多用：

1. “包括但不限于”。
2. “可以为”。
3. “不限于物理存储于同一记录”。
4. “真实数字环境包括实际运行的软件系统、网页、应用、API、数据库、工具链或虚拟世界运行实例”。
5. “执行主体可以为自动或半自动主体”。

### 21.37 无效章节补充总结

抽身审视后，正式说明书还应特别补强以下五类内容：

1. **非专利文献防线**：明确区别一般 agent memory/reflection/skill library。
2. **功能性限定防线**：每个功能至少给一种工程实现。
3. **全范围支持防线**：用共同技术条件限定物理/数字环境的适用边界。
4. **边界一致性防线**：真实数字环境、虚拟世界、测试环境、仿真环境之间的边界必须正面定义。
5. **文本解释防线**：术语定义要足够宽但有技术锚点，防止无效中被窄解释或被指不清楚。
