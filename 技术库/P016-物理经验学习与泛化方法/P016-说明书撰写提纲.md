## 三、背景技术

### 3.1 现有技术概述

现有机器人技能系统大体包括以下路线：

1. 原子技能或动作原语路线：将抓取、移动、释放、插入等动作作为离散技能调用。
2. 状态机或行为树路线：通过有限状态机、行为树或任务图组织机器人任务执行流程。
3. 参数化技能或轨迹模板路线：通过参数化动作原语、DMP、轨迹模板或技能模板实现一定程度复用。
4. 任务规划或动作 schema 路线：通过前提条件、效果、动作模型或技能效果模型进行任务编排。
5. 多模态执行监控路线：利用视觉、力觉、触觉、深度等数据判断执行是否成功或是否出现异常。

### 3.2 现有技术缺陷

#### 3.2.1 过程结构和环境参数混在一起

现有技术通常将物理动作的稳定过程结构与当前环境下的具体参数混写在同一个技能定义或策略模型中。例如倒水动作中“倾斜、保持、回正”的过程结构具有较强稳定性，而杯子高度、壶嘴位置、流速阈值、液面高度阈值等参数随环境和对象变化。若二者混在一起，经验难以跨环境复用。

#### 3.2.2 阶段切换容易停留在特定任务控制层

现有技术中已有机器人跳跃、行走、装配、模仿学习中的阶段检测和阶段切换，但通常用于特定运动控制或模型训练。其阶段条件往往固定在特定任务或特定模型中，未与可复用过程模板、偶然绑定和因果事实回流形成统一闭环。

#### 3.2.3 前提和效果难以与真实执行结果可靠闭合

任务规划、动作 schema、minimanipulation library 等技术中常出现 precondition/postcondition 或前提/效果，但这些事实往往用于规划表达，未必在真实执行中被独立确认。若执行后世界状态仅由程序假定更新，机器人在遮挡、接触失败、传感器漂移或执行中断时容易形成错误世界状态。

#### 3.2.4 多源验证缺少和过程模板及因果事实的耦合

多源传感器验证、视觉/力觉执行监控、抓取成功检测等技术可以判断执行异常或成功概率，但常以动作结果或传感器一致性为对象。本案要解决的是：如何将需要确认的对象限定为过程模板关联的目标因果事实，并将其最终成立状态回流到阶段推进、恢复执行或后续物理动作触发。

#### 3.2.5 通用流程引擎或配置系统不能解决真实物理执行问题

通用流程引擎、业务规则系统或配置中心可以实现流程节点编排、参数配置和状态跳转，但其通常不处理机器人执行器、被操作对象和真实环境之间的物理交互，也不基于连续物理观测值与当前环境绑定阈值触发阶段跃迁，更不会将经观测通道确认的物理世界事实状态回流到机器人后续动作控制。因此，单纯的软件流程抽象无法解决真实物理动作经验复用、可信执行和中断恢复问题。

### 3.3 本案要解决的技术问题

本案要解决的技术问题可以表述为：

如何在机器人真实物理动作执行中，将物理动作的不变过程结构与当前执行环境下的具体参数解耦，并在执行过程中利用连续状态变量观测和因果事实最终成立状态，实现物理动作经验的跨环境复用、阶段推进、恢复执行和后续物理动作触发。

可进一步拆成三个子问题：

1. 如何避免每个对象、每个环境都重新定义物理动作。
2. 如何让阶段跃迁依赖当前环境下绑定后的传感器和阈值。
3. 如何避免执行结果仅由程序假定，而是回流为可用于继续执行的事实状态。

## 四、发明内容

### 4.1 发明目的

本发明的目的在于提供一种物理动作执行与经验复用方法、系统及存储介质，使得机器人能够：

1. 使用过程模板表达物理动作的不变过程结构。
2. 通过偶然绑定数据适配当前执行环境中的具体对象、传感器和阈值。
3. 在执行过程中基于连续状态变量观测触发阶段跃迁。
4. 通过目标因果事实的最终成立状态控制阶段推进、恢复执行或后续物理动作触发。

### 4.2 技术方案总述

本发明的技术方案包括以下闭环：

1. 获取物理动作的过程模板。
2. 获取当前执行环境下的偶然绑定数据。
3. 利用偶然绑定数据填充过程模板的参数槽，生成物理动作实例。
4. 执行物理动作实例，基于连续状态变量观测值与绑定阈值触发阶段跃迁。
5. 从过程模板关联的因果事实声明中确定目标因果事实。
6. 通过至少两个独立观测通道生成目标因果事实的成立判断，并确定最终成立状态。
7. 将最终成立状态用于阶段推进、恢复执行或后续物理动作触发。

### 4.3 有益效果

本发明至少具有以下技术效果：

1. 降低跨环境复用成本：同一过程模板可在不同对象、不同传感器和不同阈值下生成不同物理动作实例。
2. 提高阶段推进可靠性：阶段跃迁不依赖抽象状态名，而依赖当前执行环境下绑定后的连续物理观测和阈值。
3. 降低错误世界状态累积风险：目标因果事实的最终成立状态由观测通道判断后回流，而非仅由程序假定更新。
4. 支持中断后恢复：可利用连续状态变量观测和事实状态匹配恢复入口阶段。
5. 支持后续动作触发：目标因果事实最终成立状态可作为后续物理动作触发或过程链继续的依据。

### 4.4 与现有技术的区别点

说明书中建议显式写出以下区别，但保持审慎，不点名具体文献：

1. 与普通状态机区别：本案不是仅定义状态和转换条件，而是将状态/阶段作为可复用过程模板的一部分，并通过偶然绑定数据在当前环境下实例化。
2. 与参数化动作原语区别：本案不是学习动作原语参数，而是通过参数槽和偶然绑定保持过程模板不变。
3. 与前提/效果规划区别：本案的目标因果事实不仅用于规划表达，还作为执行过程中需要确认并回流的事实状态。
4. 与多源验证区别：本案验证对象是过程模板关联的目标因果事实，验证结果用于阶段推进、恢复执行或后续物理动作触发。

### 4.5 与业务流程或通用软件规则的区别

说明书中应补充一段反“业务逻辑剥离”的技术性表述，避免本案被理解为普通流程引擎、配置中心或任务状态管理。

写作要点：

1. 本案处理对象是机器人真实物理动作，而非审批、工单、交易或其他非物理业务流程。
2. 过程模板中的阶段序列、连续状态变量类型和跃迁条件逻辑结构均服务于执行器、被操作对象和环境之间的物理交互。
3. 偶然绑定数据绑定的是当前执行环境中的具体传感器、具体阈值和具体被操作对象，不是普通业务参数。
4. 阶段跃迁由连续物理观测值与绑定阈值的比较触发，不是由人工业务状态或抽象事件触发。
5. 目标因果事实最终成立状态表示真实物理世界状态，并回流到阶段推进、恢复执行或后续物理动作触发。

建议表述：

本发明并非针对非物理业务流程、审批流程或通用软件任务编排的规则配置方法。本发明所述过程模板、偶然绑定数据、阶段跃迁和目标因果事实最终成立状态均用于机器人真实物理动作的执行控制。其中，过程模板描述物理动作在不同执行环境中相对稳定的过程结构；偶然绑定数据将该过程结构中的抽象传感器引用、阈值标识和对象引用绑定为当前执行环境下的具体传感器、具体阈值和具体被操作对象；阶段跃迁基于连续物理观测值与绑定阈值的比较结果触发；目标因果事实的最终成立状态通过观测通道确定后回流到阶段推进、恢复执行或后续物理动作触发。因此，本发明解决的是机器人真实物理动作经验复用和可信执行控制中的技术问题。

### 4.6 闭环数据依赖和控制依赖

说明书中应显式写出各技术特征不是简单并列拼接，而是形成数据流和控制流闭环。

建议表述：

本发明中，过程模板、偶然绑定数据、阶段跃迁和目标因果事实的最终成立状态并非彼此独立的功能模块。偶然绑定数据用于将过程模板中的参数槽具体化为当前执行环境下的传感器、阈值和对象；绑定后的传感器和阈值用于触发阶段跃迁；阶段执行过程中需要确认的目标因果事实由过程模板关联的因果事实声明确定；目标因果事实的最终成立状态又用于阶段推进、恢复执行或后续物理动作触发。因此，各技术特征之间存在明确的数据依赖和控制依赖。

### 6.2 过程模板的数据结构

过程模板至少包括：

1. 模板标识。
2. 阶段序列。
3. 每个阶段的阶段标识。
4. 每个阶段关联的连续状态变量类型。
5. 每个阶段的跃迁条件逻辑结构。
6. 参数槽。
7. 可选的阶段状态特征。
8. 可选的失败检测条件。
9. 可选的因果事实声明引用。

示例表达：

```text
process_template:
  id: pour_liquid
  stages:
    - align
    - tilt
    - maintain_flow
    - return
  variables:
    tilt_angle
    flow_rate
    liquid_level
  guards:
    tilt -> maintain_flow: flow_rate >= FLOW_MIN
    maintain_flow -> return: liquid_level >= TARGET_LEVEL
  slots:
    FLOW_MIN
    TARGET_LEVEL
    CUP_OBJECT
    CAMERA_SENSOR
```

广义实现形式：

1. 状态图。
2. 行为树。
3. 任务图。
4. 技能图。
5. 工作流。
6. 规则集合。
7. DSL 配置。
8. JSON、YAML、表格或数据库记录。
9. 图数据库节点集合。

写作重点：

1. 不限定“过程模板”这一名称。
2. 不限定文件格式、图结构形式或存储位置。
3. 只要其定义物理动作的阶段关系、连续状态变量类型、跃迁条件逻辑结构以及可由当前执行环境绑定的参数槽，即可作为本案过程模板。

2. 坐标系映射。
3. 时间窗口映射。
4. 确认窗口映射。
5. 用户偏好阈值映射。

写作重点：

1. 偶然绑定数据改变具体实例，不改变过程模板。
2. 具体传感器、具体阈值、具体对象均由偶然绑定提供。
3. 与普通参数化技能区别：本案绑定的是过程模板中的跃迁逻辑、观测来源和对象引用。
4. 偶然绑定不是普通业务参数配置，而是将抽象过程结构落到当前物理执行环境。

绑定时机和形式：

1. 运行期绑定。
2. 部署期绑定。
3. 场景加载期绑定。
4. 任务编排期绑定。
5. 现场标定绑定。
6. 历史经验推荐绑定。
7. 用户偏好绑定。
8. 环境上下文绑定。
9. 对象识别结果绑定。
10. 传感器注册表绑定。

### 6.4 物理动作实例生成

描述：

1. 读取过程模板。
2. 读取当前环境识别结果和传感器注册信息。
3. 根据对象识别结果绑定对象引用。
4. 根据可用传感器绑定传感器引用。
5. 根据环境、对象、用户偏好或历史经验绑定阈值。
6. 生成当前环境下的物理动作实例。

重点强调：

同一过程模板生成的多个实例共享阶段序列、连续状态变量类型和跃迁条件逻辑结构，仅具体绑定值不同。

### 6.5 阶段跃迁执行

描述：

1. 系统确定当前执行阶段。
2. 根据偶然绑定数据确定当前阶段所需传感器。
3. 获取连续状态变量观测值。
4. 将观测值代入绑定后的跃迁条件。
5. 若满足跃迁条件，进入下一阶段。
6. 若不满足，继续执行、等待、调整或触发失败检测。

说明书保留：

1. 简单条件。
2. 复合条件。
3. 持续时间窗口。
4. 连续确认次数。
5. 积分累计窗口。
6. 趋势判断。

阈值类型可以包括：

1. 数值阈值。
2. 区间边界。
3. 概率阈值。
4. 置信度阈值。
5. 相似度阈值。
6. 时间阈值。
7. 持续次数阈值。
8. 变化率阈值。
9. 由模型输出后用于阶段跃迁判定的判别边界。

连续状态变量来源可以包括：

1. 传感器直接读数。
2. 传感器数据处理后的连续特征值。
3. 感知模型输出的连续估计值。
4. 融合模型输出的融合值。
5. 状态估计模块输出的状态量。
6. 置信度、距离、角度、力、速度、流量、液位或对象相对位姿。

### 6.6 因果事实声明

因果事实声明可以包括：

1. 因果前提事实：动作执行前应满足的事实。
2. 因果产出事实：动作执行成功后产生的事实。
3. 因果销毁事实：动作执行成功后被改变或不再成立的事实。

示例：

抓取动作：

1. 前提事实：目标对象可见、夹爪为空。
2. 产出事实：目标对象被夹持。
3. 销毁事实：目标对象位于桌面原位置。

倒水动作：

1. 前提事实：容器已对准杯口、壶中有液体。
2. 产出事实：杯中液位达到目标高度。
3. 销毁事实：壶中液量保持原值。

说明：

本案不把前提/效果本身作为单独创新点，而是将因果事实作为执行闭环中需要被确认并回流的事实状态。

### 6.7 目标因果事实最终成立状态确定

描述：

1. 从过程模板关联的因果事实声明中确定目标因果事实。
2. 通过第一独立观测通道生成成立判断。
3. 通过第二独立观测通道生成成立判断。
4. 根据两个判断的一致性或仲裁策略确定最终成立状态。

写作边界：

1. 不将“跨层具身验真”作为本案核心发明点。
2. 可写成“可选地，事实验真模块可以调用已配置的跨层验真能力”。
3. 不使用已有案子的招牌术语作为本案权利要求核心。

可选通道：

1. 执行器内置传感器通道。
2. 力觉/触觉通道。
3. 编码器/电气状态通道。
4. 外部视觉通道。
5. 深度/环境传感通道。
6. 多模态感知模型通道。

独立观测通道的独立性来源可以包括：

1. 不同传感来源。
2. 不同特征分支。
3. 不同模型分支。
4. 不同判断逻辑。
5. 不同时间窗口。
6. 不同空间观测位置。
7. 不同数据处理链路。

最终成立状态可以包括：

1. 成立。
2. 不成立。
3. 未知。
4. 待重新观测。
5. 冲突需仲裁。
6. 置信不足。
7. 需恢复执行。

### 6.8 最终成立状态回流

最终成立状态可用于：

1. 阶段推进：确认当前阶段成功完成。
2. 恢复执行：确定中断或失败后的入口阶段。
3. 后续物理动作触发：确认下一动作前提满足。
4. 当前世界状态更新：写入产出事实或销毁事实。

重点：

这部分是本案闭环的关键。说明书要强调最终成立状态不是仅用于报警或记录，而是参与控制流程。

### 6.9 恢复执行实施例

描述：

1. 执行中断或失败。
2. 读取连续状态变量当前观测值。
3. 读取目标因果事实最终成立状态。
4. 将观测值与各阶段预设状态特征匹配。
5. 确定恢复入口阶段。
6. 从该入口阶段继续执行或执行恢复物理动作。

示例：

插入动作中断时，根据接触力、位移、姿态偏差和“接触已成立”事实，判断恢复入口为“插入阶段”而不是“对准阶段”。

### 6.10 后续物理动作触发实施例

描述：

1. 过程链包含多个物理动作实例。
2. 前一动作的因果产出事实经过验真后成立。
3. 系统将该成立事实作为后一动作的前提满足依据。
4. 触发后一动作实例。

示例：

抓取动作产出“对象已被夹持”；该事实成立后触发“移动对象”动作。

后续动作触发方式可以包括：

1. 预定义过程链触发。
2. 任务规划器触发。
3. 视觉语言动作模型触发。
4. 世界模型触发。
5. 行为树触发。
6. 规则引擎触发。
7. 其他决策模块基于目标因果事实最终成立状态触发。

### 6.11 实施例一：抓取动作

过程模板：

1. 接近阶段。
2. 对准阶段。
3. 闭合阶段。
4. 提起确认阶段。

连续状态变量：

1. 夹爪与目标距离。
2. 夹爪闭合距离。
3. 夹持力。
4. 目标视觉位姿。

偶然绑定：

1. 目标对象绑定到当前识别对象。
2. 力阈值绑定到对象材质/重量。
3. 视觉传感器绑定到当前可用相机。

目标因果事实：

1. 目标对象被夹持。

最终成立状态回流：

1. 成立：进入移动阶段。
2. 不成立：恢复到对准或闭合阶段。

### 6.12 实施例二：倒水动作

过程模板：

1. 定位阶段。
2. 倾斜阶段。
3. 保持流动阶段。
4. 回正阶段。

连续状态变量：

1. 倾角。
2. 流速。
3. 液位高度。
4. 壶嘴到杯口距离。

偶然绑定：

1. 杯对象。
2. 壶对象。
3. 液位传感/视觉通道。
4. 目标液位阈值。
5. 用户偏好阈值，例如半杯、七分满。

目标因果事实：

1. 杯中液位达到目标高度。

最终成立状态回流：

1. 成立：进入回正阶段或触发下一动作。
2. 不成立：继续保持或触发停止/恢复。

### 6.13 实施例三：插入或装配动作

过程模板：

1. 对准阶段。
2. 接触阶段。
3. 插入阶段。
4. 到位确认阶段。

连续状态变量：

1. 姿态偏差。
2. 接触力。
3. 位移。
4. 阻抗变化。

偶然绑定：

1. 插头对象。
2. 插座对象。
3. 力传感器。
4. 接触阈值。
5. 位移阈值。

目标因果事实：

1. 插头已到位。

最终成立状态回流：

1. 成立：触发释放或下一任务。
2. 不成立：回到对准或接触阶段。

### 6.14 实施例四：云端经验库与机器人端协同执行

部署结构：

1. 云端经验库保存物理动作过程模板和历史执行记录。
2. 边缘端或部署端生成或校验偶然绑定数据。
3. 机器人端执行物理动作实例并获取连续物理观测。
4. 内部或外部事实验真模块生成目标因果事实成立判断。
5. 最终成立状态通过接口回流到机器人端控制流程。

以倒水动作为例：

1. 云端经验库保存倒水动作过程模板。
2. 机器人端识别当前杯子、水壶、相机和液位观测方式。
3. 部署端或机器人端生成偶然绑定数据，包括杯对象、水壶对象、视觉传感器、液位阈值和流速阈值。
4. 机器人端根据绑定后的物理动作实例执行定位、倾斜、保持和回正阶段。
5. 事实验真服务分别基于外部视觉和执行器姿态或流速估计生成“杯中液位达到目标高度”的成立判断。
6. 机器人端根据最终成立状态进入回正阶段、继续保持流动或触发恢复执行。
7. 执行记录回传云端，用于更新该模板的推荐阈值、确认窗口、通道置信度或恢复入口参数。

写作重点：

1. 该实施例支撑云端经验库、SDK/API、机器人端真实执行、第三方验真服务和经验回传更新。
2. 不因模块位于不同设备或不同服务而改变闭环技术关系。
3. 该实施例为未来技能包、经验库授权和共创训练服务保留商业实施基础。

## 八、关键术语定义

### 8.1 物理动作

指机器人在真实物理环境中通过执行器对对象、环境或自身状态产生物理改变的动作。

### 8.2 过程模板

指用于描述物理动作不变过程结构的数据结构，至少包括阶段序列、连续状态变量类型、跃迁条件逻辑结构和参数槽。

过程模板不限于模板文件，也可以实现为状态图、行为树、任务图、技能图、工作流、规则集合、DSL 配置、数据库记录或图数据库节点集合。

### 8.3 偶然绑定数据

指在当前执行环境下为过程模板中的参数槽提供具体绑定值的数据，至少包括传感器映射、阈值绑定和对象绑定。

偶然绑定数据可以在运行期、部署期、场景加载期或任务编排期提供，也可以表现为映射表、配置文件、环境上下文、对象识别结果、传感器注册表、用户偏好参数或历史执行参数。

### 8.4 物理动作实例

指通过将偶然绑定数据填充至过程模板后生成的、适用于当前执行环境的可执行物理动作定义。

### 8.5 连续状态变量

指在物理动作执行过程中物理可观测且连续变化的物理量。

连续状态变量可以是传感器直接读数，也可以是由传感器数据、感知模型、融合模型或状态估计模块生成的连续估计值、连续特征值、置信度、距离、角度、力、速度、流量、液位或对象相对位姿。

### 8.6 目标因果事实

指从过程模板关联的因果事实声明中确定的、在物理动作实例执行过程中需要确认其成立状态的事实。

目标因果事实可以以谓词、键值状态、布尔标志、事件记录、任务状态、动作成功标志或世界状态图谱节点形式表示。

### 8.7 最终成立状态

指基于至少两个独立观测通道生成的成立判断确定的目标因果事实状态。

最终成立状态可以包括成立、不成立、未知、待重新观测、冲突需仲裁、置信不足或需恢复执行等状态。

### 8.8 独立观测通道

指用于生成目标因果事实成立判断的观测或判断路径。独立观测通道不限于物理上完全独立的硬件通道，也可以是基于不同传感来源、不同特征分支、不同模型分支、不同判断逻辑、不同时间窗口或不同数据处理链路形成的判断通道。

### 8.9 回流

指将目标因果事实的最终成立状态提供给物理动作实例的后续控制流程。回流可以表现为函数调用返回、消息发送、状态写入、事件触发、控制信号输出或任务上下文更新。
