# P016 实施主体与接口拆分攻防分析

> 对应版本：`权利要求书-v9-定稿.md`、`P016-说明书撰写提纲.md`  
> 目的：预判未来 SDK、API、云端经验库、机器人本体、第三方验真服务之间出现部署拆分时，竞对或合作方可能主张“没有单一主体完整实施”的规避路径，并提前在说明书中补强系统部署、接口调用和模块协同的技术边界。  
> 结论：P016 的说明书必须写清楚本方法既可以在单一机器人本体内实施，也可以在云端、边缘端、机器人端或第三方服务之间分布式实施；只要各模块通过接口协同完成“过程模板 + 偶然绑定 + 执行跃迁 + 目标因果事实最终成立状态回流”的闭环，即属于本方案的实施方式。

## 一、为什么这是遗漏点

前面的攻防主要解决三类问题：

1. 竞对是否能从术语上规避。
2. 无效宣告中是否会被现有技术组合攻击。
3. 审查中是否会被打成业务逻辑或抽象流程。

但还缺一个商业实施层面的关键风险：本案未来很可能不是卖整机，而是卖 SDK、API、经验库、训练平台或技能共创系统。此时技术链条天然会被拆成：

1. 云端经验库保存过程模板。
2. 机器人公司本体侧提供传感器和执行器。
3. 部署端根据场景生成偶然绑定。
4. 第三方模型或已有模块提供事实验真。
5. 机器人本体执行动作并回传状态。

如果说明书没有覆盖这种分布式实施方式，后续维权、许可谈判或商业合同中会被对方抓住接口边界做文章。

## 二、可能的规避路径

### 2.1 路线 A：模板在云端，机器人端只执行

竞对可能做法：

1. 云端平台保存过程模板和经验库。
2. 机器人端只接收已经实例化后的动作配置。
3. 对外声称机器人端没有“获取过程模板”和“偶然绑定”步骤。

风险：

如果权利要求和说明书只按单机流程写，对方可能主张机器人端只是普通执行器，云端只是普通配置服务，二者均未单独完整实施全部步骤。

防守要点：

1. 说明书要写明过程模板可以由本地存储、云端服务、边缘节点或外部经验库提供。
2. 获取过程模板包括直接读取、接口调用、远程下载、订阅推送和缓存加载。
3. 物理动作实例可以在云端生成后下发，也可以在机器人端生成，也可以由云端和机器人端协同生成。

### 2.2 路线 B：偶然绑定在客户现场完成

竞对可能做法：

1. 平台只提供模板。
2. 客户或集成商在现场配置传感器、阈值和对象。
3. 平台方声称自己没有实施偶然绑定。

风险：

如果后续商业模式是授权模板库或经验库，对方可能把关键绑定步骤放给客户完成，从而弱化平台方的侵权风险。

防守要点：

1. 说明书要写明偶然绑定数据可以由机器人端、部署工具、云端服务、现场标定模块、用户配置模块或历史执行记录生成。
2. 偶然绑定模块可以是独立部署工具，也可以是平台 API 的一部分。
3. 平台若提供绑定字段、绑定规则、绑定接口、绑定推荐或绑定校验，也应被描述为参与偶然绑定。

### 2.3 路线 C：事实验真由第三方模型服务完成

竞对可能做法：

1. 执行系统调用外部视觉模型、多模态模型或安全验真服务。
2. 事实判断结果由第三方返回。
3. 主系统声称自己没有执行“至少两个独立观测通道生成成立判断”。

风险：

这会把 P016 的闭环拆到外部服务上。

防守要点：

1. 说明书要写明事实验真模块可以是系统内部模块、外部接口、第三方服务或既有验真能力。
2. 独立观测通道可以在同一系统内实现，也可以由多个服务分别输出判断结果。
3. 关键不是验真模块归属，而是目标因果事实来自过程模板关联声明，且最终成立状态回流到阶段推进、恢复执行或后续动作触发。

### 2.4 路线 D：只输出 SDK，不直接执行机器人动作

竞对可能做法：

1. 提供 SDK 给机器人厂商集成。
2. SDK 包括模板结构、绑定接口和状态回流接口。
3. 实际机器人动作由厂商系统执行。
4. SDK 方声称自己不执行物理动作。

风险：

这正好对应我们未来可能的商业模式。如果说明书只写“机器人执行”，没有写 SDK/API 形态，后续授权谈判会弱。

防守要点：

1. 说明书中应写明系统可以实现为软件开发工具包、API 服务、中间件、机器人操作系统插件、云端服务或边缘端组件。
2. SDK 可以提供过程模板管理、偶然绑定生成、阶段跃迁判断、事实验真调用和最终成立状态回流接口。
3. 机器人本体执行器可以由外部系统控制，但 SDK 输出的阶段推进、恢复入口或后续动作触发信号仍参与物理执行闭环。

### 2.5 路线 E：机器人公司只用经验库，不暴露内部实现

竞对可能做法：

1. 使用经验库或模板库生成机器人技能。
2. 内部实现不公开。
3. 对外只说是“技能包”“场景包”“机器人训练数据”。

风险：

如果说明书没有覆盖“经验库/技能包/场景包”的实施方式，后续取证和解释都会更难。

防守要点：

1. 说明书应写明过程模板可以作为经验库条目、技能包、场景包、动作配置包或模板服务保存和分发。
2. 技能包中即使不使用“过程模板”名称，只要包含阶段序列、连续状态变量类型、跃迁条件逻辑结构和参数槽，也可对应本案过程模板。
3. 场景包中若包含传感器映射、阈值绑定和对象绑定，则可对应偶然绑定数据。

## 三、说明书补强方向

### 3.1 分布式部署形态

建议写入：

> 本发明的方法可以在单一机器人本体内实施，也可以由云端服务器、边缘计算节点、机器人本体控制器、传感器处理模块、第三方验真服务和用户部署终端中的至少两个协同实施。过程模板、偶然绑定数据、物理动作实例、目标因果事实判断和最终成立状态回流可以位于同一计算设备，也可以通过网络接口、消息队列、机器人操作系统通信机制或 API 调用在不同设备之间传递。

### 3.2 SDK/API 产品形态

建议写入：

> 在一种实施方式中，本发明可以实现为软件开发工具包、应用程序接口服务、中间件、机器人操作系统插件、云端经验库服务或边缘端执行组件。所述软件开发工具包或接口服务可提供过程模板获取接口、偶然绑定接口、物理动作实例生成接口、阶段跃迁判断接口、事实验真调用接口以及最终成立状态回流接口。

### 3.3 模块外置不影响闭环

建议写入：

> 事实验真模块、过程模板模块或偶然绑定模块可以是系统内部模块，也可以是外部服务或第三方模块。只要其输入输出参与本发明所述闭环，即由过程模板确定目标因果事实、由偶然绑定数据生成当前环境下的物理动作实例、由连续物理观测驱动阶段跃迁，并将目标因果事实最终成立状态回流到物理动作控制流程中，即不影响其作为本发明实施方式的一部分。

### 3.4 获取、生成、回流的广义接口

建议写入：

> 本说明书中的“获取”包括本地读取、远程调用、接口接收、订阅推送、缓存加载、数据库查询和消息队列接收；“生成”包括本地计算生成、云端生成后下发、边缘端协同生成和由多个模块组合生成；“回流”包括函数调用返回、消息发送、状态写入、事件触发、控制信号输出和任务上下文更新。

## 四、对权利要求的影响

当前 V9 权利要求 10 已经写了系统模块，但还偏“单系统模块化”。是否修改权利要求，要看后续代理人策略。

当前不建议立刻扩张独权，因为用户已经明确希望控制在 10 条以内，且避免非正常审查风险。更稳妥的方式是：

1. 在说明书中充分写分布式部署和 SDK/API 实施方式。
2. 在系统实施例中写云端、边缘端、机器人端三种部署。
3. 在存储介质或电子设备权利要求是否加入，交给最终代理撰写时权衡。
4. 后续如果需要分案，可围绕“机器人经验库 SDK/API 接口闭环”单独展开。

## 五、无效和规避中的答辩口径

### 5.1 如果对方说各模块分属不同主体

答辩方向：

本案说明书已经记载各模块可以位于不同计算设备或服务中，通过接口协同完成闭环。技术贡献不在模块物理部署位置，而在过程模板、偶然绑定、阶段跃迁、目标因果事实最终成立状态回流之间的数据依赖和控制依赖。

### 5.2 如果对方说云端只是提供数据

答辩方向：

若云端提供的数据包含过程模板，且该过程模板具有阶段序列、连续状态变量类型、跃迁条件逻辑结构和参数槽，并被用于生成当前环境下的物理动作实例，则其并非普通数据，而是参与物理动作执行闭环的技术数据结构。

### 5.3 如果对方说 SDK 只是开发工具

答辩方向：

若 SDK 提供过程模板获取、偶然绑定、阶段跃迁判断、事实验真调用或最终成立状态回流接口，并通过这些接口影响机器人真实物理动作的阶段推进、恢复执行或后续动作触发，则其不是普通开发工具，而是本发明闭环实施的技术组件。

### 5.4 如果对方说第三方验真不属于其系统

答辩方向：

本案不要求事实验真算法必须由同一设备实现。只要系统将过程模板关联的目标因果事实发送至验真模块或服务，并利用返回的最终成立状态控制阶段推进、恢复执行或后续动作触发，即形成本案所述事实状态回流。

## 六、商业许可视角

这个点对后续商业化尤其重要，因为我们可能不做机器人本体，而是提供：

1. 经验闭环训练 SDK。
2. 过程模板库。
3. 偶然绑定工具链。
4. 事实状态回流接口。
5. 场景技能包共创平台。
6. 云端经验库 API。

因此，说明书中必须预埋“SDK/API/云端服务/边缘组件/机器人端插件”的实施形态。这样后续无论我们卖 SDK、卖 API、卖技能包，还是做机器人公司共创训练服务，都能和 P016 的技术特征形成对应。

## 七、建议加入说明书的实施例

建议在具体实施方式中新增一个“云端经验库与机器人端协同实施例”：

1. 云端经验库保存倒水动作过程模板。
2. 机器人端识别当前杯子、水壶、相机和液位观测方式。
3. 部署端或机器人端生成偶然绑定数据，包括杯对象、水壶对象、视觉传感器、液位阈值和流速阈值。
4. 机器人端根据绑定后的物理动作实例执行定位、倾斜、保持和回正阶段。
5. 事实验真服务分别基于外部视觉和执行器姿态/流速估计生成“杯中液位达到目标高度”的成立判断。
6. 机器人端根据最终成立状态进入回正阶段或继续保持流动。
7. 执行记录回传云端，用于更新该模板的推荐阈值、确认窗口或恢复入口参数。

该实施例能同时支撑：

1. 云端经验库。
2. SDK/API 商业形态。
3. 机器人端真实执行。
4. 第三方验真服务。
5. 经验回传更新。

## 八、最终判断

“实施主体与接口拆分”是 P016 在商业落地中最值得补的遗漏点。它不像新颖性、创造性那样直接出现在授权审查第一轮，但会影响后续：

1. SDK/API 商业价值。
2. 机器人公司合作谈判。
3. 技能包和经验库授权。
4. 分布式系统取证。
5. 竞对云端化、接口化规避。

当前最优策略不是扩张 V9 权利要求，而是在说明书中把分布式部署、SDK/API、外部服务、模块接口和云端经验库实施方式写足，给未来许可和分案留下空间。
