# P016 业务逻辑剥离攻防分析

> 对应版本：`权利要求书-v9-定稿.md`、`P016-说明书撰写提纲.md`  
> 目的：预判审查、无效或竞对规避中可能出现的“这只是业务逻辑、流程规则、配置管理或抽象编排”攻击，并提前形成技术性防线。  
> 结论：P016 不应被写成“把业务流程抽象成模板”的方案，而应被写成“将真实物理动作的不变过程结构与当前执行环境下的物理绑定数据解耦，并通过物理观测驱动阶段跃迁和事实状态回流”的技术方案。

## 一、为什么需要单独分析

前两份攻防已经覆盖了竞对规避和无效组合攻击，但还缺少一个更偏审查口径的风险：对方可能不从现有机器人技术文献攻击，而是把本案降格为“业务逻辑剥离”或“流程配置抽象”。

这种攻击在软件和 AI 相关专利中很常见，其基本说法是：

1. 过程模板只是业务流程或规则配置。
2. 偶然绑定只是参数配置或运行时变量赋值。
3. 因果事实只是业务状态或任务状态。
4. 最终成立状态回流只是软件流程跳转。
5. 因此整体不是技术方案，而是业务规则、抽象算法或管理方法。

如果说明书没有把“物理动作执行”和“真实传感闭环控制”写足，这个攻击会比较危险。

## 二、本案不能被写成什么

### 2.1 不能写成普通流程引擎

危险写法：

1. 定义流程模板。
2. 给流程模板绑定参数。
3. 根据条件流转节点。
4. 根据执行结果进入下一节点。

风险：

这套表述可以套到工单系统、审批系统、游戏任务、业务规则引擎，不足以体现机器人真实物理动作中的技术性。

### 2.2 不能写成普通配置中心

危险写法：

1. 模板里留参数。
2. 当前环境提供参数。
3. 运行时替换参数。

风险：

容易被理解为“将固定配置改为运行时配置”，属于一般软件工程手段。

### 2.3 不能写成普通任务状态管理

危险写法：

1. 执行任务。
2. 判断成功或失败。
3. 根据成功或失败进入下一任务。

风险：

容易被理解为通用任务调度逻辑，而不是机器人物理执行中的可信事实回流。

## 三、本案的技术性支点

P016 要守住的技术性支点不是“抽象模板”，而是下面五个彼此耦合的技术事实：

1. 处理对象是机器人真实物理动作，而不是业务流程。
2. 过程模板描述的是物理动作的不变过程结构，包括阶段序列、连续状态变量类型、跃迁条件逻辑结构和参数槽。
3. 偶然绑定数据绑定的是当前执行环境中的具体传感器、具体阈值和具体被操作对象。
4. 阶段跃迁由连续物理观测值与绑定阈值的比较触发，而不是由人工业务状态或抽象事件触发。
5. 目标因果事实的最终成立状态由至少两个独立观测通道判断，并回流到阶段推进、恢复执行或后续物理动作触发。

只要这五点写实，本案就不是“业务逻辑剥离”，而是“物理动作经验复用与可信执行控制”的技术方案。

## 四、可能攻击路径与防守

### 4.1 攻击 A：过程模板只是业务流程模板

对方可能说：

本案的过程模板不过是把业务流程中的节点、条件和参数抽象出来，与通用工作流、状态机、规则引擎没有本质区别。

防守要点：

1. 过程模板不是任意流程模板，而是物理动作的不变过程结构。
2. 阶段不是审批节点或任务节点，而是机器人执行器与物理对象发生相互作用的执行阶段。
3. 连续状态变量类型必须对应物理可观测且连续变化的物理量。
4. 跃迁条件逻辑结构最终要接收真实物理观测值，而不是业务字段。

说明书补强表述：

> 本说明书中的过程模板用于描述机器人真实物理动作在执行过程中的不变过程结构，所述阶段序列对应执行器、被操作对象或环境之间的物理交互阶段，所述连续状态变量类型对应在物理动作执行过程中可由传感器、状态估计模块或感知模型获得的连续物理量。因此，过程模板不同于用于审批、工单、交易或其他非物理业务流程的通用流程模板。

### 4.2 攻击 B：偶然绑定只是参数配置

对方可能说：

偶然绑定数据只是运行时参数配置，传感器映射、阈值绑定和对象绑定都是普通配置项。

防守要点：

1. 偶然绑定不是单纯软件参数替换，而是将抽象过程结构落到当前物理执行环境。
2. 传感器映射决定观测数据来源。
3. 阈值绑定决定物理阶段跃迁边界。
4. 对象绑定决定机器人当前作用的真实对象。
5. 三者共同生成当前环境下可执行的物理动作实例。

说明书补强表述：

> 偶然绑定数据不是用于界面显示或业务规则选择的普通配置数据，而是用于将过程模板中的抽象传感器引用、阈值标识和对象引用绑定为当前执行环境下的具体传感器、具体阈值和具体被操作对象。绑定结果直接参与连续物理观测的获取、阶段跃迁判断以及物理动作实例的执行。

### 4.3 攻击 C：阶段跃迁只是软件状态跳转

对方可能说：

本案只是根据条件把一个状态切到另一个状态，属于普通状态机。

防守要点：

1. 阶段跃迁发生在物理动作实例执行过程中。
2. 跃迁依据是连续物理观测值与当前环境绑定阈值的比较结果。
3. 该跃迁会影响机器人执行器的后续控制、恢复入口或下一物理动作。
4. 与普通状态机相比，本案的跃迁条件来自真实物理观测，并且其阈值由偶然绑定提供。

说明书补强表述：

> 阶段跃迁不是仅在软件任务状态之间进行标记切换，而是在物理动作实例执行过程中，基于当前执行阶段关联的连续状态变量观测值与当前环境绑定阈值之间的比较结果，对机器人后续执行阶段、恢复入口或后续物理动作触发进行控制。

### 4.4 攻击 D：因果事实只是业务状态

对方可能说：

“对象已抓住”“液位已达到”“插头已到位”只是任务状态或业务标签，不构成技术特征。

防守要点：

1. 目标因果事实表示真实物理世界中的状态改变。
2. 其成立与否不是由程序假定，而是由观测通道生成成立判断。
3. 最终成立状态参与控制，而不是仅用于日志、报表或业务记录。
4. 因果事实的技术意义在于避免机器人错误地假定世界状态已经改变。

说明书补强表述：

> 目标因果事实用于表示物理动作执行前、执行中或执行后与真实物理世界相关的状态，例如对象是否被夹持、液位是否达到目标高度或装配件是否到位。该事实状态不是由流程节点完成后直接假定成立，而是通过至少两个独立观测通道生成成立判断，并将最终成立状态回流到物理动作实例的后续控制。

### 4.5 攻击 E：最终成立状态回流只是业务流程分支

对方可能说：

成功则下一步，失败则重试，这是任何业务流程都有的逻辑。

防守要点：

1. 回流对象是目标因果事实的最终成立状态，不是普通成功/失败码。
2. 回流用途包括阶段推进、恢复执行、后续物理动作触发，均与物理执行控制相关。
3. 恢复执行依赖连续状态变量和事实状态匹配入口阶段，这不是普通业务重试。
4. 后续动作触发依赖真实物理前提是否已经成立，而不是流程节点是否完成。

说明书补强表述：

> 最终成立状态的回流并非仅用于业务流程分支选择，而是用于控制机器人物理动作实例的阶段推进、恢复执行入口确定或后续物理动作触发。例如，在抓取动作后，系统不是因为抓取节点执行完毕而假定对象已被夹持，而是基于目标因果事实“对象被夹持”的最终成立状态决定是否进入移动动作或恢复至对准、闭合阶段。

## 五、审查意见答复预案

### 5.1 如果审查员认为本案属于业务规则或抽象流程

可答复：

本案解决的不是业务流程编排问题，而是机器人真实物理动作经验难以跨环境复用且执行结果难以可信回流的问题。权利要求中的过程模板、偶然绑定数据、连续状态变量、传感器映射、阈值绑定、对象绑定、独立观测通道和最终成立状态回流，均直接服务于机器人在真实物理环境中的动作执行控制。

### 5.2 如果审查员认为过程模板是普通状态机

可答复：

普通状态机通常固定描述特定任务中的状态和跳转条件。本案的过程模板不包含具体阈值数值、具体对象标识和具体传感器标识，而是通过偶然绑定数据在当前执行环境下生成物理动作实例。并且，阶段跃迁依赖绑定后的传感器观测和绑定阈值，目标因果事实最终成立状态还会回流到阶段推进、恢复执行或后续物理动作触发。

### 5.3 如果审查员认为偶然绑定是普通参数化

可答复：

本案的偶然绑定不是对动作参数的简单赋值，而是同时完成传感器映射、阈值绑定和对象绑定，使同一过程模板能够在不同物理对象、不同传感器配置和不同阈值条件下生成可执行物理动作实例。绑定结果直接影响观测数据来源、阶段跃迁边界以及执行对象。

### 5.4 如果审查员认为因果事实是业务状态

可答复：

本案的目标因果事实表示机器人动作导致或依赖的真实物理世界状态，且该状态通过独立观测通道判断后才确定最终成立状态。该最终成立状态不是用于报表、统计或业务记录，而是用于控制机器人后续物理执行。

## 六、无效宣告中的防守重点

无效攻击中，对方可能把本案拆成：

1. 状态机。
2. 参数配置。
3. 多传感器验证。
4. 任务成功/失败分支。

我方应强调：

1. 单独看每个点可能已有基础，但本案不是这些点的并列堆叠。
2. 本案的关键是数据流和控制流闭环：过程模板提供不变结构，偶然绑定提供当前物理环境的具体绑定，绑定后的观测与阈值驱动阶段跃迁，目标因果事实最终成立状态再回流到物理动作控制。
3. 如果缺少物理观测驱动和事实状态回流，本案会退化为普通流程配置；如果缺少过程模板和偶然绑定，本案会退化为特定任务控制；正是二者耦合形成了技术效果。

## 七、竞对规避中的防守重点

竞对可能通过以下方式规避：

1. 把模板叫作 skill graph、behavior tree、policy graph 或 workflow。
2. 把偶然绑定叫作 runtime context、environment profile、scene config 或 adapter。
3. 把因果事实叫作 task state、world fact、effect state 或 success condition。
4. 把最终成立状态回流包装成 planner feedback、execution feedback 或 recovery signal。

防守时不要执着术语，而要抓实质：

1. 是否存在物理动作不变过程结构。
2. 是否存在当前环境下的传感器、阈值、对象绑定。
3. 是否基于连续物理观测触发阶段跃迁。
4. 是否对真实物理世界中的目标事实生成最终成立状态。
5. 是否将该最终成立状态用于阶段推进、恢复执行或后续物理动作触发。

## 八、对权利要求的影响

当前 V9 独权已经具备抵御“业务逻辑剥离”的主要技术锚点：

1. “物理动作”限定了对象。
2. “连续状态变量类型”限定了物理可观测量。
3. “传感器映射、阈值绑定、对象绑定”限定了物理执行环境。
4. “基于观测值与具体阈值比较触发阶段跃迁”限定了物理执行控制。
5. “至少两个独立观测通道”限定了事实状态判断来源。
6. “用于阶段推进、恢复执行或后续物理动作触发”限定了技术回流用途。

因此不建议为了防“业务逻辑剥离”再扩张权利要求。更好的做法是在说明书中增加技术性定义、反业务流程表述和具体物理实施例。

## 九、说明书必须补强的位置

建议在说明书中补强以下位置：

1. 技术领域：明确本案属于机器人真实物理动作执行与经验复用。
2. 背景技术：说明普通流程引擎、状态机或业务规则不能解决真实物理动作跨环境复用和可信回流。
3. 发明内容：强调传感器映射、阈值绑定、对象绑定与物理观测驱动的阶段跃迁。
4. 术语定义：把过程模板、偶然绑定、目标因果事实都定义到物理执行语境中。
5. 具体实施例：每个实施例都写出物理变量、传感器、阈值、对象、事实状态和回流控制。
6. 有益效果：避免写成“提高流程复用效率”，改写为“降低跨环境物理动作复用成本、提高物理执行阶段推进可靠性、降低错误世界状态累积风险”。

## 十、建议写入说明书的总防线段落

可在发明内容或具体实施方式开头加入以下段落：

> 本发明并非针对非物理业务流程、审批流程或通用软件任务编排的规则配置方法。本发明所述过程模板、偶然绑定数据、阶段跃迁和目标因果事实最终成立状态均用于机器人真实物理动作的执行控制。其中，过程模板描述物理动作在不同执行环境中相对稳定的过程结构；偶然绑定数据将该过程结构中的抽象传感器引用、阈值标识和对象引用绑定为当前执行环境下的具体传感器、具体阈值和具体被操作对象；阶段跃迁基于连续物理观测值与绑定阈值的比较结果触发；目标因果事实的最终成立状态通过观测通道确定后回流到阶段推进、恢复执行或后续物理动作触发。因此，本发明解决的是机器人真实物理动作经验复用和可信执行控制中的技术问题。

## 十一、最终判断

“业务逻辑剥离”确实是 P016 需要补的一条攻防线，但它不要求改变 V9 的独权结构。当前最优策略是：

1. 权利要求保持克制，不引入过多业务无关解释。
2. 说明书把技术语境写牢，避免被读成通用流程引擎。
3. 实施例必须有真实物理变量、传感器、阈值、对象和回流控制。
4. 答复口径围绕“技术问题、技术手段、技术效果”展开。

这样处理后，P016 的定位会更清楚：不是流程抽象专利，而是真实物理动作经验复用与可信执行闭环专利。
