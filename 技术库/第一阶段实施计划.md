# 第一阶段实施计划

## 一、阶段目标

第一阶段目标不是制造机器人，也不是训练大模型，而是在真实数字环境中跑通“自动化执行体运行时治理 + 经验回写”的最小闭环。

第一阶段主链路：

> 任务计划 -> 计划预验真 -> 来源链校验 -> 代理权预算 -> 裁决令牌 -> 动作执行 -> 后果记录 -> 审计摘要

阶段完成后，应当形成：

1. 一套可运行的核心对象和 API。
2. 一个真实数字环境 Demo。
3. 一组可复核的执行日志、经验记录和审计摘要。
4. 一份可用于对外展示的演示脚本。
5. 一批可作为“已经开始实施”的项目证据材料。

## 二、MVP 场景选择

推荐第一阶段采用“文件/空间对象整理 Agent”场景。

原因：

1. 不依赖机器人硬件。
2. 能作用于真实数字环境。
3. 容易构造低风险、高风险、候选态、阻断态动作。
4. 能覆盖计划预验真、来源链、代理权预算、裁决令牌、后果记录和审计摘要。
5. 后续可自然迁移到三维空间对象、智能家居、机器人和物理 AI。

## 三、Demo 任务示例

用户输入：

> 整理项目资料，把说明文档归类，把临时文件移到归档区，把涉及权限或敏感内容的文件只生成候选操作，不要直接修改。

系统生成执行计划：

1. 扫描目标目录或空间对象列表。
2. 读取文件名、元数据或对象标签。
3. 识别说明文档、草稿、敏感文件和可归档文件。
4. 对低风险文件执行移动或标记。
5. 对敏感文件生成候选态操作。
6. 对不明来源文件阻断自动处理。
7. 输出整理结果和审计摘要。

## 四、2 周目标

### 4.1 核心数据结构

完成以下对象的最小字段定义和样例：

1. `ExecutionPlan`
2. `StepSummary`
3. `PlanPrecheckResult`
4. `ActionDeclaration`
5. `ProvenanceChain`
6. `AgencyBudget`
7. `DecisionToken`
8. `ExperienceRecord`
9. `AuditSummary`

### 4.2 静态流程跑通

用固定 JSON 样例跑通以下流程：

1. 输入执行计划。
2. 输出计划预验真结果。
3. 为动作生成来源链。
4. 根据预算输出裁决。
5. 生成审计摘要。

### 4.3 证据材料

形成：

1. 样例输入 JSON。
2. 样例输出 JSON。
3. 一次完整流程的审计摘要。
4. 流程截图或控制台输出。

## 五、4 周目标

### 5.1 接入真实数字环境

选择一个真实目录或空间对象集合，允许系统读取对象列表和元数据。

支持动作：

1. 读取对象。
2. 标记对象。
3. 移动低风险对象。
4. 对高风险对象生成候选态操作。
5. 对来源不明对象阻断自动操作。

### 5.2 实现基础治理规则

规则包括：

1. 敏感对象不得直接正式修改。
2. 来源链缺失不得执行高风险动作。
3. 预算不足时冻结后续动作。
4. 不可逆动作需要人工确认。
5. 执行计划若后续步骤越界，触发重新规划或限制执行。

### 5.3 经验回写

每个已执行动作都生成 `ExperienceRecord`。

至少记录：

1. 执行动作。
2. 动作前情境。
3. 实际后果。
4. 裁决结果。
5. 是否成功。

### 5.4 审计输出

每个任务生成 `AuditSummary`，能够串联：

1. 任务计划。
2. 计划预验真结果。
3. 来源链状态。
4. 预算状态。
5. 裁决令牌。
6. 动作结果。
7. 经验记录引用。

## 六、8 周目标

### 6.1 Demo 可演示化

形成一个可演示流程：

1. 用户输入任务。
2. 系统展示计划。
3. 系统展示预验真结果。
4. 系统展示哪些动作放行、限制、候选、阻断。
5. 执行低风险动作。
6. 生成经验记录。
7. 生成审计摘要。

### 6.2 加入偏好与补救

加入最小偏好逻辑：

1. 用户可对执行结果反馈“接受、纠正、不要这样做”。
2. 系统将反馈写入 `PreferenceRecord` 或经验记录。
3. 后续相似任务中调用该偏好。

加入最小补救逻辑：

1. 当动作失败时记录失败情境。
2. 人工指定补救动作。
3. 写入 `RecoveryExperienceRecord`。

### 6.3 形成对外材料

形成：

1. Demo 演示脚本。
2. 技术架构图。
3. API 文档初稿。
4. 典型审计摘要样例。
5. 典型经验记录样例。
6. 商业场景说明。

## 七、第一阶段不做什么

第一阶段明确不做：

1. 不训练大模型。
2. 不制造机器人。
3. 不做复杂三维渲染。
4. 不做完整多主体调度。
5. 不做完整企业权限系统。
6. 不做复杂 UI。

第一阶段只做一件事：

> 证明自动化执行体的真实动作可以经过计划预验真、来源链、代理权预算和裁决令牌控制，并将后果回写为经验。

## 八、验收清单

第一阶段完成时，至少应满足：

1. 能读取一个真实对象集合。
2. 能生成或导入一个多步执行计划。
3. 能识别至少一个越界步骤。
4. 能校验至少一个来源链缺失场景。
5. 能模拟或执行至少一个预算扣减场景。
6. 能生成至少三种裁决：放行、候选态、阻断。
7. 能执行至少一个真实数字动作。
8. 能记录至少一条经验记录。
9. 能生成至少一条审计摘要。
10. 能用演示脚本复现完整流程。
