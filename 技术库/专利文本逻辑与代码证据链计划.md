# 专利文本逻辑与代码证据链计划

## 一、结论

本项目必须建立“专利文本逻辑 -> 代码逻辑 -> 输出证据 -> Git 提交”的强对应关系。

但该证据链不适合完整公开。

建议采用双层证据体系：

1. 内部证据链：完整保留专利文本逻辑、权利要求特征、代码模块、输出文件、验证命令和研发记录。
2. 公开证据链：只展示功能模块、schema、demo、API 和普通技术说明，不展示权利要求级映射、审查策略和内部研发记录。

## 二、为什么不建议完整公开证据链

完整证据链包含：

1. 专利权利要求拆解。
2. 技术特征与代码模块的精确映射。
3. 实施证据和验证路径。
4. 专利防御、审查答复和非正常申诉线索。
5. 未来分案和商业许可边界。

这些材料一旦公开，可能产生以下风险：

1. 竞争者可以直接按证据链理解核心保护点。
2. 竞争者可以围绕权利要求设计规避方案。
3. 潜在侵权方可以提前准备抗辩路径。
4. 公开仓库会暴露项目专利布局和商业策略。
5. 审查或争议中，本方内部推理材料可能被外部片面解读。

因此，完整证据链应留在内部仓库，不进入 GitHub 开源仓库。

## 三、公开仓库应该展示什么

GitHub 开源仓库可以展示：

1. 项目定位。
2. 基础 schema。
3. 最小 demo。
4. API 或脚本用法。
5. 示例输入输出。
6. 技术模块的普通工程说明。
7. 贡献指南。

公开仓库不应展示：

1. 权利要求逐项拆解。
2. 专利说明书原文。
3. FTO、检索结论、近似文献分析。
4. 非正常审查应对材料。
5. 内部商业策略。
6. 完整代码-专利特征映射表。
7. 研发记录目录。

## 四、内部四联表标准

每一个小阶段都应形成以下四联表。

| 项 | 内容 |
|---|---|
| 专利文本逻辑 | 对应母案、权利要求/说明书特征、技术目的 |
| 代码逻辑 | 文件、函数、schema、输入输出对象 |
| 验证输出 | 运行命令、输出 JSON、摘要结果、审计报告 |
| 提交记录 | 中文摘要、提交正文、研发记录日期、远端推送 |

## 五、小阶段提交规范

每个小阶段应尽量只推进一个可验证点。

推荐节奏：

1. 定义一个技术特征。
2. 实现一个 schema、函数、脚本或接口。
3. 增加一个样例输入。
4. 生成一个输出证据。
5. 更新内部证据链文档。
6. 更新当日研发记录。
7. 使用中文提交摘要。
8. 推送到内部 Gitee。
9. 如属于开源白名单，再通过临时发布仓库推送 GitHub。

## 六、提交摘要建议格式

内部仓库提交摘要建议使用：

```text
实现P011经验内化记录与输出证据
补充P014执行恢复补救记录链路
建立P013任务语义到技能步骤映射
实现P015偏好反馈记录与技能约束
增加P012概念模式与技能包生成证据
```

提交正文必须包含：

```text
技术对应：
- 对应技术层：
- 对应专利/技术链特征：
- 对应文档或代码模块：

证据链：
- 输入证据：
- 输出证据：
- 验证方式：
```

## 七、公开提交摘要建议

GitHub 公开仓库提交摘要应避免暴露权利要求级映射。

公开提交可以写：

```text
Add experience record schema
Add recovery record demo
Add skill co-creation pipeline
Add preference feedback examples
Add skill package output
```

公开提交不建议写：

```text
实现P011权利要求1的情境动作后果三元组
补充P014说明书第X段执行恢复证据
按P015权利要求拆解偏好约束逻辑
```

## 八、内部证据文档结构

建议后续新增：

```text
技术库/专利代码映射/
  P010_空间语义自动生成_代码证据.md
  P011_物理经验内化_代码证据.md
  P012_概念归纳迁移_代码证据.md
  P013_任务语义翻译_代码证据.md
  P014_执行恢复_代码证据.md
  P015_人类偏好_代码证据.md
```

每份文档包含：

1. 专利文本逻辑摘要。
2. 关键技术特征。
3. 对应代码文件。
4. 对应 schema。
5. 样例输入。
6. 输出证据。
7. 验证命令。
8. 关联提交。
9. 当日研发记录。

## 九、当前优先级

第一阶段优先补 P010-P015。

建议顺序：

1. P011 经验内化：因为当前已有 `experience_records.json`。
2. P014 执行恢复：因为当前已有 `recovery_records.json`。
3. P015 人类偏好：因为当前已有 `preference_records.json`。
4. P013 任务语义翻译：因为当前已有 `task_plan.json`。
5. P012 概念归纳迁移：因为当前已有 `concept_patterns.json` 和 `skill_package.json`。
6. P010 空间语义：因为当前已有 `scenario.json` 和 `space_object.schema.json`。

## 十、研发记录要求

每次小阶段完成后，当日研发记录必须写明：

1. 对应 P 编号。
2. 对应技术特征。
3. 代码文件。
4. 输入文件。
5. 输出文件。
6. 验证命令。
7. 提交号。
8. 是否已公开发布。
9. 若未公开，说明原因。

## 十一、结论

本项目的专利证据链要强，但不应裸露。

内部要做到：

> 专利文本逻辑、代码逻辑、输出证据、Git 提交一一对应，经得起查。

公开要做到：

> 代码能跑、接口清晰、范式能被理解，但不暴露权利要求级攻防材料。
