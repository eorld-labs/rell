# P000-P020 与 P1-P6 边界总复核

## 一、文档用途

本文件是技术库内部的总边界底图，用于：

1. 新方案立项前判断是否已有相邻方案覆盖；
2. 分案撰写前判断输入起点、核心处理和输出责任；
3. 审查答复前准备“不重复、不越界”的解释路径；
4. 识别共用数据结构与真正功能重叠的区别；
5. 防止已提交文本被后续新方案无意扩张或反向污染。

本文件不修改任何已提交申请文本，也不替代各 P 方案的权利要求书。已提交文本以提交版本和提交记录为基线；本文件只作为后续撰写、检索和答复的内部导航。

## 二、总架构分层

### 2.1 P000-P020：能力与学习主链

```text
空间数据与空间交互
  -> P000/P001/P002/P003/P004
空间对象、制度、主体进入与虚实交互
  -> P005/P006/P007/P008/P009
空间语义与端侧认知
  -> P010
交互经验、概念、任务与恢复
  -> P011/P012/P013/P014/P015
物理执行、跨空间跨本体迁移、运行时仲裁
  -> P016/P017/P018
端侧概念内化与具身教学
  -> P019/P020
```

### 2.2 P1-P6：治理与预适配横切链

```text
人类真实意图与代理权边界       P1
物理执行链路高风险验真与接管    P2
代理权预算与预算传播            P3
动作来源链完整性与执行准入      P4
计划执行前步骤边界预验真        P5
受保护主体策略与行为适配        P6
```

P1-P6 不应被理解为 P000-P020 的替代执行层，而是在动作、计划、任务、经验或教学候选进入高风险边界时提供并行的准入、约束、审计和保护能力。

## 五、最重要的跨案接口

### 5.1 空间接口

```text
P000/P003/P004/P008/P010
  -> 空间模型、坐标、区域、关系、可进入性和语义先验
P013/P017/P018/P019/P020
  -> 使用这些空间数据形成计划、迁移绑定、快照、概念落地或教学复做
```

空间数据被多个方案消费，不构成功能重叠。P010 生成语义认知模型，P017 做经验适配，P018 做运行时仲裁，三者输出责任不同。

### 5.2 经验接口

```text
P011：单条/相似经验的后果预测
P012：多条经验归纳公共概念
P013：历史任务计划类比
P014：偏离后的补救经验
P016：物理过程模板和验真事实
P017：既有经验跨空间/跨本体适配
P019：端侧概念候选、经验检索和回退
P020：原始教学到首次可信经验
```

“经验”是公共数据对象，但每案对经验的形成方式、用途和验证责任不同。

### 5.3 运行时状态接口

P016、P017、P018 都可能使用任务期状态或事实，但应保持以下边界：

- P016：用连续物理观测推动阶段跃迁并确认事实；
- P017：用快照判断迁移经验绑定和可执行性；
- P018：在交互输入改变控制前读取快照并做仲裁。

同一 `RuntimeWorldState` 数据结构可以共享，但不能因为字段相同就把三案合并。

### 5.4 治理接口

```text
P020/P019/P013/P015 生成候选、偏好、教学和概念输入
  -> P1 验证人类意图和代理权
  -> P3 检查代理权预算
  -> P4 验证来源链完整性
  -> P5 预验真计划边界
  -> P6 适配受保护主体策略
  -> P2 对高风险物理控制链路验真和接管
  -> P018 进行当前任务运行时仲裁
  -> P016 完成物理事实验真
```

P1-P6 可以并行参与，但不应互相吞并。代理权、预算、来源、计划边界、保护策略和物理接管是不同的裁决对象。

## 七、立项前检查清单

每个新 P 方案立项前，必须回答：

1. 输入是原始数据、候选、已验证经验、任务计划、运行时状态还是治理事件？
2. 本案是否必须从多个经验归纳概念？如果是，优先检查 P012。
3. 本案是否以既有不变量经验进入新空间/新本体为起点？如果是，优先检查 P017。
4. 本案是否处理任务执行期间的新输入、暂停、切换或快照释放？如果是，优先检查 P018。
5. 本案是否处理物理连续过程、阶段跃迁或双通道事实验真？如果是，优先检查 P016。
6. 本案是否处理原始教学材料到首次可信经验？如果是，优先检查 P020。
7. 本案是否属于高风险意图、预算、来源、计划边界、保护策略或物理接管？如果是，优先检查 P1-P6。
8. 输出是概念、经验、计划、绑定候选、仲裁令牌、保护策略还是已成立事实？
9. 是否把“候选”误写成“事实”或“执行命令”？
10. 是否把共享数据结构误写成新的独立发明步骤？
11. 是否保存了其他方案明确排除的绝对坐标、单一本体轨迹、会话身份或原始日志？
12. 是否有独立的验证责任和失败回退入口？

## 八、提交后文本处理规则

1. 已提交文本保留原始版本、提交日期和提交 hash，不因内部复核直接覆盖；
2. 发现边界压力时，先做内部解释备忘录和备用限缩文本；
3. 是否主动修改、如何修改以及是否存在新增事项风险，由专利代理师结合案件状态判断；
4. 新增技术构思优先进入后续分案种子，不回写已提交母案；
5. 工程代码映射表可以更新，但不能把工程新增能力反向宣称为已提交权利要求内容；
6. 审查答复时优先使用“输入起点、处理对象、输出责任、验证责任”的功能边界，不只争论名词不同。

## 九、当前总判断

P000-P020 与 P1-P6 可以形成“空间基础设施 + 经验学习链 + 运行时治理链”的组合专利族：

```text
空间和对象基础设施
  + 经验、概念、计划、恢复和偏好
  + 物理执行、迁移、仲裁、端侧内化和教学
  + 意图、物理链路、预算、来源、计划和保护治理
```

当前最需要防守的不是每个方案完全隔离，而是确保每个方案的独立主链只从自己的输入起点开始，以自己的处理结果结束；组合实施可以共享数据、接口和验证回流，但不能因此把相邻方案的核心处理写成自己的必要步骤。
