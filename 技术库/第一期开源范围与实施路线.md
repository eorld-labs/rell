# 第一期开源范围与实施路线

## 一、结论

第一期开源不展开全部技术体系，只聚焦 P010-P015 六个经验世界模型闭环。

第一期开源主题为：

> 真实世界经验闭环训练框架。

本期不以空间治理 P005-P007 为开源主线，也不以完整安全治理体系为开源主线。

边界如下：

1. P010-P015：作为第一期开源核心。
2. P005-P007：暂时作为后期多主体空间治理与商业资产保留。
3. 运行时安全治理：在方案中内化，并通过 Sayfos SDK 形成适配关系。

## 二、为什么本期只做 P010-P015

本项目当前最需要占领的是“机器人真实部署后的经验闭环训练范式”。

如果第一期同时展开空间治理、多主体治理、安全治理、经验训练和商业平台，会导致主线过宽，开发者和机器人公司难以快速理解项目到底解决什么问题。

因此，第一期应当只回答一个问题：

> 机器人如何在真实任务中沉淀可复用经验、补救能力、偏好记录和技能包。

P010-P015 正好构成这条主线：

1. P010 解决空间语义输入。
2. P011 解决真实动作经验内化。
3. P012 解决经验归纳和跨场景迁移。
4. P013 解决任务语义到可执行步骤的翻译。
5. P014 解决执行失败后的补救经验。
6. P015 解决人类偏好和长期适配。

## 三、本期开源能力清单

| 模块 | 对应专利 | 开源内容 | 目标 |
|---|---|---|---|
| 空间语义对象 | P010 | 最小空间对象、区域、关系、约束 schema | 让真实场景可以被结构化输入 |
| 经验记录 | P011 | `{情境, 动作, 后果}` 经验记录 schema 和样例 | 让真实执行结果可沉淀 |
| 概念归纳 | P012 | 从多条经验中提取概念标签和技能模式的最小逻辑 | 让经验可复用、可迁移 |
| 任务翻译 | P013 | 人类任务目标到任务步骤的样例 pipeline | 让真实任务可拆解 |
| 执行恢复 | P014 | 失败、偏离、人工接管到补救记录的 schema | 让失败变成训练资产 |
| 人类偏好 | P015 | 接受、拒绝、纠正、习惯和禁忌的偏好记录 schema | 让技能适应人类 |
| 技能包 | P012/P013/P014 | skill package 基础格式 | 让经验可封装和分发 |
| 评测脚本 | P010-P015 | 检查经验记录、技能包和补救链完整性的脚本 | 让社区可以复核结果 |
| 示例 demo | P010-P015 | 轻量任务场景 demo | 让开发者能跑起来 |

## 四、本期不开放内容

### 4.1 暂不开放 P005-P007

P005-P007 对应空间对象制度流转、空间对象治理和空间制度运行，更适合在多主体、多空间、多权限、多机器人阶段使用。

第一期暂不开放这些能力的完整实现，原因是：

1. 当前阶段主线是经验闭环训练，不是多主体空间治理。
2. P005-P007 是后期空间秩序层和商业护城河。
3. 过早开放会让项目边界变重，也会削弱后续商业资产。

第一期只保留最小钩子：

1. 对象敏感度字段。
2. 候选动作字段。
3. 审计引用字段。
4. 后续治理状态字段。

### 4.2 不重复开源完整安全治理

运行时安全治理不作为本期 worldmodels 的主线。

原因是：

1. 安全治理已经可通过 Sayfos SDK 形成独立开源表达。
2. worldmodels 本期应聚焦经验闭环和技能形成。
3. 安全治理可以作为适配层内化到 demo 和技能共创方案中。

本期只提供最小适配关系：

1. 技能包可以附带治理策略引用。
2. 任务步骤可以附带动作声明引用。
3. 经验记录可以附带裁决结果引用。
4. 审计摘要可以兼容 Sayfos 风格的治理结果。

## 五、与 Sayfos SDK 的关系

建议对外表达为：

> Sayfos 负责动作执行前后的安全治理和裁决；worldmodels 负责真实任务中的经验记录、技能沉淀和能力复用。

两者关系如下：

| 能力 | Sayfos SDK | worldmodels 第一期 |
|---|---|---|
| 动作声明 | 主能力 | 引用或适配 |
| 来源链 | 主能力 | 写入经验上下文 |
| 代理权预算 | 主能力 | 写入执行结果 |
| 计划预适配 | 主能力 | 写入任务步骤验证结果 |
| 裁决令牌 | 主能力 | 写入经验记录和审计摘要 |
| 空间语义 | 适配对象 | 主能力 |
| 经验回写 | 适配对象 | 主能力 |
| 技能包 | 可治理对象 | 主能力 |
| 补救经验 | 可审计对象 | 主能力 |
| 人类偏好 | 可约束对象 | 主能力 |

## 六、第一期开源仓库应包含

建议第一期开源仓库包含以下内容：

```text
README.md
LICENSE
docs/
schemas/
examples/
worldmodels/
skills/
evals/
adapters/
```

### 6.1 `docs/`

文档内容：

1. 项目定位。
2. 六个经验闭环说明。
3. 快速开始。
4. 技能包格式说明。
5. Sayfos 适配说明。
6. 贡献指南。

### 6.2 `schemas/`

基础 schema：

1. `space_object.schema.json`
2. `experience_record.schema.json`
3. `concept_pattern.schema.json`
4. `task_plan.schema.json`
5. `recovery_record.schema.json`
6. `preference_record.schema.json`
7. `skill_package.schema.json`

### 6.3 `examples/`

示例数据：

1. 家庭/办公整理任务。
2. 园区巡检任务。
3. 仓储辅助任务。
4. 养老提醒任务。

### 6.4 `worldmodels/`

最小运行逻辑：

1. 读取任务。
2. 生成空间语义上下文。
3. 写入经验记录。
4. 生成补救记录。
5. 生成偏好记录。
6. 封装技能包。

### 6.5 `skills/`

技能包样例：

1. `desk_organize_basic`
2. `patrol_report_basic`
3. `warehouse_sort_basic`
4. `care_reminder_basic`

### 6.6 `evals/`

基础评测：

1. 经验记录完整性检查。
2. 技能包字段完整性检查。
3. 补救链完整性检查。
4. 偏好记录可追溯性检查。

### 6.7 `adapters/`

适配器：

1. Sayfos SDK 适配说明。
2. 简单机器人能力清单适配。
3. Agent 工具调用适配。

## 七、第一期 demo 目标

第一期 demo 不追求复杂机器人控制，而是追求闭环完整。

最小 demo 应当能跑通：

1. 输入一个真实任务目标。
2. 生成任务步骤。
3. 读取空间对象和约束。
4. 执行或模拟执行动作。
5. 记录 `{情境, 动作, 后果}`。
6. 对失败动作生成补救记录。
7. 对人类反馈生成偏好记录。
8. 从多条经验中抽象一个技能包。
9. 输出技能包和审计摘要。

## 八、第一期对外话术

建议对外表达：

> worldmodels 是一个真实世界经验闭环训练框架，用于让机器人和 Agent 在真实任务中沉淀空间语义、动作经验、补救经验、人类偏好和可复用技能包。

不建议第一期强调：

1. 通用机器人大脑。
2. 完整世界大模型。
3. 完整空间治理平台。
4. 完整企业安全网关。

这些说法会把项目引向过大的竞争场。

## 九、第一期验收标准

第一期完成时，至少应满足：

1. 开发者能在本地运行 demo。
2. demo 能输出经验记录。
3. demo 能输出补救记录。
4. demo 能输出偏好记录。
5. demo 能输出技能包。
6. schema 能被独立复用。
7. evals 能检查输出完整性。
8. 文档能说明和 Sayfos SDK 的边界。
9. README 能清楚表达“真实世界经验闭环训练范式”。

## 十、结论

第一期开源不应铺开全部技术版图，而应聚焦 P010-P015 六个经验闭环。

P005-P007 保留为后期多主体空间治理和商业护城河。

安全治理通过 Sayfos SDK 形成适配关系，不作为 worldmodels 第一期主叙事。

这能让项目用最小成本占住一个清晰位置：

> 真实世界经验闭环训练范式。
