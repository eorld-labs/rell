# P015 人类偏好工程证据说明

## 一、中文摘要

本工程证据包用于固定 P015“人类偏好”在当前 `rell_sample` 主链中的最小可运行闭环。证据链通过独立校验脚本和结构化 JSON 输出，逐段证明：系统将人类偏好作为独立于经验层和概念层的偏好记录库存储；在任务迁移时，只把与当前意图和空间情境匹配的偏好加载进任务期运行时世界状态快照；当用户询问“当前偏好约束是什么”时，系统只从当前任务期快照读取答案，而不是回退到长期经验推理；新增的人类偏好能够即时附着到当前任务，并在后续迁移判断中先约束可行集合，再将违反偏好的步骤标记为部分不可执行；同时，偏好层的读写不会直接改写经验库或概念库的持久化内容。

这份证据包的重点不是展示 UI，而是把 P015 的核心技术事实收束成可重复运行、可落盘、可复核、可直接对应审查答复口径的工程输出。

## 二、验证命令

单独生成 P015 工程证据：

```powershell
python demo_runtime\rell_sample\validate_p015_preference_alignment.py
```

全量样品检查并同步刷新证据：

```powershell

```

## 四、证据文件索引

### 1. `evidence_index.json`

用途：机器可读的证据索引。  
对应内容：列明每个证据文件对应的技术特征、主张步骤、关键字段和代码来源。

### 2. `01_preference_library_baseline.json`

用途：证明人类偏好以独立偏好记录库存储，而不是直接写入经验模型参数。  
对应技术点：将当前情境、动作和人类偏好指示作为偏好记录持久化。  
关键字段：

- `preference_library.preference_records`
- `preference_library.preference_records[].preference_id`
- `preference_library.preference_records[].enforcement_policy`

### 3. `02_runtime_snapshot_preference_loading.json`

用途：证明任务期运行时世界状态快照会加载与当前任务匹配的人类偏好。  
对应技术点：后续动作确定时基于当前情境调取匹配的人类偏好指示。  
关键字段：

- `migration.runtime_world_state_snapshot.active_preferences`
- `migration.runtime_world_state_snapshot.preference_context`
- `migration.execution_feasibility.preference_advisories`

### 4. `03_runtime_preference_query.json`

用途：证明状态问答链只从当前任务期快照读取偏好约束，而不回退到长期经验推理。  
对应技术点：基于当前情境与当前任务状态读取适用的人类偏好指示。  
关键字段：

- `preference_query.answer`
- `preference_query.evidence.active_preferences`
- `preference_query.source`

### 5. `04_preference_record_and_attachment.json`

用途：证明新增偏好记录后，可以即时附着到当前任务期快照供后续判断读取。  
对应技术点：获取人类反馈并形成新的偏好记录。  
关键字段：

- `recorded_preference.preference_record`
- `attachment.active_preferences`
- `runtime_query_after_attachment.evidence.active_preferences`

### 6. `05_preference_constrained_feasibility.json`

用途：证明偏好约束先限制可行集合，再将违反偏好的步骤标记为部分不可执行。  
对应技术点：基于约束可行性判断对候选动作进行可行性限定。  
关键字段：

- `constrained_migration.execution_feasibility.result`
- `constrained_migration.execution_feasibility.infeasible_reasons`
- `constrained_migration.experience_gap_record.preference_refs`

### 7. `06_preference_non_rewrite_boundary.json`

用途：证明偏好层作为独立运行时约束读入，不直接改写经验层或概念层持久化内容。  
对应技术点：偏好记录独立于经验层和概念层存储并参与后续动作确定。  
关键字段：

- `before_hashes.experience_library_sha1`
- `after_hashes.experience_library_sha1`
- `before_hashes.concept_library_sha1`
- `after_hashes.concept_library_sha1`

## 五、典型审查质疑对应

### 1. 被质疑“只是把用户喜好写进配置项，没有形成独立技术层”

可引用证据：

- `01_preference_library_baseline.json`
- `04_preference_record_and_attachment.json`

答复重点：P015 不是把偏好散落进任务参数，而是以独立偏好记录库、独立 `preference_id` 和独立执行策略字段进行管理，并能够在运行时按需附着到当前任务。

### 2. 被质疑“偏好只是经验推理的一部分，没有当前任务隔离”

可引用证据：

- `02_runtime_snapshot_preference_loading.json`
- `03_runtime_preference_query.json`

答复重点：系统将适用偏好加载到任务期运行时世界状态快照，并在状态问答时显式返回 `source = runtime_world_state_snapshot_only`，证明偏好读取以当前任务快照为准，而非回退到长期经验库。

### 3. 被质疑“偏好不会真正影响后续执行”

可引用证据：

- `05_preference_constrained_feasibility.json`

答复重点：偏好约束会先参与可行性限定，并将违反阻断型偏好的步骤标记为 `partially_inexecutable`，同时在 `infeasible_reasons` 和 `experience_gap_record.preference_refs` 中保留阻断依据。

### 4. 被质疑“偏好层会污染经验层或概念层，边界不清”

可引用证据：

- `06_preference_non_rewrite_boundary.json`

答复重点：脚本在偏好记录、偏好附着和偏好约束迁移前后分别对经验库和概念库计算哈希，证明 P015 的运行时约束不会直接改写 P011 经验层或概念层的持久化内容。
