# P018 实施主体与接口拆分攻防补强 v1

对象：一种面向自动化执行体的状态优先任务交互仲裁方法及系统。

目的：防止竞对或无效请求人通过“云端/端侧/执行体/第三方服务拆分实施”弱化系统权、规避侵权主体，或将本案解释为多个互不闭合的接口调用流程。

## 一、为什么需要补强

P018 天然可能采用分布式架构：

1. 交互输入来自用户端、App、语音端、远程管理系统或第三方自动化模块；
2. 语义解析可在云端；
3. 状态快照可在端侧或执行体本体；
4. 仲裁可在边缘端、云端或端侧；
5. 执行控制链路可在机器人控制器、RPA 执行器、数字执行体运行时或软件代理中；
6. 审计和偏好记忆可在另一个服务中。

如果说明书不写清楚逻辑控制主体和接口关系，竞对可能主张：其单个主体没有完整实施权利要求，云端只解析，端侧只执行，状态服务只提供查询，因此不构成完整系统。

## 二、核心补强口径

建议说明书明确：

> 本发明不要求交互接收、状态快照读取、输入判定、任务控制仲裁、后续步骤生成、执行控制和状态释放全部部署在同一物理设备中。上述模块可以部署于云端、边缘端、端侧设备、自动化执行体本体、管理服务器或第三方服务中的至少一个位置；只要其在同一仲裁控制主体的逻辑协调下，完成新交互输入基于任务期运行时世界状态快照的分流、仲裁、续推和释放，即属于本发明的一种实施方式。

其中，“同一仲裁控制主体”不必是单台机器，而是能够控制或协调上述接口调用顺序、数据依赖和执行结果的逻辑主体。

## 三、实施主体拆分风险

### 风险 1：云端只做自然语言解析

竞对做法：云端接收语音，把“去接水”解析成任务意图；端侧机器人自己读取状态并执行。

风险：竞对主张云端没有读取快照和仲裁，端侧没有解析输入，因此无人完整实施。

防御：P018 不以自然语言解析为核心。如果端侧在执行控制改变前读取快照、完成输入分流或任务控制仲裁，并生成后续步骤集合，则端侧或端侧控制器可构成实施主体。云端解析只是输入预处理。

说明书补强：解析模块可仅输出候选交互输入类型、候选任务目标或候选控制意图，仍需由仲裁控制主体基于任务期快照完成仲裁。

### 风险 2：状态服务只提供快照接口

竞对做法：一个独立状态服务提供 `GET /runtime_state`；任务编排器调用它后决定是否跳过步骤。

风险：状态服务提供商主张自己只提供状态，不做仲裁；任务编排器提供商主张状态来自外部。

防御：系统权应覆盖“状态快照读取单元”和“仲裁单元”可以分布式部署但逻辑协同。方法权实施主体可落在调用状态服务并完成仲裁的任务编排器或控制平台。

说明书补强：任务期快照可由外部状态服务提供，读取该快照并基于其完成分流、仲裁或续推的控制主体实施本发明。

### 风险 3：端侧只执行云端下发的后续步骤

竞对做法：云端根据当前状态生成后续步骤，端侧只执行。

风险：端侧机器人厂商主张自己不做仲裁，云端服务商主张自己不控制底层执行。

防御：云端若接收或读取任务期状态，并输出继续/暂停/切换/合并/终止/人工确认或后续步骤集合，已经完成 P018 核心仲裁。端侧执行只是执行控制链路。系统权可覆盖云端仲裁控制平台。

说明书补强：执行控制链路可以为外部接口、机器人 SDK、RPA 执行器、数字执行体 API 或其他执行端服务；仲裁控制主体将后续步骤集合交由该执行控制链路即可。

### 风险 4：第三方治理/安全服务输出阻断

竞对做法：安全服务输出“不可执行”，任务系统据此停止。

风险：对方主张这是治理服务，不是 P018 仲裁。

防御：P018 不主张治理判断本身。若任务系统将治理输出与任务期快照结合，输出暂停、终止、人工确认或重新生成后续步骤集合，则任务系统实施 P018。治理服务只是仲裁输入来源。

### 风险 5：多机器人/多执行体交接

竞对做法：A 机器人中断，B 机器人继承状态继续执行。

风险：对方主张这是多执行体状态交接，不是同一执行体运行时仲裁。

防御：如果该流程是高优先级任务抢占和跨执行体状态继承，可能更接近现有技术 CN121560537A，不一定落入 P018 主链。但如果新交互输入到来后，系统先基于任务期快照分流查询/控制，再决定由原执行体继续、暂停、切换到新任务或交给新执行体执行，则 P018 可覆盖仲裁控制主体。

说明书补强：自动化执行体可为单个执行体，也可为由同一仲裁控制主体协调的一组执行体；但本案核心不是状态交接本身，而是新交互输入的状态优先仲裁。

## 四、接口拆分风险

### 接口 A：交互输入接口

可包括语音、文本、按钮、远程控制请求、系统事件、API 调用或第三方模块消息。

补强写法：

> 交互输入接口用于接收可能影响当前任务或待切换任务的输入，所述输入可以为自然语言，也可以为结构化控制请求或系统事件。

### 接口 B：状态快照接口

可包括本地内存读取、数据库查询、事件流聚合、场景图查询、黑板读取、状态向量服务、模型状态特征输出。

补强写法：

> 状态快照读取接口不限于读取固定数据表，也可通过订阅事件流、查询工作记忆、读取场景图、获取状态向量或调用模型状态特征服务获得任务期运行时世界状态快照。

### 接口 C：仲裁接口

输入：交互输入、快照、候选任务、候选约束、治理输出、底层安全事件。

输出：继续、暂停、暂停并切换、合并、终止、人工确认、候选后续步骤集合或拒绝执行。

补强写法：

> 仲裁接口输出的结果为机器可解析控制结果，而非仅供人工阅读的建议。

### 接口 D：续推接口

输入：候选步骤链、当前事实、步骤前提事实、目标事实、动作完成度、阶段状态。

输出：后续待执行步骤集合。

补强写法：

> 续推接口可独立于任务规划器实现，也可嵌入任务规划器、执行编排器或恢复控制器中；只要其基于当前事实排除已满足、已完成或目标事实已成立步骤，即属于本发明的续推处理。

### 接口 E：执行控制接口

可包括机器人 SDK、ROS/MoveIt、RPA 执行器、数字执行体 API、仿真执行器、软件代理工具调用接口。

补强写法：

> 执行控制接口不要求与仲裁控制主体位于同一物理设备。仲裁控制主体将后续待执行步骤集合交由执行控制接口，即完成本发明的执行衔接。

### 接口 F：状态释放接口

可包括删除、失效标记、引用断开、任务会话关闭、快照版本冻结、过期策略或禁止后续任务直接复用。

补强写法：

> 删除或释放任务期快照不限于物理删除数据，也包括将快照标记为失效、断开其与当前任务的活动引用、关闭任务会话、冻结快照版本或禁止后续任务直接复用其中的临时事实。

## 五、系统权补强建议

系统权 9 已有模块化写法。建议说明书中写厚以下逻辑：

1. 交互接收单元、状态快照读取单元、输入判定单元、仲裁单元、续推控制单元、状态释放单元可以部署在不同物理节点；
2. 上述单元通过同一仲裁控制主体进行逻辑协调；
3. 同一仲裁控制主体用于保证数据依赖顺序：先读快照，再分流，再仲裁，再续推，再执行，再释放；
4. 如果任一单元调用外部服务，外部服务的输出作为该单元的输入或中间结果，不改变本发明的逻辑闭环；
5. 自动化执行体包括数字执行体、机器人物理自动化执行体和软件自动执行体。

## 六、方法权的实施主体解释

方法权的完整实施主体不一定是机器人本体。可能是：

1. 云端机器人任务控制平台；
2. 边缘端任务编排器；
3. 端侧执行体控制器；
4. RPA 或软件自动执行平台；
5. 数字执行体运行时；
6. 机器人厂商 SDK 控制服务；
7. 集成多个接口的仲裁控制中间件。

判断标准不是物理部署位置，而是该主体是否控制或协调完成：

1. 接收新交互输入；
2. 改变执行控制前读取任务期快照；
3. 基于输入和快照分流；
4. 对查询输出状态响应；
5. 对控制输入输出仲裁结果；
6. 基于当前事实生成后续步骤集合；
7. 交由执行控制链路继续执行；
8. 释放任务期快照。

## 七、竞对拆分规避攻防

### 规避 1：卖云端语义服务，不卖机器人

若云端语义服务只输出候选语义，不读取任务期快照、不仲裁、不续推，则不落入 P018 核心，但可能落入 P013/交互层。

若云端语义服务接收当前状态并输出仲裁结果或后续步骤集合，则具有 P018 风险。

### 规避 2：卖状态中台，不卖任务控制

若只提供通用状态查询，不处理输入分流和任务控制仲裁，P018 覆盖较弱。

若状态中台提供“基于新输入和当前状态的仲裁/续推 API”，则进入 P018 风险区。

### 规避 3：卖机器人执行器，不卖仲裁平台

若机器人只执行外部步骤，不处理新输入，P018 覆盖较弱。

若机器人端在执行中接收“继续/暂停/换任务”等输入，并基于本地状态跳过已完成步骤，则可构成 P018 实施主体。

### 规避 4：把仲裁和续推拆成两个微服务

只要两个微服务由同一平台编排，且形成“快照 -> 分流 -> 仲裁 -> 续推”的数据依赖闭环，可主张同一仲裁控制主体逻辑实施。

### 规避 5：用第三方插件完成快照读取

插件只是状态读取实现。核心在于调用方是否基于该快照完成分流、仲裁和续推。

## 八、说明书建议补写段落

建议加入以下正文：

> 在一些实施方式中，交互接收单元、状态快照读取单元、输入判定单元、仲裁单元、续推控制单元、执行控制接口和状态释放单元可以部署于不同的物理设备或服务节点。例如，交互接收单元部署于用户终端，状态快照读取单元部署于自动化执行体本体，仲裁单元部署于边缘服务器，执行控制接口部署于机器人控制器或软件执行平台。上述分布式部署不影响本发明的实施，只要上述单元在同一仲裁控制主体的逻辑协调下，按照交互输入触发、任务期快照读取、输入分流、任务控制仲裁、当前事实续推、执行控制衔接和状态释放的顺序形成闭环。

> 所述同一仲裁控制主体可以表现为单个控制器、任务编排服务、云端控制平台、边缘控制节点、端侧运行时、中间件、软件开发工具包或由多个服务组成的逻辑控制系统。其特征不在于物理部署位置，而在于能够约束上述接口调用顺序和数据依赖关系，使任务控制输入在改变执行控制行为之前基于任务期运行时世界状态快照完成仲裁。

> 在一些实施方式中，任务期运行时世界状态快照可以由外部状态服务、执行体本体、传感器融合模块、场景图服务、黑板系统或模型状态特征服务提供。读取上述状态并基于其完成交互输入分流、任务控制仲裁或后续待执行步骤集合生成的控制主体，仍属于本发明的实施方式。

## 九、权利要求是否需要改

当前不建议改独权。原因：

1. 独权已经用“自动化执行体对应的执行控制链路”避免绑定具体执行端；
2. 系统权已经写了多个单元并允许云端、边缘端、端侧或本体不同位置部署；
3. 说明书补强即可支撑分布式实施和多接口形态；
4. 如果独权加入过多接口部署细节，反而会缩窄保护范围。

可考虑在系统权或说明书中保留“同一仲裁控制主体逻辑协调”的表述，作为多主体拆分的反制核心。

## 十、取证视角

将来判断竞品是否落入 P018，可看以下外显证据：

1. 是否有执行中接收新输入的 API 或日志；
2. 新输入到来后是否读取当前任务状态、工作记忆、场景图或状态向量；
3. 是否区分状态查询和任务控制；
4. 状态查询是否不触发任务控制改变；
5. 任务控制是否输出暂停、继续、切换、合并、终止或人工确认；
6. 是否基于当前状态跳过已完成步骤；
7. 是否在任务结束后关闭、失效或释放任务期状态；
8. 云端、端侧和执行体之间是否存在上述调用链。

## 十一、结论

P018 需要做实施主体与接口拆分补强。补强重点不是修改独权，而是在说明书中明确：

1. 物理部署可以分散；
2. 逻辑仲裁控制主体必须闭合；
3. 外部服务只是输入来源或接口实现；
4. 执行控制链路可以在外部执行体；
5. 快照释放可表现为删除、失效、断链或禁止复用；
6. 云边端分布式不改变“先快照、再分流、再仲裁、再续推、再释放”的数据依赖闭环。

这样可以降低竞对通过“我只卖一个模块”“状态在第三方”“云端不控制机器人”“端侧不做语义解析”等方式规避的空间。
