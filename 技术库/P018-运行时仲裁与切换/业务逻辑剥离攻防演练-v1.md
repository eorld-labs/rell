# P018 业务逻辑剥离攻防演练 v1

对象：一种面向自动化执行体的状态优先任务交互仲裁方法及系统。

目的：从无效、审查和撰写稳定性角度，防止 P018 被理解为“业务规则、人工交互规则、任务管理制度或普通对话流程”，而要固定为自动化执行体任务执行期间的新交互输入运行时控制机制。

## 一、总原则

P018 的保护对象不是用户要做什么，也不是业务上谁优先、谁批准、谁满意，而是：

> 自动化执行体已经处于任务执行期间时，新交互输入进入执行控制链路之前，系统如何基于当前任务期运行时世界状态快照进行状态查询/任务控制分流、任务控制仲裁、当前事实续推和任务期状态释放。

因此，正式说明书和权利要求应尽量把“业务意图”写成输入来源或应用场景，把“状态快照、输入分流、仲裁、续推、释放”写成技术处理。

## 二、业务逻辑与技术逻辑的边界

| 层级 | 内容 | 是否作为 P018 核心 | 写法建议 |
|---|---|---|---|
| 人类业务意图 | “给我倒水”“去拿苹果”“先别做了” | 否 | 作为交互输入示例 |
| 自然语言理解 | 将语句识别为查询、控制、约束、教学 | 可选实现 | 不绑定 LLM 或规则 |
| 任务期状态快照 | 当前区域、持物、已成立事实、阶段进度、约束 | 是 | 技术核心 |
| 输入分流 | 状态查询输入或任务控制输入 | 是 | 技术核心 |
| 任务控制仲裁 | 继续、暂停、切换、合并、终止、人工确认 | 是 | 技术核心 |
| 当前事实续推 | 排除已完成/已满足/目标事实成立步骤 | 是 | 技术核心 |
| 用户偏好 | 严格跟随、自主变通、水不要太满 | 否/可选 | 放 P015 或可选实施 |
| 治理规则 | 安全、法规、代理权、来源链、预算 | 否/外部输入 | 不作为 P018 核心 |
| 业务审批 | 谁授权、谁确认、谁负责 | 否 | 只写人工确认为仲裁结果 |

## 三、攻击 1：这是普通业务流程/任务管理规则

### 对方攻击话术

权利要求只是规定用户输入到来后，系统判断是查询还是控制，再决定继续、暂停、切换、合并或终止。这属于业务流程或任务管理规则，没有技术贡献。

### 防御主轴

P018 不保护“继续、暂停、切换”的业务含义，而保护这些结果产生前的技术处理链：

1. 自动化执行体处于任务执行期间；
2. 交互输入可能改变执行控制行为；
3. 改变执行控制行为之前读取任务期运行时世界状态快照；
4. 基于快照分流查询/控制；
5. 基于当前事实生成后续待执行步骤集合；
6. 释放任务期快照避免状态污染。

### 技术效果

该技术处理避免自动化执行体重复执行、错误切换、误中断和状态污染，直接影响执行控制链路，不是业务人员的管理安排。

## 四、攻击 2：状态查询只是用户界面交互

### 对方攻击话术

“你在哪”“下一步做什么”“杯子有没有水”只是对用户界面的状态展示，属于人机交互信息展示。

### 防御主轴

P018 中的状态查询不是孤立 UI 展示，而是新交互输入统一入口的一条分支。其技术意义在于：

1. 在任务执行期间识别该输入不会改变执行控制路径；
2. 直接读取任务期快照输出状态响应；
3. 避免将状态查询误判为任务控制输入，导致任务中断或重规划。

### 写法建议

不要把状态查询写成“为用户提供更好体验”。应写成：

> 通过将状态查询输入从任务控制输入中分离，避免状态查询触发执行控制链路改变，从而降低误中断和错误重规划风险。

## 五、攻击 3：任务优先级和切换是业务调度

### 对方攻击话术

暂停、切换、合并、终止本质上是任务调度策略，属于业务规则。谁优先、谁后执行，是管理逻辑。

### 防御主轴

P018 不主张具体优先级规则，也不主张某类任务必须优先。P018 主张的是：无论采用何种优先级或业务规则，任务控制输入在改变执行控制前必须先基于当前任务期状态快照仲裁。

例如，“去拿苹果”是否比“倒水”优先，可以由业务层或用户偏好决定；但如果执行体当前持有杯子，则系统必须基于持物状态输出人工确认、先释放对象、暂停或终止等仲裁结果，而不是直接执行拿苹果。

### 写法建议

把“优先级、偏好、业务规则”写成仲裁输入之一，而不是仲裁机制本身。

## 六、攻击 4：人工确认是业务审批

### 对方攻击话术

请求人工确认只是让人批准或选择，属于业务审批流程。

### 防御主轴

P018 中的人工确认不是审批制度，而是基于当前任务期状态快照产生的一种执行控制仲裁结果。其触发原因可包括：

1. 当前持有对象与新任务冲突；
2. 新任务目标未落地；
3. 对象指认不充分；
4. 当前事实与新输入存在冲突；
5. 治理层或底层安全环输出阻断事件。

### 技术效果

人工确认作为仲裁结果，防止自动化执行体在状态不确定或动作有风险时直接改变执行路径。

## 七、攻击 5：偏好记忆/教学输入是人的主观规则

### 对方攻击话术

“水不要太满”“轻一点”“下次按我的步骤做”都是人的主观偏好或教学规则，不具备技术性。

### 防御主轴

P018 不把偏好本身作为核心。偏好或教学输入只是任务控制输入的一种来源。技术处理在于：

1. 将偏好/教学输入转换为候选约束、候选步骤、候选目标事实或澄清问题；
2. 回到任务期快照进行仲裁；
3. 判断该约束是否可合并入当前任务、是否需要确认、是否会改变后续步骤集合。

### 边界

偏好记忆属于 P015 或可选实施，不应写成 P018 的独权核心。

## 八、攻击 6：治理层规则是制度/法律业务逻辑

### 对方攻击话术

安全、法律、代理权、来源链、预算都是制度规则或业务规则，不能支撑 P018 技术性。

### 防御主轴

P018 不主张治理规则本身。治理层输出仅可作为仲裁输入，例如：

1. 安全阻断事件；
2. 代理权不足；
3. 来源链异常；
4. 预算不足；
5. 物理后果风险。

P018 的技术处理是：这些输入与任务期快照一起进入仲裁，决定当前任务是否继续、暂停、切换、合并、终止或人工确认。

### 写法建议

不要在 P018 正文中过多展开法律法规或制度内容。只写“外部约束结果或安全事件可作为仲裁输入”。

## 九、攻击 7：自然语言解析是业务语义理解

### 对方攻击话术

判断“你在哪”是查询，“去拿苹果”是控制，是语言理解或业务语义分类，不是技术方案。

### 防御主轴

自然语言解析不是 P018 的核心。P018 不限定输入如何被解析，可由规则、模型、结构化接口、按钮、远程控制请求或系统事件产生。核心在于输入到达后，必须与任务期状态快照共同决定分流和仲裁。

### 写法建议

说明书中可写：

> 交互输入可为自然语言、结构化指令、系统事件或远程控制请求，本发明不限定其解析方式。解析结果仅作为候选输入类型或候选任务控制意图，仍需基于任务期运行时世界状态快照完成仲裁。

## 十、攻击 8：快照释放只是数据生命周期管理

### 对方攻击话术

任务结束后释放状态，只是普通缓存清理或数据生命周期管理。

### 防御主轴

P018 的释放不是孤立缓存清理，而是任务期运行时状态隔离的一部分。其技术效果是：

1. 防止当前任务的临时持物状态被后续任务误用；
2. 防止当前约束、当前步骤、挂起任务进度污染后续任务；
3. 防止状态查询读到已结束任务的过期事实；
4. 防止任务控制输入基于失效状态进行错误仲裁。

### 写法建议

释放步骤应与“任务期”绑定，强调其用于状态隔离和执行安全，而不是普通内存释放。

## 十一、攻击 9：当前事实裁剪只是业务上的不重复劳动

### 对方攻击话术

已经做过的步骤不再做，是业务常识，不具备创造性。

### 防御主轴

P018 的“裁剪”不是人工判断不重复劳动，而是机器可执行的状态比对：

1. 候选步骤有执行前提事实和目标事实；
2. 任务期快照中有当前事实；
3. 系统比对当前事实与步骤前提/目标事实；
4. 排除已满足执行前提、已完成执行或目标事实已成立的步骤；
5. 生成后续待执行步骤集合并交给执行控制链路。

### 技术效果

避免自动化执行体在真实或数字空间中重复移动、重复抓取、重复写入、重复调用接口或重复占用资源。

## 十二、业务剥离后的权利要求解释

权利要求中的关键特征应按技术含义解释：

| 术语 | 技术含义 | 非核心业务含义 |
|---|---|---|
| 交互输入 | 进入执行控制链路前的外部或内部输入事件 | 用户想做什么 |
| 状态查询输入 | 不改变执行控制路径的状态读取型输入 | 问答体验 |
| 任务控制输入 | 可改变执行路径的控制型输入 | 用户命令 |
| 仲裁结果 | 对执行控制链路的机器可解析控制结论 | 管理决策 |
| 当前事实 | 当前任务期状态依据 | 人的判断 |
| 后续待执行步骤集合 | 交由执行链路继续执行的机器步骤集合 | 工作安排 |
| 快照释放 | 任务期状态隔离和清理 | 普通缓存清理 |

## 十三、撰写禁区

正式说明书中尽量避免以下表述成为主线：

1. “提高用户体验”作为主要效果；
2. “更符合人的意图”作为主要效果；
3. “业务流程更加顺畅”作为主要效果；
4. “根据用户优先级安排任务”作为核心；
5. “人工审批后执行”作为核心；
6. “聊天更自然”作为核心；
7. “满足管理制度要求”作为核心。

应替换为：

1. 避免执行控制误中断；
2. 避免重复执行已完成动作；
3. 避免当前持物状态被忽略导致错误切换；
4. 避免过期任务期状态污染后续任务；
5. 使状态查询不进入任务控制链路；
6. 使任务控制输入基于当前世界状态续推。

## 十四、业务剥离后的实施例写法

### 例 1：用户问“杯子有没有水”

业务表达：用户想知道杯子状态。

技术表达：系统识别为状态查询输入，读取任务期快照中的 `cup_contains_water`、`cup_has_water` 或液位状态表征，输出状态响应，不改变当前执行控制路径。

### 例 2：用户说“去水源处接一杯水”

业务表达：用户下达新任务。

技术表达：系统识别为任务控制输入，读取当前快照，发现 `cup_in_gripper` 已成立，生成后续待执行步骤集合时排除“移动到操作台”和“拿起杯子”，仅保留移动至水源和接水。

### 例 3：用户说“别做了，去拿苹果”

业务表达：用户改口。

技术表达：系统识别为任务切换输入，读取当前快照，发现执行体持有杯子且新目标对象未落地，输出请求人工确认、先释放当前持有对象、暂停或终止当前任务中的至少一种仲裁结果。

### 例 4：倒水一半时用户输入新约束

业务表达：用户希望水不要太满。

技术表达：系统读取当前过程阶段、液位高度、流速、倾角等连续状态变量，将新约束转换为候选目标阈值或候选度条件，仲裁是否合并入当前任务，并从当前保持流动阶段续推。

## 十五、与 P015、治理层、P016/P017 的边界

P015：处理人类偏好、教学追认和端侧偏好记忆。P018 只处理偏好输入进入当前任务控制链路前的仲裁。

治理层：处理安全、法律、来源链、代理权、预算权等可执行边界。P018 只接收治理层输出作为仲裁输入。

P016：处理物理动作过程模板、阶段跃迁、因果验真和恢复。P018 只处理新交互输入如何影响当前任务执行链路。

P017：处理真实经验跨空间跨本体迁移适配。P018 使用任务期快照进行运行时交互仲裁，不主张经验迁移本身。

## 十六、无效/审查答辩核心句

可使用以下答辩句式：

1. 本案不保护用户任务安排或业务优先级，而保护自动化执行体任务执行期间新交互输入进入执行控制链路前的状态优先仲裁机制。
2. 本案的任务期运行时世界状态快照不是业务记录，而是执行控制链路用于避免误中断、重复执行、错误切换和状态污染的运行时状态依据。
3. 本案的状态查询输入不是普通信息展示，而是从任务控制输入中分离出来的执行安全分支，用于防止查询触发任务控制路径改变。
4. 本案的人工确认不是业务审批，而是当当前事实与任务控制输入存在冲突或不确定性时，由系统基于快照输出的执行控制仲裁结果。
5. 本案的当前事实裁剪不是人工排班，而是对候选步骤执行前提事实和目标事实进行机器比对后生成后续待执行步骤集合。

## 十七、结论

从业务逻辑剥离角度看，P018 的撰写重点应避免落入“用户意图理解、业务任务调度、偏好管理或人工审批”。所有业务语义都应被压缩为输入来源、候选意图或外部约束；真正进入权利要求和发明内容主线的，是任务期快照、输入分流、任务控制仲裁、当前事实续推和快照释放。

只要这个边界守住，P018 就不容易被打成抽象业务方法；它的技术贡献可以稳定落在自动化执行体运行时控制和状态隔离上。
