# P018 近似检索综合报告

## 一、技术特征拆解

P018 独权的七段主链：

```
交互输入到来 → 先读任务期快照 → 分流(状态查询/任务控制)
→ 状态查询：直接读快照返回
→ 任务控制：仲裁(继续/暂停/暂停切换/合并/终止/人工确认)
→ 允许继续/合并/切换：基于当前事实裁剪已满足步骤
→ 从当前状态续推，不从初始重跑
→ 任务结束后释放快照
```

**与现有技术最核心的三个区分点：**

1. **"改变执行控制行为之前"先读快照** — 不是事后记录，不是并行通知，是强制顺序门
2. **输入类型分流(状态查询 vs 任务控制)** — 不是所有输入都进同一个处理管道
3. **基于当前事实裁剪续推步骤** — 删除已满足前提/已完成/目标已成立的步骤，不从零重跑

## 四、最高风险件：CN122201291A

这是 13 件中唯一同时触及"任务执行期间交互""状态感知""可中断交互""对话与任务融合"的文献。

**必须拉全文逐项打勾的 5 个检查点：**

| 检查项 | 打勾确认 |
|--------|:--------:|
| 是否有"改变执行控制行为之前必须先读当前状态"的强制性门 | 是/否 |
| 是否把输入区分为查询型和控制型两个处理管道 | 是/否 |
| 是否输出包含"继续/暂停/暂停切换/合并/终止/人工确认"的仲裁集 | 是/否 |
| 是否基于当前已成立事实删除已满足/已完成的步骤 | 是/否 |
| 是否有任务期状态的隔离释放机制 | 是/否 |

如果 CN122201291A 在以上 5 个检查项中命中 3 项或以上，P018 的独权需要收窄。如果只命中 1-2 项，区分度够。

## 六、权利要求修改建议（基于当前风险）

当前 v4 结构完整，但有一处可收紧：

**条 5（持物状态约束）当前写"输出请求人工确认、要求先释放当前持有对象、暂停当前任务或终止当前任务中的至少一种"。** 应该把"暂停当前任务"默认设为最高优先级（新输入没落地前保持当前态），改为：

> "输出暂停当前任务或请求人工确认中的至少一种仲裁结果，在新任务目标未在任务期运行时世界状态快照中得到落地确认之前不执行实际切换动作。"

这样把"暂停"确立为默认安全态，避免审查员说"你的暂停和别人的中止/暂停没区别"。

## 七、综合结论

1. P018 独权主链的单点要素（中断恢复、状态快照、交互输入）在已检文献中都有分散出现，但**合体闭环**（状态优先门+输入分流+仲裁集+事实裁剪续推+快照释放）无单件覆盖。
2. CN122201291A 是最高风险件，必须拉全文逐项打勾。
3. 中文线重点查"任务执行中交互仲裁+状态分流"、"事实裁剪+续推不重跑"两个方向。
4. 条 5 收紧"默认暂停"策略可以加固持物安全约束的区分度。
