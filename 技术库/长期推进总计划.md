# 长期推进总计划

## 一、总目标

把本项目做成一套可持续迭代、可落地、可审查、可迁移的“真实世界自动化执行体底座”。

核心不是一次性做大，而是持续把以下三件事串成闭环：

1. 技术特征可以被定义。
2. 技术特征可以被代码实现。
3. 代码实现可以被证据验证。

## 二、总原则

1. 母案专利不是单点材料，而是连续系统的一部分。
2. 每个技术点必须对应专利特征、代码模块、输出证据和验证命令。
3. 每天只推进一个最小可验证改动，但必须留下研发记录。
4. 任何提交都必须能回答四个问题：
   - 做了什么。
   - 对应哪个母案/分案特征。
   - 代码在哪里。
   - 怎么验证。

## 三、技术主线

### 3.1 空间治理与对象秩序层

对应母案：

- P005 空间对象制度流转方法
- P006 空间对象治理
- P007 空间制度运行

目标：

1. 把空间对象、区域、权限、候选态、确认流程和审计链做成底座字段。
2. 把对象变化从“直接改写”变成“候选流转”。
3. 为后续多主体、多区域、多状态治理预埋接口。

### 3.2 空间经验世界模型层

对应母案：

- P010 空间语义自动生成
- P011 物理经验内化机制
- P012 概念归纳跨域迁移
- P013 任务语义翻译
- P014 执行恢复
- P015 人类偏好

目标：

1. 把空间语义、任务经验、后果记录、补救记录和偏好记录统一起来。
2. 把执行结果回写成可复用经验。
3. 让经验参与后续计划和动作决策。

### 3.3 运行时执行治理层

对应母案：

- 代理权边界治理
- 物理执行链路验真
- 代理权预算
- 来源链
- 计划预适配
- 保护策略声明

目标：

1. 把自动化执行体的高风险动作变成可裁决、可扣减、可阻断的对象。
2. 把执行来源、预算、计划边界和物理后果统一到运行时治理。
3. 把治理结果输出为机器可解析的令牌、摘要和审计对象。

## 四、阶段推进路线

### 阶段一：总纲与证据骨架

目标：

1. 统一项目口径。
2. 统一专利母案与技术主线。
3. 统一提交规范、研发记录和证据链。

最低产出：

- 项目总纲
- 核心对象模型
- 研发记录制度
- 提交规范与专利链映射
- 代码-技术对应证据链

### 阶段二：最小运行时闭环

目标：

1. 跑通计划预验真。
2. 跑通来源链校验。
3. 跑通代理权预算裁决。
4. 跑通裁决令牌输出。
5. 跑通经验回写和审计摘要。

最低产出：

- `demo_runtime/run_demo.py`
- `demo_runtime/api_server.py`
- 输出 JSON 证据
- 验证日志

### 阶段三：经验增强

目标：

1. 让经验记录更细。
2. 引入补救经验与偏好记录。
3. 让任务语义翻译和经验回写互相调用。

最低产出：

- 经验数据结构升级
- 偏好记录结构
- 补救经验结构
- 检索与回写逻辑

### 阶段四：空间对象治理增强

目标：

1. 把 P005-P007 的候选态、确认态、审计态做完整。
2. 引入多主体冲突处理。
3. 把空间对象治理做成可对外调用的接口。

最低产出：

- 空间对象治理 API
- 候选态流转规则
- 冲突处理规则
- 审计追踪对象

### 阶段五：外部化与行业接入

目标：

1. 输出 SDK/API。
2. 输出经验库和规则库。
3. 输出行业试点方案。

最低产出：

- API 文档
- SDK 草案
- 行业场景说明
- 商业叙事与许可边界说明

## 五、每日推进模式

每天只做一小步，但必须形成闭环：

1. 选定一个小目标。
2. 改一个文档、对象、规则、代码或测试。
3. 跑一次验证。
4. 写一条研发记录。
5. 做一次中文提交。
6. 推送到远端。

## 六、证据标准

每个技术点至少保留以下四类材料中的三类，优先保留四类：

1. 文档。
2. 代码。
3. 输出证据。
4. 验证命令或日志。

如果是重点技术点，建议保留完整四类：

- 技术定义文档
- 代码实现
- 输出 JSON/日志
- Git 提交记录

## 七、审查应对要求

为了应对非正常、复审或后续争议，所有母案都要满足：

1. 技术特征能落到代码。
2. 代码能落到输出。
3. 输出能落到提交。
4. 提交能落到研发记录。
5. 研发记录能落到当天事实。

## 八、结论

本项目的长期打法不是散点式做功能，而是把专利母案、代码实现、验证证据和研发记录串成一条连续链。

以后每天推进一点点，也要让这条链越来越长、越来越密、越来越经得起查。
